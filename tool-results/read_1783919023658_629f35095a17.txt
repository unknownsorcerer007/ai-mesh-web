    33→# tasteskill: Anti-Slop Frontend Skill
    34→
    35→> Landing pages, portfolios, and redesigns. Not dashboards, not data tables, not multi-step product UI.
    36→> Every rule below is **contextual**. None of it fires automatically. First read the brief, then pull only what fits.
    37→
    38→---
    39→
    40→## 0. BRIEF INFERENCE (Read the Room Before Anything Else)
    41→
    42→Before touching code or tweaking dials, **infer what the user actually wants**. Most LLM design output is bad because the model jumps to a default aesthetic instead of reading the room.
    43→
    44→### 0.A Read these signals first
    45→1. **Page kind** - landing (SaaS / consumer / agency / event), portfolio (dev / designer / creative studio), redesign (preserve vs overhaul), editorial / blog.
    46→2. **Vibe words** the user used - "minimalist", "calm", "Linear-style", "Awwwards", "brutalist", "premium consumer", "Apple-y", "playful", "serious B2B", "editorial", "agency-y", "glassy", "dark tech".
    47→3. **Reference signals** - URLs they linked, screenshots they pasted, products they named, brands they're competing with.
    48→4. **Audience** - B2B procurement panel vs. design-conscious consumer vs. recruiter scanning a portfolio. The audience picks the aesthetic, not your taste.
    49→5. **Brand assets that already exist** - logo, color, type, photography. For redesigns, these are starting material, not optional input (see Section 11).
    50→6. **Quiet constraints** - accessibility-first audiences, public-sector, regulated industries, trust-first commerce, kids' products. These constraints OVERRIDE aesthetic preference.
    51→
    52→### 0.B Output a one-line "Design Read" before generating
    53→Before any code, state in one line: **"Reading this as: \<page kind> for \<audience>, with a \<vibe> language, leaning toward \<design system or aesthetic family>."**
    54→
    55→Example reads:
    56→- *"Reading this as: B2B SaaS landing for technical buyers, with a Linear-style minimalist language, leaning toward Tailwind utilities + Geist + restrained motion."*
    57→- *"Reading this as: solo designer portfolio for hiring managers, with an editorial / kinetic-type language, leaning toward native CSS + scroll-driven animation + custom typography."*
    58→- *"Reading this as: redesign of a public-sector service site, with a trust-first language, leaning toward GOV.UK Frontend or USWDS."*
    59→
    60→### 0.C If the brief is ambiguous, ask one question, do not guess
    61→Ask exactly **one** clarifying question - never a multi-question dump - and only when the design read genuinely diverges. Example: *"Should this feel closer to Linear-clean or Awwwards-experimental?"*
    62→
    63→If you can confidently infer from context, **do not ask**. Just declare the design read and proceed.
    64→
    65→### 0.D Anti-Default Discipline
    66→Do not default to: AI-purple gradients, centered hero over dark mesh, three equal feature cards, generic glassmorphism on everything, infinite-loop micro-animations everywhere, Inter + slate-900. These are the LLM defaults. Reach past them deliberately based on the design read.

    70→## 1. THE THREE DIALS (Core Configuration)
    71→
    72→After the design read, set three dials. Every layout, motion, and density decision below is gated by these.
    73→
    74→* **`DESIGN_VARIANCE: 8`** - 1 = Perfect Symmetry, 10 = Artsy Chaos
    75→* **`MOTION_INTENSITY: 6`** - 1 = Static, 10 = Cinematic / Physics
    76→* **`VISUAL_DENSITY: 4`** - 1 = Art Gallery / Airy, 10 = Cockpit / Packed Data
    77→
    78→**Baseline:** `8 / 6 / 4`. Use these unless the design read overrides them. Do not ask the user to edit this file - overrides happen conversationally.
    79→
    80→### 1.A Dial Inference (design read → dial values)
    81→| Signal | VARIANCE | MOTION | DENSITY |
    82→|---|---|---|---|
    83→| "minimalist / clean / calm / editorial / Linear-style" | 5-6 | 3-4 | 2-3 |
    84→| "premium consumer / Apple-y / luxury / brand" | 7-8 | 5-7 | 3-4 |
    85→| "playful / wild / Dribbble / Awwwards / experimental / agency" | 9-10 | 8-10 | 3-4 |
    86→| "landing page / portfolio / marketing site (default)" | 7-9 | 6-8 | 3-5 |
    87→| "trust-first / public-sector / regulated / accessibility-critical" | 3-4 | 2-3 | 4-5 |
    88→| "redesign - preserve" | match existing | +1 | match existing |
    89→| "redesign - overhaul" | +2 | +2 | match existing |
    90→
    91→### 1.B Use-Case Presets
    92→| Use case | VARIANCE | MOTION | DENSITY |
    93→|---|---|---|---|
    94→| Landing (SaaS, mainstream) | 7 | 6 | 4 |
    95→| Landing (Agency / creative) | 9 | 8 | 3 |
    96→| Landing (Premium consumer) | 7 | 6 | 3 |
    97→| Portfolio (Designer / studio) | 8 | 7 | 3 |
    98→| Portfolio (Developer) | 6 | 5 | 4 |
    99→| Editorial / Blog | 6 | 4 | 3 |
   100→| Public-sector service | 3 | 2 | 5 |
   101→| Redesign - preserve | match | match+1 | match |
   102→| Redesign - overhaul | +2 | +2 | match |
   103→
   104→### 1.C How the Dials Drive Output
   105→Use these (or user-overridden values) as global variables. Cross-references throughout this document refer to these exact variable names - never invent aliases like `LAYOUT_VARIANCE` or `ANIM_LEVEL`.

   109→## 2. BRIEF → DESIGN SYSTEM MAP
   110→
   111→Once you have the design read (Section 0) and dials (Section 1), pick the right foundation. Do not invent CSS for things that have an official package. Do not pretend an aesthetic trend is an official system.
   112→
   113→### 2.A When to reach for a real design system (use official packages)
   114→| Brief reads as… | Reach for | Why |
   115→|---|---|---|
   116→| Microsoft / enterprise SaaS / dashboards | `@fluentui/react-components` or `@fluentui/web-components` | Official Fluent UI, Microsoft tokens, accessibility done |
   117→| Google-ish UI, Material-flavored product | `@material/web` + Material 3 tokens | Official, theme-able via Material Theming |
   118→| IBM-style B2B / enterprise analytics | `@carbon/react` + `@carbon/styles` | Official Carbon, mature data-density patterns |
   119→| Shopify app surfaces | `polaris.js` web components / Polaris React | Required for Shopify admin UI |
   120→| Atlassian / Jira-style product | `@atlaskit/*` + `@atlaskit/tokens` | Official Atlassian DS |
   121→| GitHub-style devtool / community page | `@primer/css` or `@primer/react-brand` | Official Primer; Brand variant for marketing |
   122→| Public-sector UK service | `govuk-frontend` | Legally / regulatorily expected |
   123→| US public-sector / trust-first | `uswds` | Same |
   124→| Fast local-business / agency MVP | Bootstrap 5.3 | Boring, fast, works |
   125→| Modern accessible React foundation | `@radix-ui/themes` | Primitives + polished theme |
   126→| Modern SaaS where you own the components | shadcn/ui (`npx shadcn@latest add ...`) | You own the code, easy to customise; never ship default state |
   127→| Tailwind-based modern SaaS / AI marketing | Tailwind v4 utilities + `dark:` variant | Default for indie + small team builds |
   128→
   129→**Honesty rule:** if the brief reads as one of the systems above, install and use the **official** package. Do not recreate its CSS by hand. Do not import a system's tokens but then override 90% of them.
   130→
   131→**One system per project.** Do not mix Fluent React with Carbon in the same tree. Do not import shadcn/ui components into a Material 3 app.
   132→
   133→### 2.B When the brief is an aesthetic, not a system
   134→For these directions, there is **no single official package**. Build with native CSS + Tailwind + a maintained component library. Be honest in code comments about what is borrowed inspiration vs. official material.
   135→
   136→| Aesthetic | Honest implementation |
   137→|---|---|
   138→| Glassmorphism / "frosted glass" | `backdrop-filter`, layered borders, highlight overlays. Provide solid-fill fallback for `prefers-reduced-transparency`. |
   139→| Bento (Apple-style tile grids) | CSS Grid with mixed cell sizes. No single library owns this. |
   140→| Brutalism | Native CSS, monospace, raw borders. No library. |
   141→| Editorial / magazine | Serif type, asymmetric grid, generous whitespace. No library. |
   142→| Dark tech / hacker | Mono + accent neon, terminal motifs. No library. |
   143→| Aurora / mesh gradients | SVG or layered radial gradients. No library. |
   144→| Kinetic typography | Native CSS animations, scroll-driven animations, GSAP for hijacks. No library. |
   145→| **Apple Liquid Glass** | Apple documents this for Apple platforms only. **There is no official `liquid-glass.css`.** Web implementations are approximations using `backdrop-filter` + layered borders + highlights. Label clearly as approximation. |

   149→## 3. DEFAULT ARCHITECTURE & CONVENTIONS
   150→
   151→Unless the design read picks a real design system (Section 2.A), these are the defaults:
   152→
   153→### 3.A Stack
   154→* **Framework:** React or Next.js. Default to Server Components (RSC).
   155→  * **RSC SAFETY:** Global state works ONLY in Client Components. In Next.js, wrap providers in a `"use client"` component.
   156→  * **INTERACTIVITY ISOLATION:** Any component using Motion, scroll listeners, or pointer physics MUST be an isolated leaf with `'use client'` at the top. Server Components render static layouts only.
   157→* **Styling:** **Tailwind v4** (default). Tailwind v3 only if the existing project demands it.
   158→  * For v4: do NOT use `tailwindcss` plugin in `postcss.config.js`. Use `@tailwindcss/postcss` or the Vite plugin.
   159→* **Animation:** **Motion** (the library formerly known as Framer Motion). Import from `motion/react` (`import { motion } from "motion/react"`). The `framer-motion` package still works as a legacy alias - prefer `motion/react` in new code.
   160→* **Fonts:** Always use `next/font` (Next.js) or self-host with `@font-face` + `font-display: swap`. Never link Google Fonts via `<link>` in production.
   161→
   162→### 3.B State
   163→* Local `useState` / `useReducer` for isolated UI.
   164→* Global state ONLY for deep prop-drilling avoidance - Zustand, Jotai, or React context.
   165→* **NEVER** use `useState` to track continuous values driven by user input (mouse position, scroll progress, pointer physics, magnetic hover). Use Motion's `useMotionValue` / `useTransform` / `useScroll`. `useState` re-renders the React tree on every change and collapses on mobile.
   166→
   167→### 3.C Icons
   168→* **Allowed libraries (priority order):** `@phosphor-icons/react`, `hugeicons-react`, `@radix-ui/react-icons`, `@tabler/icons-react`.
   169→* **Discouraged:** `lucide-react`. Acceptable only when the user explicitly asks for it or the project already depends on it.
   170→* **NEVER hand-roll SVG icons.** If a glyph is missing, install a second library or compose from primitives - do not draw icon paths from scratch.
   171→* **One family per project.** Do not mix Phosphor with Lucide in the same component tree.
   172→* **Standardize `strokeWidth` globally** (e.g. `1.5` or `2.0`).
   173→
   174→### 3.D Emoji Policy
   175→Discouraged by default in code, markup, and visible text. Replace symbols with icon-library glyphs. **Override:** allow emojis only when the user explicitly asks for a playful / chat-style / social-native vibe - and even then use them sparingly with intent.
   176→
   177→### 3.E Responsiveness & Layout Mechanics
   178→* Standardize breakpoints (`sm 640`, `md 768`, `lg 1024`, `xl 1280`, `2xl 1536`).
   179→* Contain page layouts using `max-w-[1400px] mx-auto` or `max-w-7xl`.
   180→* **Viewport Stability:** NEVER use `h-screen` for full-height Hero sections. ALWAYS use `min-h-[100dvh]` to prevent layout jumping on mobile (iOS Safari address bar).
   181→* **Grid over Flex-Math:** NEVER use complex flexbox percentage math (`w-[calc(33%-1rem)]`). ALWAYS use CSS Grid (`grid grid-cols-1 md:grid-cols-3 gap-6`).
   182→
   183→### 3.F Dependency Verification (mandatory)
   184→Before importing ANY 3rd-party library, check `package.json`. If the package is missing, output the install command first. **Never** assume a library exists.

   188→## 4. DESIGN ENGINEERING DIRECTIVES (Bias Correction)
   189→
   190→LLMs default to clichés. Override these defaults proactively. Each rule has a context-aware override path.
   191→
   192→### 4.1 Typography
   193→* **Display / Headlines:** Default `text-4xl md:text-6xl tracking-tighter leading-none`.
   194→* **Body / Paragraphs:** Default `text-base text-gray-600 leading-relaxed max-w-[65ch]`.
   195→* **Sans font choice:**
   196→  * **Discouraged as default:** `Inter`. Pick `Geist`, `Outfit`, `Cabinet Grotesk`, `Satoshi`, or a brand-appropriate serif first.
   197→  * **Override:** Inter is acceptable when the user explicitly asks for a neutral / standard / Linear-style feel, or when the brief is a public-sector / accessibility-first site.
   198→* **Pairings to know:** `Geist` + `Geist Mono`, `Satoshi` + `JetBrains Mono`, `Cabinet Grotesk` + `Inter Tight`, `GT America` + `IBM Plex Mono`.
   199→
   200→* **SERIF DISCIPLINE (VERY DISCOURAGED AS DEFAULT):**
   201→  * Serif is **very discouraged as the default font for any project.** "It feels creative / premium / editorial" is NOT a reason to reach for serif. The agent's default mental model that "creative brief = serif" is the single most-tested AI tell in production rounds.
   202→  * **Serif is only acceptable when ONE of these is explicitly true:**
   203→    - The brand brief literally names a serif font, OR
   204→    - The aesthetic family is genuinely editorial / luxury / publication / manuscript / heritage / vintage AND you can articulate why this specific serif fits this specific brand
   205→  * For everything else (creative agency, design studio, modern brand, premium consumer, portfolio, lifestyle), **default sans-serif display** (Geist Display, ABC Diatype, Söhne Breit, Cabinet Grotesk Display, Migra Sans, GT Walsheim, Inter Display, PP Neue Montreal). Sans display fonts are not "boring" — they are the default for the same reason black is the default in fashion.
   206→  * **EMPHASIS RULE (related):** When you want to emphasize a word within a headline (the kinetic "and `spatial` design" type move), use **italic or bold of the SAME font**. Do NOT inject a random serif word into a sans headline (or vice versa) just to add visual interest. Mixed-family emphasis is amateur. Italic/bold emphasis in the same family is the right move.
   207→  * **Specifically BANNED as defaults:** `Fraunces` and `Instrument_Serif` (the two LLM-favorite display serifs).
   208→  * **If a serif is justified** (rare, per the above), rotate from this pool, do NOT reuse the same serif across consecutive projects: PP Editorial New, GT Sectra Display, Cardinal Grotesque, Reckless Neue, Tiempos Headline, Recoleta, Cormorant Garamond, Playfair Display, EB Garamond, IvyPresto, Migra, Editorial Old, Saol Display, Söhne Breit Kursiv, Domaine Display, Canela, Schnyder, Tobias, NB Architekt, ITC Galliard.
   209→
   210→* **ITALIC DESCENDER CLEARANCE (mandatory):** When italic is used in display type and the word contains a descender letter (`y g j p q`), `leading-[1]` or `leading-none` will clip the descender. Use `leading-[1.1]` minimum and add `pb-1` or `mb-1` reserve on the wrapping element. Audit every italic word in display headlines before shipping.
   211→
   212→### 4.2 Color Calibration
   213→* Max 1 accent color. Saturation < 80% by default.
   214→* **THE LILA RULE:** The "AI Purple / Blue glow" aesthetic is discouraged as a default. No automatic purple button glows, no random neon gradients. Use neutral bases (Zinc / Slate / Stone) with high-contrast singular accents (Emerald, Electric Blue, Deep Rose, Burnt Orange, etc.).
   215→* **Override:** if the brand or brief explicitly asks for purple / violet / lila, embrace it. But execute with intent: consistent palette, harmonised neutrals, restrained gradients. Not generic AI gradient slop.
   216→* **One palette per project.** Do not fluctuate between warm and cool grays within the same project.
   217→* **COLOR CONSISTENCY LOCK (mandatory):** Once an accent color is chosen for a page, it is used on the WHOLE page. A warm-grey site does not suddenly get a blue CTA in section 7. A rose-accented site does not get a teal status badge in the footer. Pick one accent, lock it, audit every component before shipping.
   218→
   219→* **PREMIUM-CONSUMER PALETTE BAN (mandatory, second-most-recurring AI-tell):**
   220→  * For premium-consumer briefs (cookware, wellness, artisan, luxury, heritage craft, DTC home goods, etc.) the LLM default is **warm beige/cream + brass/clay/oxblood/ochre + espresso/ink dark text**. Concretely banned hex families as default backgrounds and accents:
   221→    - Backgrounds: `#f5f1ea`, `#f7f5f1`, `#fbf8f1`, `#efeae0`, `#ece6db`, `#faf7f1`, `#e8dfcb` (all "warm paper / cream / chalk / bone")
   222→    - Accents: `#b08947`, `#b6553a`, `#9a2436`, `#9c6e2a`, `#bc7c3a`, `#7d5621` (all "brass / clay / oxblood / ochre")
   223→    - Text: `#1a1714`, `#1a1814`, `#1b1814` (all "espresso / warm near-black")
   224→  * This palette is BANNED as the default reach for premium-consumer briefs. Every premium-consumer site you have ever shipped uses this exact palette. The brand becomes invisible.
   225→  * **Default alternatives (rotate, do not reuse):**
   226→    - **Cold Luxury:** silver-grey + chrome + smoke (think Tesla, Apple Watch Hermes-without-the-leather)
   227→    - **Forest:** deep green + bone + amber accent (think Filson, Patagonia premium)
   228→    - **Black and Tan:** true off-black + warm tan, sharp contrast, no beige
   229→    - **Cobalt + Cream:** saturated blue against a single neutral, no brass
   230→    - **Terracotta + Slate:** warm rust against cool grey, no brass
   231→    - **Olive + Brick + Paper:** muted olive plus brick-red accent
   232→    - **Pure monochrome + single saturated pop:** off-white + off-black + one bright accent (electric blue, emerald, hot pink, etc.)
   233→  * **Palette-rotation rule:** if the previous premium-consumer project you generated used the beige+brass family, this one MUST use a different family. Do not ship the same warm-craft palette twice in a row.
   234→  * **Override:** the beige+brass+espresso palette is acceptable ONLY when the brand brief explicitly names those colors, or when the brand identity is genuinely vintage / artisan / warm-craft AND you can articulate why this specific palette fits this specific brand. Default-reaching for it because "this is a cookware brief" is banned.
   235→
   236→### 4.3 Layout Diversification
   237→* **ANTI-CENTER BIAS:** Centered Hero / H1 sections are avoided when `DESIGN_VARIANCE > 4`. Force "Split Screen" (50/50), "Left-aligned content / right-aligned asset", "Asymmetric white-space", or scroll-pinned structures.
   238→* **Override:** centered hero is OK for editorial / manifesto / launch-announcement briefs where the message itself is the design.
   239→
   240→### 4.4 Materiality, Shadows, Cards
   241→* Use cards ONLY when elevation communicates real hierarchy. Otherwise group with `border-t`, `divide-y`, or negative space.
   242→* When a shadow is used, tint it to the background hue. No pure-black drop shadows on light backgrounds.
   243→* For `VISUAL_DENSITY > 7`: generic card containers are banned. Data metrics breathe in plain layout.
   244→* **SHAPE CONSISTENCY LOCK (mandatory):** Pick ONE corner-radius scale for the page and stick to it. Options: all-sharp (radius 0), all-soft (radius 12-16px), all-pill (full radius for interactive). Mixed systems are allowed only when there is a documented rule (e.g. "buttons are full-pill, cards are 16px, inputs are 8px") and that rule is followed everywhere. Round buttons in a square layout, or square cards on a pill-button page, is broken design.
   245→
   246→### 4.5 Interactive UI States
   247→LLMs default to "static successful state only." Always implement full cycles:
   248→* **Loading:** Skeletal loaders matching the final layout's shape. Avoid generic circular spinners.
   249→* **Empty States:** Beautifully composed; indicate how to populate.
   250→* **Error States:** Clear, inline (forms), or contextual (toasts only for transient).
   251→* **Tactile Feedback:** On `:active`, use `-translate-y-[1px]` or `scale-[0.98]` to simulate a physical push.
   252→* **BUTTON CONTRAST CHECK (mandatory, a11y):** Before shipping any button, verify the button text is readable against the button background. White button + white text, `bg-white` CTA with `text-white` label, transparent button against the page background with no border → all banned. Audit every CTA: contrast ratio WCAG AA min (4.5:1 for body, 3:1 for large text 18px+). Same rule applies to ghost buttons over photographic backgrounds (use a backdrop, scrim, or stroke).
   253→* **CTA BUTTON WRAP BAN (mandatory):** Button text MUST fit on one line at desktop. If a label like "VIEW SELECTED WORK" wraps to 2 or 3 lines, the button is broken. Fix by EITHER shortening the label (3 words max for primary CTAs, ideally 1-2) OR widening the button (do not artificially constrain `max-width` on CTAs). Wrapped CTAs at desktop are a Pre-Flight Fail.
   254→* **NO DUPLICATE CTA INTENT (mandatory):** Two CTAs with the same intent on one page is a Pre-Flight Fail. Examples of same intent: "Get in touch" + "Contact us" + "Let's talk" + "Start a project" + "Start something" + "Reach out" = all "contact" intent → pick ONE label and use it everywhere on the page (nav, hero, footer). Same for "Try free" + "Get started" + "Sign up free" (all "signup" intent) and "View work" + "See selected work" + "Browse projects" (all "portfolio" intent). One label per intent.
   255→* **FORM CONTRAST CHECK (mandatory, a11y):** Form inputs, placeholder text, focus rings, helper text, and error text all pass WCAG AA contrast against the section background. Light placeholders on a near-white form, white form on white page section, form labels grayer than 4.5:1 contrast → all banned. Audit every form before shipping.
   256→
   257→### 4.6 Data & Form Patterns
   258→* Label ABOVE input. Helper text optional but present in markup. Error text BELOW input. Standard `gap-2` for input blocks.
   259→* No placeholder-as-label. Ever.
   260→
   261→### 4.7 Layout Discipline (Hard Rules. Failing any of these is shipping broken work)
   262→
   263→* **Hero MUST fit in the initial viewport.** Headline max 2 lines on desktop, subtext max **20 words** AND max 3-4 lines, CTAs visible without scroll. If the copy is too long: reduce font scale OR cut copy. If you cannot describe the value-prop in 20 words of subtext, the value-prop is unclear, not the rule too tight. Never let the hero overflow and force scroll to find the CTA.
   264→* **Hero font-scale discipline.** Plan font size and image size *together*. If the hero asset is large and the headline is more than 6 words, do not start at `text-7xl/text-8xl`. Default sensible range: `text-4xl md:text-5xl lg:text-6xl` for most heroes; `text-6xl md:text-7xl` only when the headline is 3-5 words. A 4-line hero headline is always a font-size error, never a copy-length error.
   265→* **HERO TOP PADDING CAP (mandatory):** Hero top padding max `pt-24` (≈6rem) at desktop. More than that means the hero content floats halfway down the viewport and reads as a layout bug, not as intentional space. If your hero needs more breathing room, increase font scale or asset size, not top padding.
   266→* **HERO STACK DISCIPLINE (max 4 text elements).** The hero is a single moment, not a feature list. Allowed text elements, max 4 in total:
   267→  1. Eyebrow (small uppercase label) OR brand strip OR neither - pick zero or one
   268→  2. Headline (max 2 lines, see above)
   269→  3. Subtext (max 20 words, max 4 lines)
   270→  4. CTAs (1 primary + max 1 secondary)
   271→  - **BANNED in the hero:** tiny tagline below CTAs ("Works with GitHub, GitLab, and self-hosted Git"), trust micro-strip ("Used by engineering teams at..."), pricing teaser ("Free for solo, $10/user for teams"), feature bullet list, social-proof avatar row. All of those move to dedicated sections directly below the hero.
   272→  - If you have an eyebrow AND a tagline below CTAs in the same hero, drop the tagline. If you have a brand strip AND a tagline, drop the tagline. One small text element per hero, max.
   273→* **"Used by" / "Trusted by" logo wall belongs UNDER the hero, never inside it.** The hero is for the value prop and primary CTA. The logo wall is a separate section directly below. Do not stuff trust logos into the same flex row as the hero copy.
   274→* **Navigation MUST render on a single line on desktop.** If items don't fit at `lg` (1024px), condense labels, drop secondary items, or move to a hamburger. A two-line nav at desktop is broken design.
   275→* **Navigation height cap: 80px max desktop, default 64-72px.** No huge "agency" nav bars that eat 15% of the viewport.
   276→* **Bento grids MUST have rhythm, not one-sided repetition.** Do not stack 6 left-image / right-text rows. Vary the composition: alternate full-width feature rows, asymmetric tile sizes, vertical breaks.
   277→* **BENTO CELL COUNT RULE (mandatory):** A bento grid has EXACTLY as many cells as you have content for. 3 items → 3 cells (1+2 split, or 2+1, or asymmetric trio). 5 items → 5 cells (2+3, 3+2, hero+4, etc.). If your grid has an empty cell in the middle or at the end, you planned wrong. Re-shape the grid; do not paste a blank tile.
   278→* **Section-Layout-Repetition Ban.** Once you use a layout family for a section (e.g., 3-column-image-cards, full-width-quote, split-text-image), that family can appear at most ONCE on the page. "Selected commissions" must not look like "What we do." A landing page with 8 sections must use at least 4 different layout families.
   279→* **ZIGZAG ALTERNATION CAP (mandatory).** Alternating "left-image + right-text" then "left-text + right-image" zigzag layout = banal. Max 2 sections in a row with this image+text-split pattern. The 3rd consecutive image+text split is a Pre-Flight Fail. Break the pattern with a full-width section, a vertical-stack section, a bento grid, a marquee, or a different layout family.
   280→* **EYEBROW RESTRAINT (mandatory, the #1 violated rule in production tests).** An "eyebrow" is the small uppercase wide-tracking label sitting above a section headline (e.g. `FOUR COLORWAYS`, `SELECTED WORK`, `THE HARDWARE`, `Git-native task management`). Typical CSS signature: `text-[11px] uppercase tracking-[0.18em]`, `font-mono text-[10.5px] uppercase tracking-[0.22em]`. Every AI-built site puts an eyebrow above EVERY section header, producing the same templated rhythm. Hard rule:
   281→  - **Maximum 1 eyebrow per 3 sections.** Hero counts as 1. So a page with 9 sections may use at most 3 eyebrows total.
   282→  - If section A has an eyebrow, the next 2 sections cannot have one.
   283→  - **Pre-Flight Check is mechanical:** count instances of `uppercase tracking` (or similar small-caps mono labels above headlines) across all section components. If count > ceil(sectionCount / 3), the output fails.
   284→  - **What to do instead of an eyebrow:** drop it entirely. The headline alone is enough. If you need to categorize a section, the section's location on the page already categorizes it; no label needed.
   285→* **SPLIT-HEADER BAN (mandatory).** The pattern "left big headline + right small explainer paragraph" as a section header (left col-span-7/8, right col-span-4/5 with a small body paragraph floating in the right column) is **banned as default**. Sections should have ONE focused message. If you genuinely need both a headline and an explainer paragraph, stack them vertically (headline on top, body below, max-width 65ch). Reach for the split-header pattern only when there is a real compositional reason (e.g., the right column carries a visual or interactive element, not just filler text).
   286→* **Bento Background Diversity (mandatory).** Bento and feature-grid sections cannot be 6 white-on-white cards with text inside. At least 2-3 cells in any multi-cell grid need real visual variation: a real image, a brand-appropriate gradient (not AI-purple), a pattern, a tinted background. A cream-on-cream bento with only typography inside reads as boring AI default, even when the rest of the page is good.
   287→* **Mobile collapse must be explicit per section.** For every multi-column layout, declare the `< 768px` fallback in the same component. No "it'll work, Tailwind handles it" assumptions.
   288→
   289→### 4.8 Image & Visual Asset Strategy
   290→
   291→Landing pages and portfolios are **visual products**. Text-only pages with fake-screenshot divs are slop.
   292→
   293→**Priority order for visual assets:**
   294→1. **Image-generation tool first.** If ANY image-gen tool is available in the environment (`generate_image`, MCP image tool, IDE-integrated gen, OpenAI image tools, etc.) you MUST use it to create section-specific assets: hero photography, product shots, texture backgrounds, mood images. Generate at the right aspect ratio for the section. Do not skip this step because hand-rolled CSS feels faster.
   295→2. **Real web images second.** When no gen tool is available, use real photography sources. Acceptable defaults:
   296→   * `https://picsum.photos/seed/{descriptive-seed}/{w}/{h}` for placeholder photography (seed should describe the section, e.g. `marrow-cookware-kitchen`)
   297→   * Actual stock or brand URLs when the brief provides them
   298→   * Open-license sources (Unsplash via direct URL, Pexels) if explicitly allowed
   299→3. **Last resort: tell the user.** If neither is possible, do NOT fill the page with hand-rolled SVG illustrations or div-based "fake screenshots." Instead, leave clearly-labeled placeholder slots (`<!-- TODO: hero product photo, 1600x1200 -->`) and at the end of the response say: *"This page needs real images at: \[list of placements\]. Please generate or provide them."*
   300→
   301→**Even minimalist sites need real images.** A pure-text page is not minimalism. It is incomplete work. Even an editorial Linear-style site needs at least 2-3 real images (hero, one product/lifestyle shot, one supporting image). Generate B&W minimalist photography if the brief is restrained; do not skip images entirely because the dial is low.
   302→
   303→**Real company logos for social proof.** When the brief calls for a "Trusted by / Used by / Customers" logo wall, do NOT default to plain text wordmarks (`<span>Acme Co</span>` styled in a row). Use real SVG logos:
   304→* **Source: Simple Icons** (`https://cdn.simpleicons.org/{slug}/ffffff` for any color, or `simple-icons` npm package). Covers most known brands.
   305→* **Alternative: devicon** for tech-stack logos (`@svgr/cli` or CDN).
   306→* **Make-up the brand name? Then make-up an SVG mark too.** Generate a simple monogram (one letter in a circle, two-letter ligature, abstract glyph) rendered as an inline `<svg>` matching the page style. Plain text wordmarks for invented brand names look generic.
   307→* **Always** ensure logos render in both light and dark mode (white-on-dark, black-on-light, or single-color theme variable).
   308→* **LOGO-ONLY rule (mandatory):** logo wall = logos and nothing else. Do NOT print industry / category labels below each logo (no `Vercel` + `hosting` underneath, no `Stripe` + `payments`, no `Cloudflare` + `infra`). The logo is the credibility, the label adds nothing the user does not already know. Optional: brand name as alt-text for screen readers, optional link to the brand's site. That is it.
   309→
   310→**Hand-rolled illustrations:**
   311→* SVG icons from libraries: fine (see Section 3.C).
   312→* Hand-rolled decorative SVGs (custom illustrations, logos, marks): **strongly discouraged**, never as default. Acceptable only when:
   313→  - The brief explicitly calls for it ("draw me an SVG logo")
   314→  - It's a single, simple geometric mark (a square, a circle, a wordmark in display type)
   315→  - You're confident in the output quality
   316→
   317→**Div-based fake screenshots are banned.** A "hand-built product preview" rendered with `<div>` rectangles, fake task lists, fake dashboards, fake terminal windows is a Tell. If you need to show a product:
   318→* Use a real screenshot URL if one exists
   319→* Generate one via image tool
   320→* Use a real component preview (an actual mini-version of the UI inside the page)
   321→* Or skip the preview entirely and use editorial photography
   322→
   323→**Hero needs a real visual.** Text + gradient blob is not a hero - it's a placeholder.
   324→
   325→### 4.9 Content Density
   326→
   327→Landing pages live on the **first impression**, not the full read. Cut ruthlessly.
   328→
   329→* **Default content shape per section:** short headline (≤ 8 words) + short sub-paragraph (≤ 25 words) + one visual asset OR one CTA. Anything more must be justified by the section's job.
   330→* **No data-dump sections.** A 20-row publication table, a 30-row award list, a giant pricing matrix on a marketing page = wrong layout. Use:
   331→  - Top 3-5 highlights + "View full list" link
   332→  - Marquee / carousel for breadth
   333→  - Different page entirely if the data is the product
   334→* **Long lists need a different UI component, not a longer list.** Default `<ul>` with bullets / `divide-y` rows is the lazy choice. If you have > 5 items, reach for one of these instead:
   335→  - 2-column split with grouped items
   336→  - Card grid with image + label per item
   337→  - Tabs / accordion if items are categorisable
   338→  - Horizontal scroll-snap pills
   339→  - Carousel for breadth-heavy lists (testimonials, logos, capabilities)
   340→  - Marquee for "lots-of-things-that-don't-need-individual-attention"
   341→  A spec sheet with 10 rows + a hairline under every row is the WORST default. Either group rows into 2-3 chunks with sparse dividers, or move to a card-per-spec layout.
   342→* **Spec sheets specifically (the Marrow-cookware pattern).** A long product specification table with `border-b` on every row is the AI default for cookware / hardware / apparel / artisan-goods briefs. Banned. Concrete alternatives:
   343→  - **2-col card grid:** each spec gets its own card with the spec name, the value (large display number), and a one-line "why it matters" body. Cards arranged 2-col on desktop, 1-col mobile.
   344→  - **Scroll-snap horizontal pills:** each spec is a pill, user can flick through.
   345→  - **Grouped chunks:** group 10 specs into 3 logical clusters (e.g. "Materials", "Cooking", "Warranty"), each cluster gets ONE soft divider and a cluster heading.
   346→  - **Featured-vs-rest:** 3-4 hero specs visualised as large display tiles, the rest collapsed under a "View full specifications" disclosure.
   347→
   348→* **COPY SELF-AUDIT (mandatory before ship):** Before declaring any task done, re-read every visible string on the page (headlines, subheads, eyebrows, button labels, body copy, captions, alt text, footer text, error messages). Flag any string that is:
   349→  - **Grammatically broken** ("free on its past", "two plans but one is honest", "to put it on the table" out of context)
   350→  - **Has unclear referents** ("we plan to stay that way" without prior context)
   351→  - **Sounds like AI hallucination** (cute-but-wrong wordplay, forced metaphors that don't track, "elegant nothing" phrases)
   352→  - **Reads like an LLM trying to sound thoughtful** (passive-aggressive humility, fake-craftsman labels, mock-poetic micro-meta)
   353→  Rewrite every flagged string. If unsure whether a string makes sense, replace it with a plain functional sentence. AI-generated cute copy is worse than boring copy.
   354→* **Fake-precise numbers are flagged.** Numbers like `92%`, `4.1×`, `48k`, `5.8 mm`, `13.4 lb` either:
   355→  - Come from real data (brief, brand guidelines, public metrics) - fine
   356→  - Are explicitly labeled as mock (`<!-- mock -->`, "example", "sample data") - fine
   357→  - Are AI-invented spec aesthetics - banned. Don't fake engineering precision the brand doesn't claim.
   358→* **One copy register per page.** Don't mix technical mono ("47 tasks · 0.6 ctx-switches/day"), editorial prose, and marketing punch in the same composition unless the brand voice explicitly calls for it.
   359→
   360→### 4.10 Quotes & Testimonials
   361→
   362→* **Max 3 lines** of quote body. Never 6. If the original quote is longer → cut it. A landing-page quote is a snippet, not the full review.
   363→* For very small font sizes (e.g. footer-style testimonials), the line cap can stretch slightly. Spirit: "fits in a glance."
   364→* **No em-dashes inside the quote text** as design flourish (long pauses, kinetic em-dashes, em-dash-bullets). See Section 9.G - em-dash is completely banned.
   365→* Attribution: name + role + (optionally) company. Never name only ("- Sarah").
   366→* Quote marks: use real typographic quotes ( " " ) or none at all. Not straight ASCII ( " ).
   367→
   368→### 4.11 Page Theme Lock (Light / Dark Mode Consistency)
   369→
   370→The page has ONE theme. Sections do not invert.
   371→
   372→* If the page is dark mode, ALL sections are dark mode. No light-mode-warm-paper section sandwiched between dark sections (or vice versa). The user must not feel they walked into a different website mid-scroll.
   373→* The exception: if the brief explicitly calls for a "Color Block Story" or "Theme Switch on Scroll" device AND that is a deliberate composition (one full theme switch with a strong transition, not random alternation), it is allowed once per page.
   374→* Default behaviour: pick light, dark, or auto (`prefers-color-scheme`) at the page level and lock it. Section-level background tints within the same theme family are fine (`bg-zinc-950` next to `bg-zinc-900`); flipping to `bg-amber-50` in the middle of a `bg-zinc-950` page is broken.
   375→* When using a design system with built-in theming (Radix Themes, shadcn/ui with `<Theme>`), set the theme ONCE in `layout.tsx` or the page root. Do not let individual sections override.

   379→## 5. CONTEXT-AWARE PROACTIVITY
   380→
   381→These are tools, not defaults. Use them when the design read calls for them. **None of these fire automatically.**
   382→
   383→* **Liquid Glass / Glassmorphism:** Appropriate for premium consumer, Apple-adjacent, luxury brand, or media-overlay vibes. Inappropriate for dashboards, public-sector, or "boring B2B." When used, go beyond `backdrop-blur`: add a 1px inner border (`border-white/10`) and a subtle inner shadow (`shadow-[inset_0_1px_0_rgba(255,255,255,0.1)]`) for physical edge refraction. Provide a solid-fill fallback under `prefers-reduced-transparency`.
   384→* **Magnetic Micro-physics:** Use when `MOTION_INTENSITY > 5` AND the brief reads premium / playful / agency. Implement EXCLUSIVELY with Motion's `useMotionValue` / `useTransform` outside the React render cycle. Never `useState`. See Section 3.B.
   385→* **Perpetual Micro-Interactions** (Pulse, Typewriter, Float, Shimmer, Carousel): Use when `MOTION_INTENSITY > 5` AND the section actively benefits from motion (status indicators, live feeds, AI-feel). **Not every card needs an infinite loop.** If a section is informational, leave it still. Apply Spring Physics (`type: "spring", stiffness: 100, damping: 20`) - no linear easing.
   386→* **"Motion claimed, motion shown."** If `MOTION_INTENSITY > 4`, the page must actually move: entry transitions on hero, scroll-reveal on key sections, hover physics on CTAs, at minimum. A static page that claims `MOTION_INTENSITY: 7` is broken. Conversely, if you cannot ship working motion in the available scope, drop the dial to 3 and ship a clean static page. Never half-build motion that breaks (cut-off ScrollTriggers, jumpy enters, missing cleanups).
   387→* **MOTION MUST BE MOTIVATED (mandatory).** Before adding any animation, ask: "what does this animation communicate?" Valid answers: hierarchy (drawing attention to the right thing), storytelling (revealing content in sequence that matches a narrative), feedback (acknowledging a user action), state transition (showing something changed). Invalid answer: "it looked cool". GSAP everywhere because GSAP is available is amateur. Each ScrollTrigger, each marquee, each pinned section needs a reason. If you cannot articulate the reason in one sentence, drop the animation.
   388→* **MARQUEE MAX-ONE-PER-PAGE (mandatory).** Horizontal scrolling text marquees ("logos endlessly scrolling", "manifesto scrolling sideways", "kinetic word strip") are appropriate at most ONCE per page. Two or more marquees on the same page reads as lazy filler. Pick the one section where the marquee actually serves the content; the others get a different layout.
   389→* **GSAP Sticky-Stack Pattern (when scroll-stack is used).** A "card stack on scroll" must be a REAL sticky-stack, not a sequential reveal list. See Section 5.A below for the canonical code skeleton. Common failure: trigger fires halfway through scroll instead of pinning at viewport top. Fix: `start: "top top"` not `start: "top center"` or `"top 80%"`.
   390→* **GSAP Horizontal-Pan Pattern (when horizontal scroll-hijack is used).** See Section 5.B below for the canonical skeleton. Common failure: animation starts before the section is pinned, so the user sees half a slide. Same fix: `start: "top top"`, pin the wrapper, scrub the inner track.
   391→
   392→### 5.A Sticky-Stack - Canonical Skeleton
   393→
   394→```tsx
   395→"use client";
   396→import { useRef, useEffect } from "react";
   397→import { gsap } from "gsap";
   398→import { ScrollTrigger } from "gsap/ScrollTrigger";
   399→import { useReducedMotion } from "motion/react";
   400→
   401→gsap.registerPlugin(ScrollTrigger);
   402→
   403→export function StickyStack({ cards }: { cards: React.ReactNode[] }) {
   404→  const ref = useRef<HTMLDivElement>(null);
   405→  const reduce = useReducedMotion();
   406→
   407→  useEffect(() => {
   408→    if (reduce || !ref.current) return;
   409→    const ctx = gsap.context(() => {
   410→      const cardEls = gsap.utils.toArray<HTMLElement>(".stack-card");
   411→      cardEls.forEach((card, i) => {
   412→        if (i === cardEls.length - 1) return;
   413→        ScrollTrigger.create({
   414→          trigger: card,
   415→          start: "top top",                              // pin at viewport top
   416→          endTrigger: cardEls[cardEls.length - 1],
   417→          end: "top top",
   418→          pin: true,
   419→          pinSpacing: false,
   420→        });
   421→        gsap.to(card, {
   422→          scale: 0.92,
   423→          opacity: 0.55,
   424→          ease: "none",
   425→          scrollTrigger: {
   426→            trigger: cardEls[i + 1],
   427→            start: "top bottom",
   428→            end: "top top",
   429→            scrub: true,
   430→          },
   431→        });
   432→      });
   433→    }, ref);
   434→    return () => ctx.revert();
   435→  }, [reduce]);
   436→
   437→  return (
   438→    <div ref={ref} className="relative">
   439→      {cards.map((card, i) => (
   440→        <div
   441→          key={i}
   442→          className="stack-card sticky top-0 min-h-[100dvh] flex items-center justify-center"
   443→        >
   444→          {card}
   445→        </div>
   446→      ))}
   447→    </div>
   448→  );
   449→}
   450→```
   451→
   452→Critical points: `start: "top top"`, `pin: true`, every card except the last is pinned, the scale/opacity transform is driven by the NEXT card's scroll trigger (so previous card shrinks as next one arrives).
   453→
   454→### 5.B Horizontal-Pan - Canonical Skeleton
   455→
   456→```tsx
   457→"use client";
   458→import { useRef, useEffect } from "react";
   459→import { gsap } from "gsap";
   460→import { ScrollTrigger } from "gsap/ScrollTrigger";
   461→import { useReducedMotion } from "motion/react";
   462→
   463→gsap.registerPlugin(ScrollTrigger);
   464→
   465→export function HorizontalPan({ children }: { children: React.ReactNode }) {
   466→  const wrap = useRef<HTMLDivElement>(null);
   467→  const track = useRef<HTMLDivElement>(null);
   468→  const reduce = useReducedMotion();
   469→
   470→  useEffect(() => {
   471→    if (reduce || !wrap.current || !track.current) return;
   472→    const ctx = gsap.context(() => {
   473→      const distance = track.current!.scrollWidth - window.innerWidth;
   474→      gsap.to(track.current, {
   475→        x: -distance,
   476→        ease: "none",
   477→        scrollTrigger: {
   478→          trigger: wrap.current,
   479→          start: "top top",                              // pin starts when section top hits viewport top
   480→          end: () => `+=${distance}`,                    // scroll distance = track width minus viewport
   481→          pin: true,
   482→          scrub: 1,
   483→          invalidateOnRefresh: true,
   484→        },
   485→      });
   486→    }, wrap);
   487→    return () => ctx.revert();
   488→  }, [reduce]);
   489→
   490→  return (
   491→    <section ref={wrap} className="relative overflow-hidden">
   492→      <div ref={track} className="flex h-[100dvh] items-center">
   493→        {children}
   494→      </div>
   495→    </section>
   496→  );
   497→}
   498→```
   499→
   500→Critical points: `start: "top top"`, `pin: true`, `end: "+=${distance}"` (scroll length = horizontal travel needed), `scrub: 1`. The wrapper is pinned, the inner track slides horizontally as the user scrolls vertically.
   501→
   502→### 5.C Scroll-Reveal Stagger - Canonical Skeleton (lighter alternative)
   503→
   504→For simple "items appear as they enter viewport" (no pinning), prefer Motion's `whileInView` over GSAP - lighter, no ScrollTrigger needed:
   505→
   506→```tsx
   507→"use client";
   508→import { motion, useReducedMotion } from "motion/react";
   509→
   510→export function RevealStagger({ items }: { items: string[] }) {
   511→  const reduce = useReducedMotion();
   512→  return (
   513→    <ul className="grid gap-6">
   514→      {items.map((item, i) => (
   515→        <motion.li
   516→          key={item}
   517→          initial={reduce ? false : { opacity: 0, y: 24 }}
   518→          whileInView={{ opacity: 1, y: 0 }}
   519→          viewport={{ once: true, amount: 0.3 }}
   520→          transition={{
   521→            duration: 0.6,
   522→            delay: i * 0.06,
   523→            ease: [0.16, 1, 0.3, 1],
   524→          }}
   525→        >
   526→          {item}
   527→        </motion.li>
   528→      ))}
   529→    </ul>
   530→  );
   531→}
   532→```
   533→
   534→Use this for: feature lists, testimonial grids, logo walls, anything that just needs "enter on scroll." Save GSAP for actual pin/scrub work.
   535→
   536→### 5.D Forbidden Animation Patterns
   537→
   538→* **`window.addEventListener("scroll", ...)`** is banned. It runs on every scroll frame, jank-prone, no batching. Use Motion's `useScroll()`, GSAP's `ScrollTrigger`, IntersectionObserver, or CSS `scroll-driven animations` (`animation-timeline: view()`).
   539→* **Custom scroll progress calculations using `window.scrollY`** in React state. Same reason. Re-renders on every frame.
   540→* **`requestAnimationFrame` loops that touch React state.** Use motion values (`useMotionValue` + `useTransform`) instead.
   541→* **Layout Transitions:** Use Motion's `layout` and `layoutId` props for visible state changes (re-ordering lists, expanding modals, shared elements between routes). Do not wrap static content in `layout` props "for safety" - it costs measurement work.
   542→* **Staggered Orchestration:** Use `staggerChildren` (Motion) or CSS cascade (`animation-delay: calc(var(--index) * 100ms)`) for reveal moments where sequence matters. For `staggerChildren`, parent (`variants`) and children MUST share the same Client Component tree.

   545→
   546→## 6. PERFORMANCE & ACCESSIBILITY GUARDRAILS
   547→
   548→### 6.A Hardware Acceleration
   549→* Animate ONLY `transform` and `opacity`. Never animate `top`, `left`, `width`, `height`.
   550→* Use `will-change: transform` sparingly - only on elements that will actually animate.
   551→
   552→### 6.B Reduced Motion (mandatory)
   553→* **Any motion above `MOTION_INTENSITY > 3` MUST honor `prefers-reduced-motion`.** This is non-negotiable.
   554→* In Motion: wrap with `useReducedMotion()` and degrade to static.
   555→* In CSS: gate animations behind `@media (prefers-reduced-motion: no-preference)` or provide an override block under `@media (prefers-reduced-motion: reduce)` that disables.
   556→* Infinite loops, parallax, scroll-hijack, and magnetic physics MUST collapse to static / instant under reduced motion.
   557→
   558→### 6.C Dark Mode (mandatory for any consumer-facing page)
   559→* Design for **both modes from the start**. Never ship light-only or dark-only without explicit user instruction.
   560→* Use Tailwind `dark:` variant OR CSS variables for tokens. Pick one strategy per project.
   561→* **Do not prescribe specific dark-mode colors here.** The brief decides. Maintain visual hierarchy, brand identity, and WCAG AA contrast (AAA for body) across both modes.
   562→* Respect `prefers-color-scheme: dark`. Default to system preference unless the brand insists on one mode.
   563→
   564→### 6.D Core Web Vitals Targets
   565→* **LCP** < 2.5s. Hero image must be `next/image priority` or preloaded.
   566→* **INP** < 200ms. Heavy work off main thread.
   567→* **CLS** < 0.1. Reserve space for images, fonts, embeds.
   568→* Run Lighthouse before declaring a page done.
   569→
   570→### 6.E DOM Cost
   571→* Apply grain / noise filters EXCLUSIVELY to fixed, `pointer-events-none` pseudo-elements (e.g., `fixed inset-0 z-[60] pointer-events-none`). NEVER on scrolling containers - continuous GPU repaints destroy mobile FPS.
   572→* Be aware of bundle size. Motion is not tiny. Three.js is large. Lazy-load anything that's not above-the-fold.
   573→
   574→### 6.F Z-Index Restraint
   575→NEVER spam arbitrary `z-50` or `z-10`. Use z-index strictly for systemic layer contexts (sticky navbars, modals, overlays, grain). Document the z-index scale in a project constants file.

   579→## 7. DIAL DEFINITIONS (Technical Reference)
   580→
   581→### DESIGN_VARIANCE (Level 1-10)
   582→* **1-3 (Predictable):** Symmetrical CSS Grid (12-col, equal fr-units), equal paddings, centered alignment.
   583→* **4-7 (Offset):** `margin-top: -2rem` overlaps, varied image aspect ratios (4:3 next to 16:9), left-aligned headers over center-aligned data.
   584→* **8-10 (Asymmetric):** Masonry layouts, CSS Grid with fractional units (`grid-template-columns: 2fr 1fr 1fr`), massive empty zones (`padding-left: 20vw`).
   585→* **MOBILE OVERRIDE:** For levels 4-10, asymmetric layouts above `md:` MUST collapse to strict single-column (`w-full`, `px-4`, `py-8`) on viewports `< 768px`.
   586→
   587→### MOTION_INTENSITY (Level 1-10)
   588→* **1-3 (Static):** No automatic animations. CSS `:hover` and `:active` states only. `prefers-reduced-motion` is the default mode anyway.
   589→* **4-7 (Fluid CSS):** `transition: all 0.3s cubic-bezier(0.16, 1, 0.3, 1)`. `animation-delay` cascades for load-ins. Focus on `transform` and `opacity`.
   590→* **8-10 (Advanced Choreography):** Complex scroll-triggered reveals, parallax, scroll-driven animation (CSS `animation-timeline` or GSAP ScrollTrigger). Use Motion hooks. **NEVER use `window.addEventListener('scroll')`** - it is a hard ban, not a "prefer-not." See Section 5.D for the allowed alternatives.
   591→
   592→### VISUAL_DENSITY (Level 1-10)
   593→* **1-3 (Art Gallery):** Lots of white space. Huge section gaps (`py-32` to `py-48`). Expensive, clean.
   594→* **4-7 (Daily App):** Standard web app spacing (`py-16` to `py-24`).
   595→* **8-10 (Cockpit):** Tight paddings. No card boxes; 1px lines separate data. Mandatory: `font-mono` for all numbers.

   599→## 8. DARK MODE PROTOCOL
   600→
   601→Dual-mode by default. Never assume light-only unless the brief is print-emulating editorial.
   602→
   603→### 8.A Token Strategy (pick one, stick to it)
   604→* **Tailwind `dark:` variant** (default for utility-first projects): every color utility paired with its dark variant (`bg-white dark:bg-zinc-950`, `text-gray-900 dark:text-gray-100`).
   605→* **CSS variables** (for shadcn/ui, Radix Themes, or component libraries with theming): define semantic tokens (`--surface`, `--surface-elevated`, `--text-primary`, `--accent`) and swap values under `[data-theme="dark"]` or `@media (prefers-color-scheme: dark)`.
   606→
   607→### 8.B Do Not Prescribe Specific Colors Here
   608→The brief and brand decide. This skill enforces only:
   609→* **Contrast** - WCAG AA minimum for body text, AAA target for hero copy.
   610→* **Hierarchy parity** - visual hierarchy that works in light must work in dark. If a CTA pops in light, it pops in dark.
   611→* **Brand fidelity** - primary brand color stays recognisable. Don't desaturate the brand into a dark mode.
   612→* **No pure `#000000` and no pure `#ffffff`** - use off-black (zinc-950, near-black warm gray) and off-white. Pure values kill depth.
   613→
   614→### 8.C Default Mode
   615→Respect `prefers-color-scheme` unless the brand insists. Add a manual toggle if either mode would lose key brand expression.
   616→
   617→### 8.D Test in Both Modes Before Finishing
   618→Open the page in both modes during development. Do not ship a page you've only seen in one mode.

   621→
   622→## 9. AI TELLS (Forbidden Patterns)
   623→
   624→Avoid these signatures unless the brief explicitly asks for them.
   625→
   626→### 9.A Visual & CSS
   627→* **NO neon / outer glows** by default. Use inner borders or subtle tinted shadows.
   628→* **NO pure black (`#000000`).** Off-black, zinc-950, or charcoal.
   629→* **NO oversaturated accents.** Desaturate to blend with neutrals.
   630→* **NO excessive gradient text** for large headers.
   631→* **NO custom mouse cursors.** Outdated, accessibility-hostile, perf-hostile.
   632→
   633→### 9.B Typography
   634→* **AVOID Inter as default.** See Section 4.1. Override path exists.
   635→* **NO oversized H1s** that just scream. Control hierarchy with weight + color, not raw scale.
   636→* **Serif constraints:** Serif for editorial / luxury / publication. Not for dashboards.
   637→
   638→### 9.C Layout & Spacing
   639→* **Mathematically perfect** padding and margins. No floating elements with awkward gaps.
   640→* **NO 3-column equal feature cards.** The generic "three identical cards horizontally" feature row is banned. Use 2-column zig-zag, asymmetric grid, scroll-pinned, or horizontal-scroll alternative.
   641→
   642→### 9.D Content & Data ("Jane Doe" Effect)
   643→* **NO generic names.** "John Doe", "Sarah Chan", "Jack Su" → use creative, realistic, locale-appropriate names.
   644→* **NO generic avatars.** No SVG "egg" or Lucide user icons → use believable photo placeholders or specific styling.
   645→* **NO fake-perfect numbers.** Avoid `99.99%`, `50%`, `1234567`. Use organic, messy data (`47.2%`, `+1 (312) 847-1928`).
   646→* **NO startup-slop brand names.** "Acme", "Nexus", "SmartFlow", "Cloudly" → invent contextual, premium names that sound real.
   647→* **NO filler verbs.** "Elevate", "Seamless", "Unleash", "Next-Gen", "Revolutionize" → concrete verbs only.
   648→
   649→### 9.E External Resources & Components
   650→* **NO hand-rolled SVG icons.** Use Phosphor / HugeIcons / Radix / Tabler. Lucide on explicit request only.
   651→* **Hand-rolled decorative SVGs strongly discouraged** as default (see Section 4.8).
   652→* **NO div-based fake screenshots.** Never build a fake product UI out of `<div>` rectangles to simulate a screenshot. Use real images, generated images, or skip the preview.
   653→* **NO broken Unsplash links.** Use `https://picsum.photos/seed/{descriptive-string}/{w}/{h}`, or generated photo placeholders, or actual assets.
   654→* **shadcn/ui customization:** Allowed, but NEVER in default state. Customize radii, colors, shadows, typography to the project aesthetic.
   655→* **Production-Ready Cleanliness:** Code visually clean, memorable, meticulously refined.
   656→
   657→### 9.F Production-Test Tells (banned outright)
   658→
   659→These patterns came out of real LLM-generated landing-page tests. They are the signatures the model defaults to when it tries to "look designed." Treat them as hard bans unless the brief explicitly calls for one.
   660→
   661→**Hero & top-of-page**
   662→* **NO version labels in the hero.** `V0.6`, `v2.0`, `BETA`, `INVITE-ONLY PREVIEW`, `EARLY ACCESS`, `ALPHA` - banned as default eyebrows. Only acceptable when the brief is explicitly about a product launch / preview status.
   663→* **NO "Brand · No. 01"-style sub-eyebrows.** "Marrow · No. 01 · The 6-quart" type micro-meta lines. Skip them.
   664→
   665→**Section numbering & micro-labels**
   666→* **NO section-number eyebrows.** `00 / INDEX`, `001 · Capabilities`, `002 · Featured commission`, `06 · how it works`, `05 · The honest table` - banned. Eyebrows should name the topic in plain language, not enumerate.
   667→* **NO `01 / 4`-style pagination on images or bento tiles.** If the user can count, they don't need the label.
   668→* **NO `Scroll · 001 Capabilities`-style scroll cues.** A simple arrow or "Scroll" is enough; no section-number prefix.
   669→* **NO "Index of Work, 2018 - 2026"-style range labels** as eyebrows. Just say what the section is.
   670→
   671→**Separators & dots**
   672→* **The middle-dot (`·`) is rationed.** Maximum 1 per line in metadata strips. Do NOT use it as the default separator for everything ("foo · bar · baz · qux · quux"). If you need a separator family, prefer line breaks, hairlines, or columns.
   673→* **NO decorative colored status dots on every list/nav/badge.** A colored dot before "ONE Q4 SLOT OPEN" or before every nav link, or every task row - banned by default. Acceptable only when the dot conveys actual semantic state (a server status, an availability flag) and is used sparingly.
   674→
   675→**Em-dashes & typography flourishes**
   676→* **NO em-dash (`—`) as a design element OR anywhere else.** See Section 9.G below for the complete, non-negotiable ban. The em-dash character is forbidden in headlines, eyebrows, pills, body copy, quotes, attribution, captions, button text, and alt text. Use the regular hyphen (`-`).
   677→* **NO `<br>`-broken-and-italicized headlines** as a default "design move." "for thirty\<br\>*years.*" type splits. Headlines should read naturally first, get clever only when the brief demands it.
   678→* **NO vertical rotated text** ("INDEX OF WORK, 2018 - 2026" rotated 90°). Agency-portfolio cliché. Use it only when the brief is explicitly agency / Awwwards / experimental AND it serves a real composition purpose.
   679→* **NO crosshair / hairline grid lines as decoration.** Vertical and horizontal lines drawn just to make the page "feel designed" - banned. Use them only when they organize real content.
   680→
   681→**Fake product previews**
   682→* **NO div-based fake product UI in the hero** (fake task list, fake terminal, fake dashboard built from styled divs). It is the #1 LLM-design Tell. Use a real screenshot, a generated image, a real component preview, or none at all.
   683→* **NO fake version footers** ("v0.6.2-rc.1", "last sync 4s ago · main") inside fake screenshots. Adds nothing, screams AI.
   684→
   685→**Marketing-copy Tells**
   686→* **NO "Quietly in use at" / "Quietly trusted by"** social-proof headers. Use natural language: "Trusted by", "Used at", "Customers include", or skip the heading entirely if the logos speak.
   687→* **NO "From the field" / "Field notes" / "Currently on the bench" / "On our desks" / "Loose plates" style poetic labels** on quote, blog, or sidebar sections. Reads as performative-craftsman. Use plain functional labels ("Testimonials", "Latest writing", "Now working on") or skip the label.
   688→* **NO "We respect the French ones"-style** mock-humble industry-references in body copy. Cute and AI-y.
   689→* **NO weather / locale strips** ("LIS 14:23 · 18°C") in headers/footers unless the brief is explicitly about a place / time-zone-distributed studio.
   690→* **NO micro-meta-sentences under eyebrows.** Sentences like *"Each of these is a feature we ship today, not a roadmap promise. The list will stay short on purpose."* sitting under a section heading are clutter. Eyebrow + Headline + Body is enough.
   691→* **NO generic step labels.** "Stage 1 / Stage 2 / Stage 3", "Step 1 / Step 2 / Step 3", "Phase 01 / Phase 02 / Phase 03", "Pass One / Pass Two / Pass Three". Banned. The actual step content is the label. If you must show progression, use the verb-noun directly ("Install", "Configure", "Ship") not "Stage 1: Install".
   692→
   693→**Pills, labels and version stamps**
   694→* **NO pills/labels/tags overlaid on images.** No `<span>` overlays on photos with tags like `Brand · 02`, `PLATE · BRAND`, `Field notes - journal`. Either let the image speak alone, or add a caption directly below (outside the image).
   695→* **NO photo-credit captions as decoration.** Strings like `Field study no. 12 · Ines Caetano`, `Plate 03 · House archive`, `Frame XII · 35mm` under stock/picsum images are pretentious. Photo credit is allowed ONLY when there is a real photographer being credited for a real photo (with permission). Otherwise: skip the caption or use a one-line functional caption ("The 6-quart, in Sage.").
   696→* **NO version footers on marketing pages.** Footer strings like `v1.4.2`, `Build 0048`, `last sync 4s ago · main` are CLI / devtool fixtures, not landing-page content. Banned on marketing/landing/portfolio pages.
   697→* **NO "Reservation 412 of 800"-style live-stock counters** as decoration. Only if the brief is explicitly a limited-run waitlist with real data.
   698→
   699→**Decoration text strips**
   700→* **NO decoration text strip at hero bottom.** Patterns like `BRAND. MOTION. SPATIAL.`, `TYPE / FORM / MOTION`, `DESIGN · BUILD · SHIP`, `ESTD. 2018 · LISBON · BRAND. MOTION. SPATIAL.` as a small mono-caps strip across the bottom of the hero are an agency-portfolio cliché. Banned by default. Only acceptable when the strip carries real, navigable links (sticky bottom nav) or real status info (cookie banner, build info on a docs site).
   701→* **NO floating top-right sub-text in section headings.** Pattern: section has a giant left-aligned headline; in the top-right corner of the same section header there is a small explainer paragraph floating with no clear alignment to anything else. That floater is the Tell. Either put the sub-text directly under the headline, or build a clean 2-column header (left: headline, right: aligned body), but not a tiny corner paragraph.
   702→
   703→**Lists, dividers and scoring**
   704→* **NO `border-t` + `border-b` on every row of a long list / spec table.** Pick one (bottom-border between rows OR top-border above the group) and use it sparsely. A 10-row spec table with hairlines under each row is the laziest layout - see Section 4.9 for alternative UI components.
   705→* **NO scoring/progress bars with filled background tracks** as comparison visuals. If you need to show "X out of Y" comparisons, prefer a number + small icon, or a tiny inline bar WITHOUT a background track. Big filled `bg-zinc-200` tracks with a partial fill on top are dashboard-UI clutter on a landing page.
   706→
   707→**Locale, time, scroll cues**
   708→* **Locale / city-name / time / weather strips are banned for 99% of briefs.** "Lisbon, working with founders" in the hero, "1200-690 Lisbon, Portugal" in the footer, "Lisbon 14:23 · 18°C" in the nav. These are agency-portfolio decoration tells. Allowed ONLY when: the brief explicitly describes a globally-distributed studio with timezone-relevant work, OR a travel-focused brand, OR a real-world physical venue. A single contact-address mention in the footer is fine; an atmospheric locale strip is not.
   709→* **Scroll cues are banned.** `Scroll`, `↓ scroll`, `Scroll to explore`, `Scroll to walk through it`, animated mouse-wheel icons. If the user has not scrolled yet, they are looking at the hero. They know what scroll is. The bottom of the viewport does not need a label.
   710→* **ZERO decorative status dots by default.** A coloured dot before nav items, before list rows, before badges, before status labels is a Tell. Only acceptable when conveying real semantic state (a live indicator on actual server status, a live availability flag) and limited to one per page section.
   711→
   712→### 9.G EM-DASH BAN (the single most-violated Tell)
   713→
   714→**Em-dash (`—`) is COMPLETELY banned.** It is the LLM's signature stylistic crutch and it is the #1 visual Tell in production tests. There is no "limited use" allowance, no "natural language frequency" allowance, no "in body copy is fine" allowance. None.
   715→
   716→* **Banned in headlines.** Use a period or a comma.
   717→* **Banned in eyebrows / labels / pills / button text / image captions / nav items.** Replace with line breaks, columns, or hairlines.
   718→* **Banned in body copy.** Restructure the sentence: two sentences with a period, OR a comma, OR parentheses, OR a colon.
   719→* **Banned in quote attribution.** Use a normal hyphen with spaces (` - `) or a line break + smaller-weight name.
   720→* **Banned in en-dash form too (`–`) when used as a separator.** Date ranges (`2018-2026`) use a hyphen. Number ranges (`€40-80k`) use a hyphen.
   721→
   722→The ONLY permitted dash characters on the page are:
   723→* Regular hyphen `-` (for compound words, ranges, line dividers in markup)
   724→* Minus sign in math (`-5°C`)
   725→
   726→If your output contains a single `—` or `–` anywhere visible to the user, the output fails the Pre-Flight Check and must be rewritten.
   727→
   728→This rule is non-negotiable. The agent has historically ignored em-dash limits when phrased as "use sparingly." The phrasing here is binary: zero em-dashes.

   810→## 11. REDESIGN PROTOCOL
   811→
   812→This skill handles **greenfield builds AND redesigns**. Misclassifying the mode is the single biggest source of bad redesign output.
   813→
   814→### 11.A Detect the Mode (first action)
   815→* **Greenfield** - no existing site, or full overhaul approved. Dial baseline from Section 1.
   816→* **Redesign - Preserve** - modernise without breaking the brand. Audit first, extract brand tokens, evolve gradually.
   817→* **Redesign - Overhaul** - new visual language on top of existing content. Treat as greenfield for visuals; preserve content and IA.
   818→
   819→If ambiguous, ask **once**: *"Should this redesign preserve the existing brand, or are we starting visually from scratch?"*
   820→
   821→### 11.B Audit Before Touching
   822→Document the current state before proposing changes:
   823→* **Brand tokens** - primary / accent colors, type stack, logo treatment, radii.
   824→* **Information architecture** - page tree, primary nav, key conversion paths.
   825→* **Content blocks** - what exists, what's doing work, what's filler.
   826→* **Patterns to preserve** - signature interactions, recognisable hero, copy voice.
   827→* **Patterns to retire** - AI-slop tells, broken layouts, dead links, generic stock imagery, perf traps.
   828→* **Dial reading of the existing site** - infer current `DESIGN_VARIANCE` / `MOTION_INTENSITY` / `VISUAL_DENSITY`. That's your starting point, not the baseline.
   829→* **SEO baseline** - current ranking pages, meta titles, structured data, OG cards. **SEO migration is the #1 redesign risk.**
   830→
   831→### 11.C Preservation Rules
   832→* **Do not change information architecture** unless asked. Keep page slugs, anchor IDs, primary nav labels stable for SEO and muscle memory.
   833→* **Extract brand colors before applying Section 4.2.** A brand that is already purple stays purple - apply the LILA RULE's override.
   834→* **Preserve copy voice** unless asked for a rewrite. Visual modernisation ≠ content rewrite.
   835→* **Honor existing accessibility wins.** Do not regress focus states, alt text, keyboard nav, contrast.
   836→* **Respect existing analytics events.** Do not rename buttons, form fields, section IDs that downstream tracking depends on.
   837→
   838→### 11.D Modernisation Levers (priority order)
   839→Apply in order - stop when the brief is satisfied:
   840→1. **Typography refresh** - biggest visual lift per unit of risk.
   841→2. **Spacing & rhythm** - increase section padding, fix vertical rhythm.
   842→3. **Color recalibration** - desaturate, unify neutrals, keep brand accent.
   843→4. **Motion layer** - add `MOTION_INTENSITY`-appropriate micro-interactions to existing components.
   844→5. **Hero & key-section recomposition** - restructure top-of-funnel using Section 10 vocabulary.
   845→6. **Full block replacement** - only when the existing block is unsalvageable.
   846→
   847→### 11.E Decision Tree: Targeted Evolution vs Full Redesign
   848→* IA, content, and SEO sound → **targeted evolution** (Levers 1-4). ~70% of value at ~40% of risk.
   849→* Visual debt is structural (broken IA, no design system, broken mobile) → **full redesign** with strict content preservation.
   850→* Brand itself is changing → **greenfield**.
   851→
   852→### 11.F What Never Changes Silently
   853→Never modify without explicit user approval:
   854→* URL structure / route slugs.
   855→* Primary nav labels.
   856→* Form field names or order (breaks analytics + autofill).
   857→* Brand logo or wordmark.
   858→* Existing legal / consent / cookie copy.

  1140→## Appendix C - Apple Liquid Glass: Honest Web Approximation
  1141→
  1142→Do **not** treat random CSS snippets as official Apple Liquid Glass.
  1143→
  1144→### What is official
  1145→Apple documents Liquid Glass inside Apple's Human Interface Guidelines and Developer Documentation for **Apple platforms**. It is a dynamic material used across Apple platform UI. Apple's native implementation belongs to Apple platform APIs and system components, **not a public web CSS package**.
  1146→
  1147→Relevant official docs:
  1148→- Apple Human Interface Guidelines → Materials
  1149→- Apple Developer Documentation → Liquid Glass
  1150→- Apple Developer Documentation → Adopting Liquid Glass
  1151→- SwiftUI → Material
  1152→
  1153→### What is NOT official
  1154→There is no `liquid-glass.css` from Apple for normal websites.
  1155→
  1156→A web approximation can use:
  1157→- `backdrop-filter`
  1158→- transparent backgrounds
  1159→- layered borders
  1160→- highlight overlays
  1161→- gradients
  1162→- motion
  1163→- strong contrast fallbacks
  1164→
  1165→But that is **web glassmorphism / frosted-glass approximation**, not official Apple Liquid Glass. Label it as such in comments.
  1166→
