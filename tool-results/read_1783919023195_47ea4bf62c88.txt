    45Ôćĺ## Core Philosophy
    46Ôćĺ
    47Ôćĺ### Taste is trained, not innate
    48Ôćĺ
    49ÔćĺGood taste is not personal preference. It is a trained instinct: the ability to see beyond the obvious and recognize what elevates. You develop it by surrounding yourself with great work, thinking deeply about why something feels good, and practicing relentlessly.
    50Ôćĺ
    51ÔćĺWhen building UI, don't just make it work. Study why the best interfaces feel the way they do. Reverse engineer animations. Inspect interactions. Be curious.
    52Ôćĺ
    53Ôćĺ### Unseen details compound
    54Ôćĺ
    55ÔćĺMost details users never consciously notice. That is the point. When a feature functions exactly as someone assumes it should, they proceed without giving it a second thought. That is the goal.
    56Ôćĺ
    57Ôćĺ> "All those unseen details combine to produce something that's just stunning, like a thousand barely audible voices all singing in tune." - Paul Graham
    58Ôćĺ
    59ÔćĺEvery decision below exists because the aggregate of invisible correctness creates interfaces people love without knowing why.
    60Ôćĺ
    61Ôćĺ### Beauty is leverage
    62Ôćĺ
    63ÔćĺPeople select tools based on the overall experience, not just functionality. Good defaults and good animations are real differentiators. Beauty is underutilized in software. Use it as leverage to stand out.

    89Ôćĺ## The Animation Decision Framework
    90Ôćĺ
    91ÔćĺBefore writing any animation code, answer these questions in order:
    92Ôćĺ
    93Ôćĺ### 1. Should this animate at all?
    94Ôćĺ
    95Ôćĺ**Ask:** How often will users see this animation?
    96Ôćĺ
    97Ôćĺ| Frequency                                                   | Decision                     |
    98Ôćĺ| ----------------------------------------------------------- | ---------------------------- |
    99Ôćĺ| 100+ times/day (keyboard shortcuts, command palette toggle) | No animation. Ever.          |
   100Ôćĺ| Tens of times/day (hover effects, list navigation)          | Remove or drastically reduce |
   101Ôćĺ| Occasional (modals, drawers, toasts)                        | Standard animation           |
   102Ôćĺ| Rare/first-time (onboarding, feedback forms, celebrations)  | Can add delight              |
   103Ôćĺ
   104Ôćĺ**Never animate keyboard-initiated actions.** These actions are repeated hundreds of times daily. Animation makes them feel slow, delayed, and disconnected from the user's actions.
   105Ôćĺ
   106ÔćĺRaycast has no open/close animation. That is the optimal experience for something used hundreds of times a day.
   107Ôćĺ
   108Ôćĺ### 2. What is the purpose?
   109Ôćĺ
   110ÔćĺEvery animation must have a clear answer to "why does this animate?"
   111Ôćĺ
   112ÔćĺValid purposes:
   113Ôćĺ
   114Ôćĺ- **Spatial consistency**: toast enters and exits from the same direction, making swipe-to-dismiss feel intuitive
   115Ôćĺ- **State indication**: a morphing feedback button shows the state change
   116Ôćĺ- **Explanation**: a marketing animation that shows how a feature works
   117Ôćĺ- **Feedback**: a button scales down on press, confirming the interface heard the user
   118Ôćĺ- **Preventing jarring changes**: elements appearing or disappearing without transition feel broken
   119Ôćĺ
   120ÔćĺIf the purpose is just "it looks cool" and the user will see it often, don't animate.
   121Ôćĺ
   122Ôćĺ### 3. What easing should it use?
   123Ôćĺ
   124ÔćĺIs the element entering or exiting?
   125Ôćĺ  Yes Ôćĺ ease-out (starts fast, feels responsive)
   126Ôćĺ  No Ôćĺ
   127Ôćĺ    Is it moving/morphing on screen?
   128Ôćĺ      Yes Ôćĺ ease-in-out (natural acceleration/deceleration)
   129Ôćĺ    Is it a hover/color change?
   130Ôćĺ      Yes Ôćĺ ease
   131Ôćĺ    Is it constant motion (marquee, progress bar)?
   132Ôćĺ      Yes Ôćĺ linear
   133Ôćĺ    Default Ôćĺ ease-out
   134Ôćĺ
   135Ôćĺ**Critical: use custom easing curves.** The built-in CSS easings are too weak. They lack the punch that makes animations feel intentional.
   136Ôćĺ
   137Ôćĺ```css
   138Ôćĺ/* Strong ease-out for UI interactions */
   139Ôćĺ--ease-out: cubic-bezier(0.23, 1, 0.32, 1);
   140Ôćĺ
   141Ôćĺ/* Strong ease-in-out for on-screen movement */
   142Ôćĺ--ease-in-out: cubic-bezier(0.77, 0, 0.175, 1);
   143Ôćĺ
   144Ôćĺ/* iOS-like drawer curve (from Ionic Framework) */
   145Ôćĺ--ease-drawer: cubic-bezier(0.32, 0.72, 0, 1);
   146Ôćĺ```
   147Ôćĺ
   148Ôćĺ**Never use ease-in for UI animations.** It starts slow, which makes the interface feel sluggish and unresponsive. A dropdown with `ease-in` at 300ms _feels_ slower than `ease-out` at the same 300ms, because ease-in delays the initial movement ÔÇö the exact moment the user is watching most closely.
   149Ôćĺ
   150Ôćĺ**Easing curve resources:** Don't create curves from scratch. Use [easing.dev](https://easing.dev/) or [easings.co](https://easings.co/) to find stronger custom variants of standard easings.
   151Ôćĺ
   152Ôćĺ### 4. How fast should it be?
   153Ôćĺ
   154Ôćĺ| Element                  | Duration      |
   155Ôćĺ| ------------------------ | ------------- |
   156Ôćĺ| Button press feedback    | 100-160ms     |
   157Ôćĺ| Tooltips, small popovers | 125-200ms     |
   158Ôćĺ| Dropdowns, selects       | 150-250ms     |
   159Ôćĺ| Modals, drawers          | 200-500ms     |
   160Ôćĺ| Marketing/explanatory    | Can be longer |
   161Ôćĺ
   162Ôćĺ**Rule: UI animations should stay under 300ms.** A 180ms dropdown feels more responsive than a 400ms one. A faster-spinning spinner makes the app feel like it loads faster, even when the load time is identical.
   163Ôćĺ
   164Ôćĺ### Perceived performance
   165Ôćĺ
   166ÔćĺSpeed in animation is not just about feeling snappy ÔÇö it directly affects how users perceive your app's performance:
   167Ôćĺ
   168Ôćĺ- A **fast-spinning spinner** makes loading feel faster (same load time, different perception)
   169Ôćĺ- A **180ms select** animation feels more responsive than a **400ms** one
   170Ôćĺ- **Instant tooltips** after the first one is open (skip delay + skip animation) make the whole toolbar feel faster
   171Ôćĺ
   172ÔćĺThe perception of speed matters as much as actual speed. Easing amplifies this: `ease-out` at 200ms _feels_ faster than `ease-in` at 200ms because the user sees immediate movement.
   173Ôćĺ
   174Ôćĺ## Spring Animations
   175Ôćĺ
   176ÔćĺSprings feel more natural than duration-based animations because they simulate real physics. They don't have fixed durations ÔÇö they settle based on physical parameters.
   177Ôćĺ
   178Ôćĺ### When to use springs
   179Ôćĺ
   180Ôćĺ- Drag interactions with momentum
   181Ôćĺ- Elements that should feel "alive" (like Apple's Dynamic Island)
   182Ôćĺ- Gestures that can be interrupted mid-animation
   183Ôćĺ- Decorative mouse-tracking interactions
   184Ôćĺ
   185Ôćĺ### Spring-based mouse interactions
   186Ôćĺ
   187ÔćĺTying visual changes directly to mouse position feels artificial because it lacks motion. Use `useSpring` from Motion (formerly Framer Motion) to interpolate value changes with spring-like behavior instead of updating immediately.
   188Ôćĺ
   189Ôćĺ```jsx
   190Ôćĺimport { useSpring } from 'framer-motion';
   191Ôćĺ
   192Ôćĺ// Without spring: feels artificial, instant
   193Ôćĺconst rotation = mouseX * 0.1;
   194Ôćĺ
   195Ôćĺ// With spring: feels natural, has momentum
   196Ôćĺconst springRotation = useSpring(mouseX * 0.1, {
   197Ôćĺ  stiffness: 100,
   198Ôćĺ  damping: 10,
   199Ôćĺ});
   200Ôćĺ```
   201Ôćĺ
   202ÔćĺThis works because the animation is **decorative** ÔÇö it doesn't serve a function. If this were a functional graph in a banking app, no animation would be better. Know when decoration helps and when it hinders.
   203Ôćĺ
   204Ôćĺ### Spring configuration
   205Ôćĺ
   206Ôćĺ**Apple's approach (recommended ÔÇö easier to reason about):**
   207Ôćĺ
   208Ôćĺ```js
   209Ôćĺ{ type: "spring", duration: 0.5, bounce: 0.2 }
   210Ôćĺ```
   211Ôćĺ
   212Ôćĺ**Traditional physics (more control):**
   213Ôćĺ
   214Ôćĺ```js
   215Ôćĺ{ type: "spring", mass: 1, stiffness: 100, damping: 10 }
   216Ôćĺ```
   217Ôćĺ
   218ÔćĺKeep bounce subtle (0.1-0.3) when used. Avoid bounce in most UI contexts. Use it for drag-to-dismiss and playful interactions.
   219Ôćĺ
   220Ôćĺ### Interruptibility advantage
   221Ôćĺ
   222ÔćĺSprings maintain velocity when interrupted ÔÇö CSS animations and keyframes restart from zero. This makes springs ideal for gestures users might change mid-motion. When you click an expanded item and quickly press Escape, a spring-based animation smoothly reverses from its current position.
   223Ôćĺ
   224Ôćĺ## Component Building Principles
   225Ôćĺ
   226Ôćĺ### Buttons must feel responsive
   227Ôćĺ
   228ÔćĺAdd `transform: scale(0.97)` on `:active`. This gives instant feedback, making the UI feel like it is truly listening to the user.
   229Ôćĺ
   230Ôćĺ```css
   231Ôćĺ.button {
   232Ôćĺ  transition: transform 160ms ease-out;
   233Ôćĺ}
   234Ôćĺ
   235Ôćĺ.button:active {
   236Ôćĺ  transform: scale(0.97);
   237Ôćĺ}
   238Ôćĺ```
   239Ôćĺ
   240ÔćĺThis applies to any pressable element. The scale should be subtle (0.95-0.98).
   241Ôćĺ
   242Ôćĺ### Never animate from scale(0)
   243Ôćĺ
   244ÔćĺNothing in the real world disappears and reappears completely. Elements animating from `scale(0)` look like they come out of nowhere.
   245Ôćĺ
   246ÔćĺStart from `scale(0.9)` or higher, combined with opacity. Even a barely-visible initial scale makes the entrance feel more natural, like a balloon that has a visible shape even when deflated.
   247Ôćĺ
   248Ôćĺ```css
   249Ôćĺ/* Bad */
   250Ôćĺ.entering {
   251Ôćĺ  transform: scale(0);
   252Ôćĺ}
   253Ôćĺ
   254Ôćĺ/* Good */
   255Ôćĺ.entering {
   256Ôćĺ  transform: scale(0.95);
   257Ôćĺ  opacity: 0;
   258Ôćĺ}
   259Ôćĺ```
   260Ôćĺ
   261Ôćĺ### Make popovers origin-aware
   262Ôćĺ
   263ÔćĺPopovers should scale in from their trigger, not from center. The default `transform-origin: center` is wrong for almost every popover. **Exception: modals.** Modals should keep `transform-origin: center` because they are not anchored to a specific trigger ÔÇö they appear centered in the viewport.
   264Ôćĺ
   265Ôćĺ```css
   266Ôćĺ/* Radix UI */
   267Ôćĺ.popover {
   268Ôćĺ  transform-origin: var(--radix-popover-content-transform-origin);
   269Ôćĺ}
   270Ôćĺ
   271Ôćĺ/* Base UI */
   272Ôćĺ.popover {
   273Ôćĺ  transform-origin: var(--transform-origin);
   274Ôćĺ}
   275Ôćĺ```
   276Ôćĺ
   277ÔćĺWhether the user notices the difference individually does not matter. In the aggregate, unseen details become visible. They compound.
   278Ôćĺ
   279Ôćĺ### Tooltips: skip delay on subsequent hovers
   280Ôćĺ
   281ÔćĺTooltips should delay before appearing to prevent accidental activation. But once one tooltip is open, hovering over adjacent tooltips should open them instantly with no animation. This feels faster without defeating the purpose of the initial delay.
   282Ôćĺ
   283Ôćĺ```css
   284Ôćĺ.tooltip {
   285Ôćĺ  transition: transform 125ms ease-out, opacity 125ms ease-out;
   286Ôćĺ  transform-origin: var(--transform-origin);
   287Ôćĺ}
   288Ôćĺ
   289Ôćĺ.tooltip[data-starting-style],
   290Ôćĺ.tooltip[data-ending-style] {
   291Ôćĺ  opacity: 0;
   292Ôćĺ  transform: scale(0.97);
   293Ôćĺ}
   294Ôćĺ
   295Ôćĺ/* Skip animation on subsequent tooltips */
   296Ôćĺ.tooltip[data-instant] {
   297Ôćĺ  transition-duration: 0ms;
   298Ôćĺ}
   299Ôćĺ```
   300Ôćĺ
   301Ôćĺ### Use CSS transitions over keyframes for interruptible UI
   302Ôćĺ
   303ÔćĺCSS transitions can be interrupted and retargeted mid-animation. Keyframes restart from zero. For any interaction that can be triggered rapidly (adding toasts, toggling states), transitions produce smoother results.
   304Ôćĺ
   305Ôćĺ```css
   306Ôćĺ/* Interruptible - good for UI */
   307Ôćĺ.toast {
   308Ôćĺ  transition: transform 400ms ease;
   309Ôćĺ}
   310Ôćĺ
   311Ôćĺ/* Not interruptible - avoid for dynamic UI */
   312Ôćĺ@keyframes slideIn {
   313Ôćĺ  from {
   314Ôćĺ    transform: translateY(100%);
   315Ôćĺ  }
   316Ôćĺ  to {
   317Ôćĺ    transform: translateY(0);
   318Ôćĺ  }
   319Ôćĺ}
   320Ôćĺ```
   321Ôćĺ
   322Ôćĺ### Use blur to mask imperfect transitions
   323Ôćĺ
   324ÔćĺWhen a crossfade between two states feels off despite trying different easings and durations, add subtle `filter: blur(2px)` during the transition.
   325Ôćĺ
   326Ôćĺ**Why blur works:** Without blur, you see two distinct objects during a crossfade ÔÇö the old state and the new state overlapping. This looks unnatural. Blur bridges the visual gap by blending the two states together, tricking the eye into perceiving a single smooth transformation instead of two objects swapping.
   327Ôćĺ
   328ÔćĺCombine blur with scale-on-press (`scale(0.97)`) for a polished button state transition:
   329Ôćĺ
   330Ôćĺ```css
   331Ôćĺ.button {
   332Ôćĺ  transition: transform 160ms ease-out;
   333Ôćĺ}
   334Ôćĺ
   335Ôćĺ.button:active {
   336Ôćĺ  transform: scale(0.97);
   337Ôćĺ}
   338Ôćĺ
   339Ôćĺ.button-content {
   340Ôćĺ  transition: filter 200ms ease, opacity 200ms ease;
   341Ôćĺ}
   342Ôćĺ
   343Ôćĺ.button-content.transitioning {
   344Ôćĺ  filter: blur(2px);
   345Ôćĺ  opacity: 0.7;
   346Ôćĺ}
   347Ôćĺ```
   348Ôćĺ
   349ÔćĺKeep blur under 20px. Heavy blur is expensive, especially in Safari.
   350Ôćĺ
   351Ôćĺ### Animate enter states with @starting-style
   352Ôćĺ
   353ÔćĺThe modern CSS way to animate element entry without JavaScript:
   354Ôćĺ
   355Ôćĺ```css
   356Ôćĺ.toast {
   357Ôćĺ  opacity: 1;
   358Ôćĺ  transform: translateY(0);
   359Ôćĺ  transition: opacity 400ms ease, transform 400ms ease;
   360Ôćĺ
   361Ôćĺ  @starting-style {
   362Ôćĺ    opacity: 0;
   363Ôćĺ    transform: translateY(100%);
   364Ôćĺ  }
   365Ôćĺ}
   366Ôćĺ```
   367Ôćĺ
   368ÔćĺThis replaces the common React pattern of using `useEffect` to set `mounted: true` after initial render. Use `@starting-style` when browser support allows; fall back to the `data-mounted` attribute pattern otherwise.
   369Ôćĺ
   370Ôćĺ```jsx
   371Ôćĺ// Legacy pattern (still works everywhere)
   372ÔćĺuseEffect(() => {
   373Ôćĺ  setMounted(true);
   374Ôćĺ}, []);
   375Ôćĺ// <div data-mounted={mounted}>
   376Ôćĺ```
   377Ôćĺ
   378Ôćĺ## CSS Transform Mastery
   379Ôćĺ
   380Ôćĺ### translateY with percentages
   381Ôćĺ
   382ÔćĺPercentage values in `translate()` are relative to the element's own size. Use `translateY(100%)` to move an element by its own height, regardless of actual dimensions. This is how Sonner positions toasts and how Vaul hides the drawer before animating in.
   383Ôćĺ
   384Ôćĺ```css
   385Ôćĺ/* Works regardless of drawer height */
   386Ôćĺ.drawer-hidden {
   387Ôćĺ  transform: translateY(100%);
   388Ôćĺ}
   389Ôćĺ
   390Ôćĺ/* Works regardless of toast height */
   391Ôćĺ.toast-enter {
   392Ôćĺ  transform: translateY(-100%);
   393Ôćĺ}
   394Ôćĺ```
   395Ôćĺ
   396ÔćĺPrefer percentages over hardcoded pixel values. They are less error-prone and adapt to content.
   397Ôćĺ
   398Ôćĺ### scale() scales children too
   399Ôćĺ
   400ÔćĺUnlike `width`/`height`, `scale()` also scales an element's children. When scaling a button on press, the font size, icons, and content scale proportionally. This is a feature, not a bug.
   401Ôćĺ
   402Ôćĺ### 3D transforms for depth
   403Ôćĺ
   404Ôćĺ`rotateX()`, `rotateY()` with `transform-style: preserve-3d` create real 3D effects in CSS. Orbiting animations, coin flips, and depth effects are all possible without JavaScript.
   405Ôćĺ
   406Ôćĺ```css
   407Ôćĺ.wrapper {
   408Ôćĺ  transform-style: preserve-3d;
   409Ôćĺ}
   410Ôćĺ
   411Ôćĺ@keyframes orbit {
   412Ôćĺ  from {
   413Ôćĺ    transform: translate(-50%, -50%) rotateY(0deg) translateZ(72px) rotateY(360deg);
   414Ôćĺ  }
   415Ôćĺ  to {
   416Ôćĺ    transform: translate(-50%, -50%) rotateY(360deg) translateZ(72px) rotateY(0deg);
   417Ôćĺ  }
   418Ôćĺ}
   419Ôćĺ```
   420Ôćĺ
   421Ôćĺ### transform-origin
   422Ôćĺ
   423ÔćĺEvery element has an anchor point from which transforms execute. The default is center. Set it to match where the trigger lives for origin-aware interactions.
   424Ôćĺ
   425Ôćĺ## clip-path for Animation
   426Ôćĺ
   427Ôćĺ`clip-path` is not just for shapes. It is one of the most powerful animation tools in CSS.
   428Ôćĺ
   429Ôćĺ### The inset shape
   430Ôćĺ
   431Ôćĺ`clip-path: inset(top right bottom left)` defines a rectangular clipping region. Each value "eats" into the element from that side.
   432Ôćĺ
   433Ôćĺ```css
   434Ôćĺ/* Fully hidden from right */
   435Ôćĺ.hidden {
   436Ôćĺ  clip-path: inset(0 100% 0 0);
   437Ôćĺ}
   438Ôćĺ
   439Ôćĺ/* Fully visible */
   440Ôćĺ.visible {
   441Ôćĺ  clip-path: inset(0 0 0 0);
   442Ôćĺ}
   443Ôćĺ
   444Ôćĺ/* Reveal from left to right */
   445Ôćĺ.overlay {
   446Ôćĺ  clip-path: inset(0 100% 0 0);
   447Ôćĺ  transition: clip-path 200ms ease-out;
   448Ôćĺ}
   449Ôćĺ.button:active .overlay {
   450Ôćĺ  clip-path: inset(0 0 0 0);
   451Ôćĺ  transition: clip-path 2s linear;
   452Ôćĺ}
   453Ôćĺ```
   454Ôćĺ
   455Ôćĺ### Tabs with perfect color transitions
   456Ôćĺ
   457ÔćĺDuplicate the tab list. Style the copy as "active" (different background, different text color). Clip the copy so only the active tab is visible. Animate the clip on tab change. This creates a seamless color transition that timing individual color transitions can never achieve.
   458Ôćĺ
   459Ôćĺ### Hold-to-delete pattern
   460Ôćĺ
   461ÔćĺUse `clip-path: inset(0 100% 0 0)` on a colored overlay. On `:active`, transition to `inset(0 0 0 0)` over 2s with linear timing. On release, snap back with 200ms ease-out. Add `scale(0.97)` on the button for press feedback.
   462Ôćĺ
   463Ôćĺ### Image reveals on scroll
   464Ôćĺ
   465ÔćĺStart with `clip-path: inset(0 0 100% 0)` (hidden from bottom). Animate to `inset(0 0 0 0)` when the element enters the viewport. Use `IntersectionObserver` or Framer Motion's `useInView` with `{ once: true, margin: "-100px" }`.
   466Ôćĺ
   467Ôćĺ### Comparison sliders
   468Ôćĺ
   469ÔćĺOverlay two images. Clip the top one with `clip-path: inset(0 50% 0 0)`. Adjust the right inset value based on drag position. No extra DOM elements needed, fully hardware-accelerated.
   470Ôćĺ
   471Ôćĺ## Gesture and Drag Interactions
   472Ôćĺ
   473Ôćĺ### Momentum-based dismissal
   474Ôćĺ
   475ÔćĺDon't require dragging past a threshold. Calculate velocity: `Math.abs(dragDistance) / elapsedTime`. If velocity exceeds ~0.11, dismiss regardless of distance. A quick flick should be enough.
   476Ôćĺ
   477Ôćĺ```js
   478Ôćĺconst timeTaken = new Date().getTime() - dragStartTime.current.getTime();
   479Ôćĺconst velocity = Math.abs(swipeAmount) / timeTaken;
   480Ôćĺ
   481Ôćĺif (Math.abs(swipeAmount) >= SWIPE_THRESHOLD || velocity > 0.11) {
   482Ôćĺ  dismiss();
   483Ôćĺ}
   484Ôćĺ```
   485Ôćĺ
   486Ôćĺ### Damping at boundaries
   487Ôćĺ
   488ÔćĺWhen a user drags past the natural boundary (e.g., dragging a drawer up when already at top), apply damping. The more they drag, the less the element moves. Things in real life don't suddenly stop; they slow down first.
   489Ôćĺ
   490Ôćĺ### Pointer capture for drag
   491Ôćĺ
   492ÔćĺOnce dragging starts, set the element to capture all pointer events. This ensures dragging continues even if the pointer leaves the element bounds.
   493Ôćĺ
   494Ôćĺ### Multi-touch protection
   495Ôćĺ
   496ÔćĺIgnore additional touch points after the initial drag begins. Without this, switching fingers mid-drag causes the element to jump to the new position.
   497Ôćĺ
   498Ôćĺ```js
   499Ôćĺfunction onPress() {
   500Ôćĺ  if (isDragging) return;
   501Ôćĺ  // Start drag...
   502Ôćĺ}
   503Ôćĺ```
   504Ôćĺ
   505Ôćĺ### Friction instead of hard stops
   506Ôćĺ
   507ÔćĺInstead of preventing upward drag entirely, allow it with increasing friction. It feels more natural than hitting an invisible wall.
   508Ôćĺ
   509Ôćĺ## Performance Rules
   510Ôćĺ
   511Ôćĺ### Only animate transform and opacity
   512Ôćĺ
   513ÔćĺThese properties skip layout and paint, running on the GPU. Animating `padding`, `margin`, `height`, or `width` triggers all three rendering steps.
   514Ôćĺ
   515Ôćĺ### CSS variables are inheritable
   516Ôćĺ
   517ÔćĺChanging a CSS variable on a parent recalculates styles for all children. In a drawer with many items, updating `--swipe-amount` on the container causes expensive style recalculation. Update `transform` directly on the element instead.
   518Ôćĺ
   519Ôćĺ```js
   520Ôćĺ// Bad: triggers recalc on all children
   521Ôćĺelement.style.setProperty('--swipe-amount', `${distance}px`);
   522Ôćĺ
   523Ôćĺ// Good: only affects this element
   524Ôćĺelement.style.transform = `translateY(${distance}px)`;
   525Ôćĺ```
   526Ôćĺ
   527Ôćĺ### Framer Motion hardware acceleration caveat
   528Ôćĺ
   529ÔćĺFramer Motion's shorthand properties (`x`, `y`, `scale`) are NOT hardware-accelerated. They use `requestAnimationFrame` on the main thread. For hardware acceleration, use the full `transform` string:
   530Ôćĺ
   531Ôćĺ```jsx
   532Ôćĺ// NOT hardware accelerated (convenient but drops frames under load)
   533Ôćĺ<motion.div animate={{ x: 100 }} />
   534Ôćĺ
   535Ôćĺ// Hardware accelerated (stays smooth even when main thread is busy)
   536Ôćĺ<motion.div animate={{ transform: "translateX(100px)" }} />
   537Ôćĺ```
   538Ôćĺ
   539ÔćĺThis matters when the browser is simultaneously loading content, running scripts, or painting. At Vercel, the dashboard tab animation used Shared Layout Animations and dropped frames during page loads. Switching to CSS animations (off main thread) fixed it.
   540Ôćĺ
   541Ôćĺ### CSS animations beat JS under load
   542Ôćĺ
   543ÔćĺCSS animations run off the main thread. When the browser is busy loading a new page, Framer Motion animations (using `requestAnimationFrame`) drop frames. CSS animations remain smooth. Use CSS for predetermined animations; JS for dynamic, interruptible ones.
   544Ôćĺ
   545Ôćĺ### Use WAAPI for programmatic CSS animations
   546Ôćĺ
   547ÔćĺThe Web Animations API gives you JavaScript control with CSS performance. Hardware-accelerated, interruptible, and no library needed.
   548Ôćĺ
   549Ôćĺ```js
   550Ôćĺelement.animate([{ clipPath: 'inset(0 0 100% 0)' }, { clipPath: 'inset(0 0 0 0)' }], {
   551Ôćĺ  duration: 1000,
   552Ôćĺ  fill: 'forwards',
   553Ôćĺ  easing: 'cubic-bezier(0.77, 0, 0.175, 1)',
   554Ôćĺ});
   555Ôćĺ```
   556Ôćĺ
   557Ôćĺ## Accessibility
   558Ôćĺ
   559Ôćĺ### prefers-reduced-motion
   560Ôćĺ
   561ÔćĺAnimations can cause motion sickness. Reduced motion means fewer and gentler animations, not zero. Keep opacity and color transitions that aid comprehension. Remove movement and position animations.
   562Ôćĺ
   563Ôćĺ```css
   564Ôćĺ@media (prefers-reduced-motion: reduce) {
   565Ôćĺ  .element {
   566Ôćĺ    animation: fade 0.2s ease;
   567Ôćĺ    /* No transform-based motion */
   568Ôćĺ  }
   569Ôćĺ}
   570Ôćĺ```
   571Ôćĺ
   572Ôćĺ```jsx
   573Ôćĺconst shouldReduceMotion = useReducedMotion();
   574Ôćĺconst closedX = shouldReduceMotion ? 0 : '-100%';
   575Ôćĺ```
   576Ôćĺ
   577Ôćĺ### Touch device hover states
   578Ôćĺ
   579Ôćĺ```css
   580Ôćĺ@media (hover: hover) and (pointer: fine) {
   581Ôćĺ  .element:hover {
   582Ôćĺ    transform: scale(1.05);
   583Ôćĺ  }
   584Ôćĺ}
   585Ôćĺ```
   586Ôćĺ
   587ÔćĺTouch devices trigger hover on tap, causing false positives. Gate hover animations behind this media query.
   588Ôćĺ
   589Ôćĺ## The Sonner Principles (Building Loved Components)
   590Ôćĺ
   591ÔćĺThese principles come from building Sonner (13M+ weekly npm downloads) and apply to any component:
   592Ôćĺ
   593Ôćĺ1. **Developer experience is key.** No hooks, no context, no complex setup. Insert `<Toaster />` once, call `toast()` from anywhere. The less friction to adopt, the more people will use it.
   594Ôćĺ
   595Ôćĺ2. **Good defaults matter more than options.** Ship beautiful out of the box. Most users never customize. The default easing, timing, and visual design should be excellent.
   596Ôćĺ
   597Ôćĺ3. **Naming creates identity.** "Sonner" (French for "to ring") feels more elegant than "react-toast". Sacrifice discoverability for memorability when appropriate.
   598Ôćĺ
   599Ôćĺ4. **Handle edge cases invisibly.** Pause toast timers when the tab is hidden. Fill gaps between stacked toasts with pseudo-elements to maintain hover state. Capture pointer events during drag. Users never notice these, and that is exactly right.
   600Ôćĺ
   601Ôćĺ5. **Use transitions, not keyframes, for dynamic UI.** Toasts are added rapidly. Keyframes restart from zero on interruption. Transitions retarget smoothly.
   602Ôćĺ
   603Ôćĺ6. **Build a great documentation site.** Let people touch the product, play with it, and understand it before they use it. Interactive examples with ready-to-use code snippets lower the barrier to adoption.
   604Ôćĺ
   605Ôćĺ### Cohesion matters
   606Ôćĺ
   607ÔćĺSonner's animation feels satisfying partly because the whole experience is cohesive. The easing and duration fit the vibe of the library. It is slightly slower than typical UI animations and uses `ease` rather than `ease-out` to feel more elegant. The animation style matches the toast design, the page design, the name ÔÇö everything is in harmony.
   608Ôćĺ
   609ÔćĺWhen choosing animation values, consider the personality of the component. A playful component can be bouncier. A professional dashboard should be crisp and fast. Match the motion to the mood.
   610Ôćĺ
   611Ôćĺ### The opacity + height combination
   612Ôćĺ
   613ÔćĺWhen items enter and exit a list (like Family's drawer), the opacity change must work well with the height animation. This is often trial and error. There is no formula ÔÇö you adjust until it feels right.
   614Ôćĺ
   615Ôćĺ### Review your work the next day
   616Ôćĺ
   617ÔćĺReview animations with fresh eyes. You notice imperfections the next day that you missed during development. Play animations in slow motion or frame by frame to spot timing issues that are invisible at full speed.
   618Ôćĺ
   619Ôćĺ### Asymmetric enter/exit timing
   620Ôćĺ
   621ÔćĺPressing should be slow when it needs to be deliberate (hold-to-delete: 2s linear), but release should always be snappy (200ms ease-out). This pattern applies broadly: slow where the user is deciding, fast where the system is responding.
   622Ôćĺ
   623Ôćĺ```css
   624Ôćĺ/* Release: fast */
   625Ôćĺ.overlay {
   626Ôćĺ  transition: clip-path 200ms ease-out;
   627Ôćĺ}
   628Ôćĺ
   629Ôćĺ/* Press: slow and deliberate */
   630Ôćĺ.button:active .overlay {
   631Ôćĺ  transition: clip-path 2s linear;
   632Ôćĺ}
   633Ôćĺ```
   634Ôćĺ
   635Ôćĺ## Stagger Animations
   636Ôćĺ
   637ÔćĺWhen multiple elements enter together, stagger their appearance. Each element animates in with a small delay after the previous one. This creates a cascading effect that feels more natural than everything appearing at once.
   638Ôćĺ
   639Ôćĺ```css
   640Ôćĺ.item {
   641Ôćĺ  opacity: 0;
   642Ôćĺ  transform: translateY(8px);
   643Ôćĺ  animation: fadeIn 300ms ease-out forwards;
   644Ôćĺ}
   645Ôćĺ
   646Ôćĺ.item:nth-child(1) {
   647Ôćĺ  animation-delay: 0ms;
   648Ôćĺ}
   649Ôćĺ.item:nth-child(2) {
   650Ôćĺ  animation-delay: 50ms;
   651Ôćĺ}
   652Ôćĺ.item:nth-child(3) {
   653Ôćĺ  animation-delay: 100ms;
   654Ôćĺ}
   655Ôćĺ.item:nth-child(4) {
   656Ôćĺ  animation-delay: 150ms;
   657Ôćĺ}
   658Ôćĺ
   659Ôćĺ@keyframes fadeIn {
   660Ôćĺ  to {
   661Ôćĺ    opacity: 1;
   662Ôćĺ    transform: translateY(0);
   663Ôćĺ  }
   664Ôćĺ}
   665Ôćĺ```
   666Ôćĺ
   667ÔćĺKeep stagger delays short (30-80ms between items). Long delays make the interface feel slow. Stagger is decorative ÔÇö never block interaction while stagger animations are playing.
   668Ôćĺ
   669Ôćĺ## Debugging Animations
   670Ôćĺ
   671Ôćĺ### Slow motion testing
   672Ôćĺ
   673ÔćĺPlay animations at reduced speed to spot issues invisible at full speed. Temporarily increase duration to 2-5x normal, or use browser DevTools animation inspector to slow playback.
   674Ôćĺ
   675ÔćĺThings to look for in slow motion:
   676Ôćĺ
   677Ôćĺ- Do colors transition smoothly, or do you see two distinct states overlapping?
   678Ôćĺ- Does the easing feel right, or does it start/stop abruptly?
   679Ôćĺ- Is the transform-origin correct, or does the element scale from the wrong point?
   680Ôćĺ- Are multiple animated properties (opacity, transform, color) in sync?
   681Ôćĺ
   682Ôćĺ### Frame-by-frame inspection
   683Ôćĺ
   684ÔćĺStep through animations frame by frame in Chrome DevTools (Animations panel). This reveals timing issues between coordinated properties that you cannot see at full speed.
   685Ôćĺ
   686Ôćĺ### Test on real devices
   687Ôćĺ
   688ÔćĺFor touch interactions (drawers, swipe gestures), test on physical devices. Connect your phone via USB, visit your local dev server by IP address, and use Safari's remote devtools. The Xcode Simulator is an alternative but real hardware is better for gesture testing.

   690Ôćĺ## Review Checklist
   691Ôćĺ
   692ÔćĺWhen reviewing UI code, check for:
   693Ôćĺ
   694Ôćĺ| Issue                                      | Fix                                                              |
   695Ôćĺ| ------------------------------------------ | ---------------------------------------------------------------- |
   696Ôćĺ| `transition: all`                          | Specify exact properties: `transition: transform 200ms ease-out` |
   697Ôćĺ| `scale(0)` entry animation                 | Start from `scale(0.95)` with `opacity: 0`                       |
   698Ôćĺ| `ease-in` on UI element                    | Switch to `ease-out` or custom curve                             |
   699Ôćĺ| `transform-origin: center` on popover      | Set to trigger location or use Radix/Base UI CSS variable (modals are exempt ÔÇö keep centered) |
   700Ôćĺ| Animation on keyboard action               | Remove animation entirely                                        |
   701Ôćĺ| Duration > 300ms on UI element             | Reduce to 150-250ms                                              |
   702Ôćĺ| Hover animation without media query        | Add `@media (hover: hover) and (pointer: fine)`                  |
   703Ôćĺ| Keyframes on rapidly-triggered element     | Use CSS transitions for interruptibility                         |
   704Ôćĺ| Framer Motion `x`/`y` props under load     | Use `transform: "translateX()"` for hardware acceleration        |
   705Ôćĺ| Same enter/exit transition speed           | Make exit faster than enter (e.g., enter 2s, exit 200ms)         |
   706Ôćĺ| Elements all appear at once                | Add stagger delay (30-80ms between items)                        |
