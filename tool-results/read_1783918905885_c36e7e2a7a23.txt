     1‚Üí     1‚Üí# Project Instructions for Claude
     2‚Üí     2‚Üí
     3‚Üí     3‚Üí## Architecture (v3.0+)
     4‚Üí     4‚Üí
     5‚Üí     5‚ÜíThere is **one** user-invocable skill, `impeccable`, with **23 commands** underneath it. Users type `/impeccable polish`, `/impeccable audit`, etc. The skill is defined in `skill/`:
     6‚Üí     6‚Üí
     7‚Üí     7‚Üí- `SKILL.src.md` ‚Äî frontmatter (with the auto-trigger-optimized description and the `allowed-tools` list), shared design laws, and the **Commands** router table. Provider `SKILL.md` files are generated from this source.
     8‚Üí     8‚Üí- `reference/` ‚Äî one `<command>.md` per command (`audit.md`, `polish.md`, `critique.md`, etc.) plus the domain reference files (`typography.md`, `color-and-contrast.md`, etc.). When a sub-command is matched, the router loads its reference file.
     9‚Üí     9‚Üí- `reference/brand.md` and `reference/product.md` ‚Äî the two register references. SKILL.md's Setup section selects one based on the task cue, the surface in focus, or the `register` field in PRODUCT.md (first match wins).
    10‚Üí    10‚Üí- `scripts/command-metadata.json` ‚Äî single source of truth for each command's description, argument hint, and (eventually) category. Both the build and `pin.mjs` read from this.
    11‚Üí    11‚Üí- `scripts/pin.mjs` ‚Äî creates/removes lightweight redirect shims so users can have `/audit` as a standalone shortcut that delegates to `/impeccable audit`.
    12‚Üí    12‚Üí
    13‚Üí    13‚Üí**Do not add standalone skills** unless there's a strong reason. The consolidation was deliberate: the `/` menu pollution problem is real and gets worse as users install more plugins.
    14‚Üí    14‚Üí
    15‚Üí    15‚Üí### Register (brand vs product)
    16‚Üí    16‚Üí
    17‚Üí    17‚ÜíEvery design task belongs to one of two registers:
    18‚Üí    18‚Üí
    19‚Üí    19‚Üí- **Brand** ‚Äî design IS the product: marketing, landing pages, brand sites, campaign surfaces, portfolios, long-form content. Distinctiveness is the bar. Spans every visual lane (tech-minimal, luxury, editorial-magazine, consumer-warm, brutalist, etc.) ‚Äî do not default to only one.
    20‚Üí    20‚Üí- **Product** ‚Äî design SERVES the product: app UI, admin, dashboards, tools. Earned familiarity is the bar ‚Äî fluent users of Linear / Figma / Notion / Raycast / Stripe should trust it.
    21‚Üí    21‚Üí
    22‚Üí    22‚ÜíPRODUCT.md at the project root carries a `## Register` section with a bare value (`brand` or `product`). `/impeccable init` asks about register first because it shapes every downstream answer.
    23‚Üí    23‚Üí
    24‚Üí    24‚ÜíSub-command reference files add a short `## Register` section near the top *only where the answer diverges between the two*. Don't restate the register files' content in sub-commands ‚Äî link instead. Sub-commands where register meaningfully diverges today: `typeset`, `animate`, `bolder`, `delight`, `colorize`, `layout`, `quieter`.
    25‚Üí    25‚Üí
    26‚Üí    26‚Üí**a11y lives in `audit.md`**, not in SKILL.md, `brand.md`, or `product.md`. Models over-cautious themselves into safe, underdesigned output when reminded about accessibility at design time. The audit command is the dedicated place for that check.
    27‚Üí    27‚Üí
    28‚Üí    28‚Üí### Platform (web / ios / android / adaptive)
    29‚Üí    29‚Üí
    30‚Üí    30‚ÜíA second axis, **orthogonal to register**. Register answers "does design IS or SERVES the product"; platform answers "what's the delivery target and which native conventions apply":
    31‚Üí    31‚Üí
    32‚Üí    32‚Üí- **web** ‚Äî a website or web app (including responsive mobile web). The default. No extra rulebook and no reference file: the General rules in SKILL.md and the register reference cover it.
    33‚Üí    33‚Üí- **ios** ‚Äî a native iOS / iPadOS app. Loads `reference/ios.md` (Apple HIG distilled) on top of the register reference.
    34‚Üí    34‚Üí- **android** ‚Äî a native Android app. Loads `reference/android.md` (Material Design 3 distilled) on top of the register reference.
    35‚Üí    35‚Üí- **adaptive** ‚Äî a cross-platform app shipping both iOS and Android from one codebase (Flutter, React Native, KMP) that adapts per OS. Loads **both** `reference/ios.md` and `reference/android.md`. A Flutter/RN app that uses one look on both platforms (Material-everywhere is the Flutter default) is not adaptive; it takes that single platform's value.
    36‚Üí    36‚Üí
    37‚Üí    37‚ÜíPRODUCT.md carries a `## Platform` section with a bare value (`web` / `ios` / `android` / `adaptive`). It's parsed by `extractPlatform()` in `skill/scripts/context.mjs` (mirroring `extractRegister()`); a **missing field defaults to `web`** so legacy projects are unaffected. A line that names both native targets (e.g. `ios, android`) is also read as `adaptive`; any other unrecognized value falls back to web **and** the `context.mjs` CLI prints a WARNING directive naming the bad value, so a toolchain name or typo never silently gets web guidance. `context.mjs` appends a NEXT STEP directive to read the native reference(s) when the value is `ios`, `android`, or `adaptive` (both). `init` (Step 3) asks platform right after register.
    38‚Üí    38‚Üí
    39‚Üí    39‚Üí`ios.md` and `android.md` are distilled from the MIT-licensed [ehmo/platform-design-skills](https://github.com/ehmo/platform-design-skills); attribution is in `NOTICE.md`.
    40‚Üí    40‚Üí
    41‚Üí    41‚ÜíWhere a command's native guidance diverges too much to share a file, it gets a **native variant**: `reference/<command>.native.md`, listed in SKILL.md's Commands table and routed **instead of** the web file when `setup.platform` is native (Setup step 2). One variant covers ios, android, and adaptive; per-OS specifics stay in the platform refs, which Setup loads regardless. Variants today: `audit.native.md`, `adapt.native.md` (their web files carry a one-line web-only guard that redirects stray native readers). `audit.native.md` mirrors `audit.md`'s report skeleton; change the skeleton in both together. Commands whose divergence the platform refs already cover (`animate`, `layout`) carry nothing extra; don't add in-file translation notes, they make native runs pay for web content.
    42‚Üí    42‚Üí
    43‚Üí    43‚Üí**Live mode, the `detect` CLI, and the design hook are web-only.** They operate on a browser / HTML rules, so SKILL.md's routing skips live and `detect.mjs` for any native (`ios` / `android` / `adaptive`) project, and the hook (`hook-lib.mjs` `resolveProjectPlatform` / `isNativePlatform`, also used by `hook-before-edit.mjs`) skips its scan when PRODUCT.md declares a native platform ‚Äî a React Native project is made of exactly the `.tsx` / `.ts` / `.js` files the hook watches.

    45‚Üí    45‚Üí## CSS
    46‚Üí    46‚Üí
    47‚Üí    47‚ÜíPlain hand-written CSS, no Tailwind. Imported into Astro pages/layouts via frontmatter `import` statements; Vite resolves `@import` chains automatically.
    48‚Üí    48‚Üí
    49‚Üí    49‚ÜíThe CSS architecture (under `site/styles/`):
    50‚Üí    50‚Üí- `main.css` ‚Äî Main entry point, imports the partials and defines tokens/reset
    51‚Üí    51‚Üí- `workflow.css` ‚Äî Commands section, glass terminal, magazine spread styles
    52‚Üí    52‚Üí- `sub-pages.css` ‚Äî `/docs`, `/anti-patterns`, `/tutorials`, detail pages
    53‚Üí    53‚Üí- `tokens.css` ‚Äî OKLCH color tokens (ink, charcoal, ash, mist, cream, accent)
    54‚Üí    54‚Üí- `footer.css` ‚Äî shared across all pages, imported in `Base.astro`
    55‚Üí    55‚Üí
    56‚Üí    56‚ÜíEdit any of these directly and the dev server hot-reloads. No rebuild needed for CSS changes.
    57‚Üí    57‚Üí
    58‚Üí    58‚Üí## Color token rule
    59‚Üí    59‚Üí
    60‚Üí    60‚Üí- **`--color-ink`** (10% lightness) is for body copy. Use it even for small text.
    61‚Üí    61‚Üí- **`--color-charcoal`** (25% lightness) reads as washed-out gray in small text. Only use for headings or larger body copy at ‚â•16px.
    62‚Üí    62‚Üí- **`--color-ash`** (55%) is for secondary labels, captions, relationship meta lines.
    63‚Üí    63‚Üí- **Never use pure black or pure white.** Use the tinted tokens.

    65‚Üí    65‚Üí## Prose: read docs/STYLE.md before writing user-facing copy
    66‚Üí    66‚Üí
    67‚Üí    67‚ÜíEditorial brief is at `docs/STYLE.md`. Read it before editing the homepage, sub-pages, command editorials, tutorials, or READMEs. The site has been called out for AI prose; the rules there exist to keep that from creeping back.
    68‚Üí    68‚Üí
    69‚Üí    69‚ÜíThe build's `validateProse` step (in `scripts/build.js`) enforces a denylist: em dashes (`‚Äî` and HTML entities), the `--` em-dash substitute, `load-bearing`, `highest-leverage`, `biggest unlock`, `seamless`, `robust`, `delve`, `elevate`, `empower`, `underscore`, `pivotal`, `tapestry`, `data-driven`, `reflex defaults`, `collapses into monoculture`, `in today's`, `gone are the days`, `whether you're`, `let's dive in`, `in summary`, `in conclusion`, `moreover`, `furthermore`. Each rule prints a rationale and a suggested replacement when it fires. **Do not silently work around the regex.** If a banned word has earned a real meaning here, raise it as a `docs/STYLE.md` amendment.
    70‚Üí    70‚Üí
    71‚Üí    71‚ÜíThe validator scans `site/pages/`, `site/content/`, `site/components/`, `site/layouts/`, `README.md`, `README.npm.md`. It deliberately skips `skill/` because LLM-facing reference instructions sometimes need technical phrasings the marketing copy can't.
    72‚Üí    72‚Üí
    73‚Üí    73‚ÜíThe deeper structural issues (negation pivot, triadic auto-pilot, uniform paragraph rhythm, hollow confidence) require human judgment. `docs/STYLE.md` lists them. Use them on every editorial pass.

   101‚Üí   101‚Üí## Social sharing image (OG card)
   102‚Üí   102‚Üí
   103‚Üí   103‚ÜíThe OG / Twitter card is generated, not hand-drawn. To regenerate after a brand or copy change:
   104‚Üí   104‚Üí
   105‚Üí   105‚Üí```bash
   106‚Üí   106‚Üíbun run og-image   # ‚Üí site/public/og-image-v2.jpg
   107‚Üí   107‚Üí```
   108‚Üí   108‚Üí
   109‚Üí   109‚Üí`scripts/generate-og-image.js` renders an inline HTML card with Playwright (Neo Kinpaku brand: lacquer ground, champagne Alumni Sans headline, kinpaku-gold accent, the kintsugi-seam art from `site/public/assets/neo-kinpaku/candidates/finalists/m-01-v2-01.png`). It renders at 2√ó and downscales to 1200√ó630 with `sharp` for crisp text. The "N commands" figure is read live from `command-metadata.json`, so it never goes stale; don't hardcode it.
   110‚Üí   110‚Üí
   111‚Üí   111‚ÜíThe card is referenced as a **sitewide default** in `site/layouts/Base.astro` (every page emits `og:image` + a `summary_large_image` Twitter card; pages may override via the `ogImage` prop). The homepage sets its own `ogImage` in `site/pages/index.astro`.
   112‚Üí   112‚Üí
   113‚Üí   113‚Üí**Cache-busting:** social scrapers cache by URL, so the filename carries a `-v2` suffix. When you ship a visibly different card, bump the suffix in three places together (`scripts/generate-og-image.js` `OUTPUT_PATH`, `Base.astro` `SITE_OG_IMAGE`, `index.astro` `ogImage`) so X/LinkedIn/Slack re-fetch instead of serving the stale image. After deploy, prime the caches by running the URL through X's Post Inspector and LinkedIn's Post Inspector once.

   115‚Üí   115‚Üí## Build System
   116‚Üí   116‚Üí
   117‚Üí   117‚ÜíThe build system compiles the impeccable skill from `skill/` to provider-specific formats in `dist/`. The default build is source-first and does not sync tracked root harness folders; the release build performs the tracked distribution sync:
   118‚Üí   118‚Üí
   119‚Üí   119‚Üí```bash
   120‚Üí   120‚Üíbun run build            # Build dist/site output without syncing root harness dirs
   121‚Üí   121‚Üíbun run build:release    # Build dist/site output and sync root harness dirs + plugin/
   122‚Üí   122‚Üíbun run rebuild          # Clean and rebuild without root harness sync
   123‚Üí   123‚Üíbun run rebuild:release  # Clean and rebuild with root harness sync
   124‚Üí   124‚Üí```
   125‚Üí   125‚Üí
   126‚Üí   126‚ÜíSource files use placeholders that get replaced per-provider:
   127‚Üí   127‚Üí- `{{model}}` ‚Äî Model name (Claude, Gemini, GPT, etc.)
   128‚Üí   128‚Üí- `{{config_file}}` ‚Äî Config file name (CLAUDE.md, .cursorrules, etc.)
   129‚Üí   129‚Üí- `{{ask_instruction}}` ‚Äî How to ask user questions
   130‚Üí   130‚Üí- `{{command_prefix}}` ‚Äî `/` or `$` depending on provider
   131‚Üí   131‚Üí- `{{available_commands}}` ‚Äî auto-populated list of commands (from `IMPECCABLE_SUB_COMMANDS` in `scripts/lib/utils.js`)
   132‚Üí   132‚Üí- `{{scripts_path}}` ‚Äî provider-aware path to the skill's scripts directory
   133‚Üí   133‚Üí
   134‚Üí   134‚Üí### Generated provider output policy
   135‚Üí   135‚Üí
   136‚Üí   136‚Üí`.claude/skills/`, `.cursor/skills/`, `.agents/skills/`, and the other harness directories are **intentionally committed to the repo**. `npx skills` reads them directly from this repo at install time, and they enable clean submodule use. Do not gitignore them.
   137‚Üí   137‚Üí
   138‚Üí   138‚ÜíThey are generated distribution artifacts, not authoring surfaces. Normal development PRs should be source-first: edit and stage `skill/`, `scripts/`, `cli/`, `site/`, `extension/`, `functions/`, and `tests/`; do not stage regenerated provider permutations unless the task is explicitly a release/generated-output sync or a build-system change. Run `bun run build` for validation after editing `skill/`, transformer code, generated site counts, or provider behavior. Use `bun run build:release` only when intentionally refreshing tracked harness outputs.
   139‚Üí   139‚Üí
   140‚Üí   140‚ÜíAfter source changes land on `main`, `.github/workflows/sync-generated-output.yml` runs `bun run build:release` and commits generated provider output directly back to `main`. Treat generated harness diffs as release artifacts and keep them out of feature PRs unless they are the point of the PR.

   142‚Üí   142‚ÜíLocal state files inside harness directories (e.g. `.claude/scheduled_tasks.lock`, `.claude/settings.local.json`) ARE gitignored.

   158‚Üí   158‚Üí## Testing
   159‚Üí   159‚Üí
   160‚Üí   160‚Üí```bash
   161‚Üí   161‚Üíbun run test                  # Default suite: unit + static framework fixtures
   162‚Üí   162‚Üíbun run test:live-e2e         # Opt-in: full-cycle live-mode E2E across framework fixtures
   163‚Üí   163‚Üíbun run test:skill-behavior   # Opt-in: LLM-backed checks that the skill text actually drives the agent's setup flow
   164‚Üí   164‚Üí```
   165‚Üí   165‚Üí
   166‚Üí   166‚ÜíUnit tests (build orchestration, detector logic) run via `bun test`. Fixture tests (jsdom-based HTML detection) run via `node --test` because bun is too slow with jsdom. The `test` script handles this split automatically.
   167‚Üí   167‚Üí
   168‚Üí   168‚Üí**Important:** `tests/build.test.js` uses `spyOn(transformers, 'transformCursor')` with the named exports from `scripts/lib/transformers/index.js`. Those named exports (`transformCursor`, `transformClaudeCode`, etc.) are kept specifically for test spying, even though `build.js` itself uses `createTransformer + PROVIDERS` directly. **Do not delete them as "dead code"** ‚Äî I made that mistake once and broke 8 tests.
   169‚Üí   169‚Üí
   170‚Üí   170‚Üí### Live-mode E2E
   171‚Üí   171‚Üí
   172‚Üí   172‚Üí`tests/live-e2e.test.mjs` drives the entire user flow (handshake ‚Üí pick ‚Üí Go ‚Üí cycle ‚Üí accept ‚Üí carbonize cleanup) against every fixture in `tests/framework-fixtures/` that declares a `runtime` block. Each fixture installs real deps, boots its framework dev server (Vite, Next, SvelteKit, Astro, Nuxt static), and runs Playwright Chromium against a deterministic fake agent that produces realistic variants in the exact format `reference/live.md` describes.
   173‚Üí   173‚Üí
   174‚Üí   174‚Üí```bash
   175‚Üí   175‚Üíbun run test:live-e2e                                       # full suite, ~2 min, 19 fixtures
   176‚Üí   176‚ÜíIMPECCABLE_E2E_ONLY=vite8-react-modal bun run test:live-e2e # scope to one fixture
   177‚Üí   177‚ÜíIMPECCABLE_E2E_DEBUG=1 bun run test:live-e2e                # dump page DOM + dev-server tail on failure
   178‚Üí   178‚Üí```
   179‚Üí   179‚Üí
   180‚Üí   180‚Üí**One-time setup**: `npx playwright install chromium` (the suite uses a specific Chromium build keyed to the bundled Playwright version).
   181‚Üí   181‚Üí
   182‚Üí   182‚Üí**Kept out of the default `bun run test`** because (a) it does real `npm install` per fixture, (b) it boots framework dev servers, (c) wall time is ~2 minutes, and (d) it requires Playwright's browser cache. Run it locally before shipping changes to anything in `skill/scripts/live-*.{mjs,js}` or `skill/scripts/live/**`.
   183‚Üí   183‚Üí
   184‚Üí   184‚ÜíThe agent is pluggable via a one-method interface in `tests/live-e2e/agent.mjs`: `generateVariants(event, context) ‚Üí { scopedCss, variants[] }`. The default fake agent emits canned variants that exercise all three param kinds (`range`, `steps`, `toggle`). The orchestrator (wrap, write, accept, carbonize) is agent-agnostic.
   185‚Üí   185‚Üí
   186‚Üí   186‚Üí**LLM agent (opt-in)**: set `IMPECCABLE_E2E_AGENT=llm` to swap the fake agent for `tests/live-e2e/agents/llm-agent.mjs`, which calls Claude (default Haiku 4.5) via `@anthropic-ai/sdk`. Requires `ANTHROPIC_API_KEY` in env; the test runner skips with a clear message when it's unset. Override the model with `IMPECCABLE_E2E_LLM_MODEL=claude-sonnet-4-6` if Haiku produces unreliable JSON. Caching is on ‚Äî live.md is the cacheable prefix, and after the first call subsequent fixtures pay only the cache-read rate. Pass rate on a typical sweep is 18/19; the modal fixture's intrinsic state-loss flake is amplified by LLM latency and may need a re-run. **This path hits the API and costs money** ‚Äî keep it out of CI unless you really want it there.
   187‚Üí   187‚Üí
   188‚Üí   188‚ÜíAdding a new fixture is a matter of cloning a directory under `tests/framework-fixtures/`, swapping the source files, and writing a `fixture.json`. See `tests/framework-fixtures/README.md` for the full schema.

   190‚Üí   190‚Üí### Skill-behavior tests
   191‚Üí   191‚Üí
   192‚Üí   192‚Üí`tests/skill-behavior/scenarios.test.mjs` is the LLM-backed safety net for edits to `skill/SKILL.src.md` and the Setup-adjacent reference files (`init.md`, `document.md`, `brand.md`, `product.md`, sub-command refs). It inlines the source `skill/SKILL.src.md` into the system prompt of a real LLM, gives the agent `bash` / `read` / `write` / `list` tools scoped to a temp workspace, and asserts on the tool-call trace ‚Äî not on the model's free-form output. The trace is the source of truth.
   193‚Üí   193‚Üí
   194‚Üí   194‚Üí```bash
   195‚Üí   195‚Üíbun run test:skill-behavior                                              # full suite (27 tests, ~5 min, ~$0.50-1.50 across providers)
   196‚Üí   196‚ÜíIMPECCABLE_SKILL_BEHAVIOR_MODELS=gemini-3.1-flash-lite bun run test:skill-behavior   # scope to one provider
   197‚Üí   197‚ÜíIMPECCABLE_SKILL_BEHAVIOR_VERBOSE=1 bun run test:skill-behavior          # dump per-scenario trace JSON to stderr (use when iterating)
   198‚Üí   198‚Üí```
   199‚Üí   199‚Üí
   200‚Üí   200‚Üí**Three providers per run, every run.** The suite always exercises `claude-sonnet-4-6`, `gpt-5.5`, and `gemini-3.1-flash-lite`. Sonnet and GPT-5.5 are production-tier, matching what users actually run, so the pass/fail signal reflects real agent behavior rather than a cheap proxy; gemini stays on the flash-lite tier. **Don't substitute Claude alone**: many of the most useful findings come from divergence between providers.
   201‚Üí   201‚Üí
   202‚Üí   202‚Üí**Auth** lives in repo-root `.env` (copied from `~/code/impeccable-evals/.env`, gitignored). Providers skip cleanly when their key is unset; they don't fail.
   203‚Üí   203‚Üí
   204‚Üí   204‚Üí**Fifteen scenarios:**
   205‚Üí   205‚Üí1. empty workspace ‚Üí agent loads `reference/init.md`
   206‚Üí   206‚Üí2. PRODUCT.md only ‚Üí loads `brand.md`
   207‚Üí   207‚Üí3. PRODUCT.md + DESIGN.md ‚Üí loads `brand.md` + consults the design system
   208‚Üí   208‚Üí4. context already loaded in turn 1 ‚Üí turn 2 does **not** re-run `context.mjs`
   209‚Üí   209‚Üí5. PRODUCT.md without `## Register` field ‚Üí agent infers `brand` from task cue
   210‚Üí   210‚Üí6. `/impeccable polish` ‚Üí loads `reference/polish.md`
   211‚Üí   211‚Üí7. `/impeccable audit` ‚Üí loads `reference/audit.md`
   212‚Üí   212‚Üí8. existing SvelteKit project ‚Üí agent reads at least one project code file
   213‚Üí   213‚Üí9. `context.mjs` emits `UPDATE_AVAILABLE` (seeded newer version) ‚Üí agent surfaces it but does **not** auto-run `npx impeccable skills update`
   214‚Üí   214‚Üí10. scoped command with no PRODUCT.md ‚Üí proceeds without forcing init
   215‚Üí   215‚Üí11. `/impeccable shape` with no PRODUCT.md ‚Üí diverts into `reference/init.md`
   216‚Üí   216‚Üí12. natural-language build intent with no PRODUCT.md ‚Üí diverts into `reference/init.md`
   217‚Üí   217‚Üí13. `/impeccable teach` ‚Üí diverts into `reference/init.md` (alias)
   218‚Üí   218‚Üí14. PRODUCT.md with `## Platform: ios` ‚Üí `context.mjs` emits the native NEXT STEP and the agent loads `reference/ios.md`
   219‚Üí   219‚Üí15. same iOS fixture, `/impeccable audit` ‚Üí agent loads `reference/audit.native.md` (route-instead variant)
   220‚Üí   220‚Üí
   221‚Üí   221‚Üí**Baseline.** The 21-22 / 24 baseline (with stable gpt scenario 6/7 failures) was measured on the old cheap tier (`claude-haiku-4-5` / `gpt-5.4-mini`). It needs re-measuring on the current `claude-sonnet-4-6` / `gpt-5.5` lineup; the production-tier models are expected to do better on the sub-command routing scenarios the old gpt tier failed. See `tests/skill-behavior/README.md`.
   222‚Üí   222‚Üí
   223‚Üí   223‚Üí**Cost.** Each run is real LLM calls, billed to the keys in `.env`. Production-tier models put a full sweep around $0.50-1.50. Keep it out of CI unless you really want it there.
   224‚Üí   224‚Üí
   225‚Üí   225‚Üí**Adding a scenario.** Write the fixture in `tests/skill-behavior/fixtures.mjs`, add the `it()` block in `scenarios.test.mjs` (the harness uses the source `skill/` dir via a symlink, so no rebuild needed), and update the baseline table in the suite's README. The harness's `fileLoaded(trace, filename)` helper checks both `read` and bash `cat` ‚Äî different models prefer different tools.
   226‚Üí   226‚Üí
   227‚Üí   227‚Üí**The harness symlinks source, not built output.** This is deliberate so SKILL.md / reference / `scripts/context.mjs` edits show up immediately without `bun run build:skills`. The trade-off: reference files surface their raw `{{placeholders}}`, but the assertions key on tool calls rather than content, so it doesn't matter for correctness.

   249‚Üí   249‚Üí## Versioning
   250‚Üí   250‚Üí
   251‚Üí   251‚ÜíThere are three independently versioned components. Only bump the one(s) that actually changed:
   252‚Üí   252‚Üí
   253‚Üí   253‚Üí**CLI** (npm package):
   254‚Üí   254‚Üí- `package.json` ‚Üí `version`
   255‚Üí   255‚Üí- Bump when: CLI code changes (`cli/bin/`, `cli/engine/detect-antipatterns.mjs`, etc.)
   256‚Üí   256‚Üí
   257‚Üí   257‚Üí**Skills** (Claude Code plugin / skill definitions):
   258‚Üí   258‚Üí- `.claude-plugin/plugin.json` ‚Üí `version` (source of truth)
   259‚Üí   259‚Üí- `.claude-plugin/marketplace.json` ‚Üí `plugins[0].version`
   260‚Üí   260‚Üí- Bump when: skill content changes (`skill/`, reference files, command metadata, etc.)
   261‚Üí   261‚Üí- After bumping, run `bun run build:release` so the committed `./plugin` subtree (`plugin/.claude-plugin/plugin.json` + `plugin/skills/impeccable/SKILL.md`) is regenerated to the new version. The build validator (`validatePluginVersions` in `scripts/build.js`) fails if `marketplace.json`, the `./plugin` manifest, or the bundled `SKILL.md` frontmatter disagree with `plugin.json` ‚Äî this guards the marketplace install path against version drift (issue #274).
   262‚Üí   262‚Üí
   263‚Üí   263‚Üí**Chrome extension**:
   264‚Üí   264‚Üí- `extension/manifest.json` ‚Üí `version`
   265‚Üí   265‚Üí- Bump when: extension code changes (`extension/`)
   266‚Üí   266‚Üí
   267‚Üí   267‚Üí**Website changelog** (`site/pages/changelog.astro`):
   268‚Üí   268‚Üí- Add a new `<article>` entry at the top of the relevant component's group, and move the `cf-entry--current` class + `Current` badge onto it (off the previous newest skill entry). The component is derived from the entry `id` prefix: `cli-*`, `ext-*`, else skill.
   269‚Üí   269‚Üí- Keep it concise and sell the release: a short `cf-entry-lead` that frames what shipped, then a handful of tight `<li>` items. Lead with the most compelling feature.
   270‚Üí   270‚Üí- User-facing only. Every item must be something an impeccable user would notice or act on (a new command behavior, rule, or fix). Leave out internal build/tooling/refactor details, dependency bumps, and generated-output syncs.
   271‚Üí   271‚Üí- Prose rules in `docs/STYLE.md` apply (the validator scans this file): no em dashes, no banned words, no AI-tell cadence.
   272‚Üí   272‚Üí
   273‚Üí   273‚ÜíAfter bumping, see **Releases** below for how to tag and publish.

   275‚Üí   275‚Üí## Releases
   276‚Üí   276‚Üí
   277‚Üí   277‚ÜíGitHub releases are tagged per-component, not per-version, since the three components ship independently. Tag prefixes: `skill-v`, `cli-v`, `ext-v`.
   278‚Üí   278‚Üí
   279‚Üí   279‚ÜíWorkflow for any component:
   280‚Üí   280‚Üí
   281‚Üí   281‚Üí1. Bump the manifest version (see Versioning above).
   282‚Üí   282‚Üí2. Add a changelog entry to `site/pages/changelog.astro` (see **Website changelog** above for placement and tone). Skill entries use a bare `vX.Y.Z` label; CLI and extension entries use the prefixed forms `CLI vX.Y.Z` and `Extension vX.Y.Z`. The release script extracts notes by matching this label, so the prefix matters.
   283‚Üí   283‚Üí3. Commit and push to `main`.
   284‚Üí   284‚Üí4. Run `bun run release:<skill|cli|ext>`. Preview first with `node scripts/release.mjs <component> --dry-run`.
   285‚Üí   285‚Üí
   286‚Üí   286‚ÜíThe script refuses to run if: the working tree is dirty, HEAD is ahead of origin, the tag already exists, the matching changelog entry is missing, or (for skill/extension) `bun run build:release` / `bun run build:extension` produces uncommitted changes ‚Äî meaning the harness output dirs or `extension/detector/` files weren't refreshed before the bump was committed.
   287‚Üí   287‚Üí
   288‚Üí   288‚ÜíSkill releases attach `dist/universal.zip`. Extension releases run `bun run build:extension` first and attach `dist/extension.zip`. CLI releases print a reminder to run `npm publish` separately; extension releases print a reminder to upload the zip to the Chrome Web Store dashboard.
   289‚Üí   289‚Üí
   290‚Üí   290‚ÜíIf you need to fix release notes after the fact (typo, missing thank-you, formatting bug): `gh release edit <tag> --notes-file <md>`. The release script's `htmlToMarkdown` function is the cleanest source for regenerating notes from the changelog.

   303‚Üí   303‚Üí8. Add its relationships (leadsTo / pairs / combinesWith) to `COMMAND_RELATIONSHIPS` in the same file

   313‚Üí   313‚Üí- `AGENTS.md` ‚Äî intro command count
   314‚Üí   314‚Üí- `.claude-plugin/plugin.json` ‚Äî description
   315‚Üí   315‚Üí- `.claude-plugin/marketplace.json` ‚Äî metadata description + plugin description

   332‚Üí   332‚Üí## Adding or modifying anti-pattern detection rules
   333‚Üí   333‚Üí
   334‚Üí   334‚Üí`cli/engine/detect-antipatterns.mjs` is the source of truth for the rule engine. It powers the CLI, the public-site overlay, the Chrome extension, and the homepage rule count. Five places stay in sync:
   335‚Üí   335‚Üí
   336‚Üí   336‚Üí| Where | How it stays in sync |
   337‚Üí   337‚Üí|---|---|
   338‚Üí   338‚Üí| `cli/engine/detect-antipatterns.mjs` (`ANTIPATTERNS` array + `checkXxx` logic) | Hand-edited |
   339‚Üí   339‚Üí| `cli/engine/detect-antipatterns-browser.js` | `bun run build:browser` |
   340‚Üí   340‚Üí| `extension/detector/detect.js` + `extension/detector/antipatterns.json` | `bun run build:extension` |
   341‚Üí   341‚Üí| `site/public/js/generated/counts.js` (`DETECTION_COUNT`) | `bun run build` |
   342‚Üí   342‚Üí| `skill/SKILL.src.md` and `reference/*.md` | Hand-edited if the rule introduces new design guidance |
   343‚Üí   343‚Üí
   344‚Üí   344‚ÜíAlways run all three builds and the test suite after a rule change:
   345‚Üí   345‚Üí
   346‚Üí   346‚Üí```bash
   347‚Üí   347‚Üíbun run build && bun run build:browser && bun run build:extension && bun run test
   348‚Üí   348‚Üí```
   349‚Üí   349‚Üí
   350‚Üí   350‚Üí### TDD order (non-negotiable)
   351‚Üí   351‚Üí
   352‚Üí   352‚Üí1. **Fixture** at `tests/fixtures/antipatterns/{rule-id}.html` with two columns (should-flag / should-pass), each case identified by a unique heading. Cover ‚â•4 flag cases and ‚â•5 false-positive shapes. Use **explicit pixel dimensions in CSS** because jsdom does no layout.
   353‚Üí   353‚Üí2. **Failing test** in `tests/detect-antipatterns-fixtures.test.mjs` using the snippet-substring pattern (regex `/"([^"]+)"/` against `SHOULD_FLAG` / `SHOULD_PASS` lists). Run it and watch it fail before implementing.
   354‚Üí   354‚Üí3. **Rule entry** in the `ANTIPATTERNS` array: `id`, `category` (`slop` for AI tells, `quality` for real design or a11y issues), `name`, `description`, optional `skillSection` and `skillGuideline`.
   355‚Üí   355‚Üí4. **Pure check function** `checkXxx(opts)` returning `[{ id, snippet }]`. No DOM access in the pure function.
   356‚Üí   356‚Üí5. **Two adapters**: `checkElementXxxDOM(el)` for the browser (`getComputedStyle` + `getBoundingClientRect`) and `checkElementXxx(el, tag, window)` for jsdom (`parseFloat(style.width)` instead of layout). Wire **both** into **both** element loops in `cli/engine/detect-antipatterns.mjs` ‚Äî the browser loop (~line 1837) and the jsdom loop in `detectHtml` (~line 2058). Forgetting one is the most common mistake; symptom is "test passes, live page silent" or vice versa.
   357‚Üí   357‚Üí6. **Verify on a live page**: `http://localhost:4321/fixtures/antipatterns/{rule-id}.html` and the homepage (no false positives). The two adapter paths can disagree, so manual browser checks catch what the fixture test can't.
   358‚Üí   358‚Üí
   359‚Üí   359‚Üí### Conventions and jsdom gotchas
   360‚Üí   360‚Üí
   361‚Üí   361‚Üí- **Snippet format**: wrap the identifying heading text in straight double quotes (e.g. `'icon tile above h3 "Lightning Fast"'`) so the fixture test can extract it. For rules not anchored to a heading, pick another stable identifier.
   362‚Üí   362‚Üí- **jsdom doesn't lay out**: `getBoundingClientRect()` returns 0√ó0. Read `parseFloat(style.width)` and `parseFloat(style.height)` from explicit CSS instead.
   363‚Üí   363‚Üí- **`background:` shorthand isn't decomposed in jsdom**: use the existing `resolveBackground()` and `resolveGradientStops()` helpers (~line 631 / 670).
   364‚Üí   364‚Üí- **Computed colors aren't normalized in jsdom**: `parseGradientColors()` handles both hex and rgb forms.
   365‚Üí   365‚Üí
   366‚Üí   366‚ÜíReference rules to copy from: `side-tab` (border, ~line 312), `low-contrast` (color + gradient, ~line 339), `icon-tile-stack` (sibling relationship, ~line 425), `flat-type-hierarchy` (page-level, ~line 1080).

   382‚Üí   382‚Üí
