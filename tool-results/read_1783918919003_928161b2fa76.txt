     6тЖТ     6тЖТ# tasteskill: Anti-Slop Frontend Skill
     7тЖТ     7тЖТ
     8тЖТ     8тЖТ> Landing pages, portfolios, and redesigns. Not dashboards, not data tables, not multi-step product UI.
     9тЖТ     9тЖТ> Every rule below is **contextual**. None of it fires automatically. First read the brief, then pull only what fits.
    10тЖТ    10тЖТ
    11тЖТ    11тЖТ---
    12тЖТ    12тЖТ
    13тЖТ    13тЖТ## 0. BRIEF INFERENCE (Read the Room Before Anything Else)
    14тЖТ    14тЖТ
    15тЖТ    15тЖТBefore touching code or tweaking dials, **infer what the user actually wants**. Most LLM design output is bad because the model jumps to a default aesthetic instead of reading the room.
    16тЖТ    16тЖТ
    17тЖТ    17тЖТ### 0.A Read these signals first
    18тЖТ    18тЖТ1. **Page kind** - landing (SaaS / consumer / agency / event), portfolio (dev / designer / creative studio), redesign (preserve vs overhaul), editorial / blog.
    19тЖТ    19тЖТ2. **Vibe words** the user used - "minimalist", "calm", "Linear-style", "Awwwards", "brutalist", "premium consumer", "Apple-y", "playful", "serious B2B", "editorial", "agency-y", "glassy", "dark tech".
    20тЖТ    20тЖТ3. **Reference signals** - URLs they linked, screenshots they pasted, products they named, brands they're competing with.
    21тЖТ    21тЖТ4. **Audience** - B2B procurement panel vs. design-conscious consumer vs. recruiter scanning a portfolio. The audience picks the aesthetic, not your taste.
    22тЖТ    22тЖТ5. **Brand assets that already exist** - logo, color, type, photography. For redesigns, these are starting material, not optional input (see Section 11).
    23тЖТ    23тЖТ6. **Quiet constraints** - accessibility-first audiences, public-sector, regulated industries, trust-first commerce, kids' products. These constraints OVERRIDE aesthetic preference.
    24тЖТ    24тЖТ
    25тЖТ    25тЖТ### 0.B Output a one-line "Design Read" before generating
    26тЖТ    26тЖТBefore any code, state in one line: **"Reading this as: \<page kind> for \<audience>, with a \<vibe> language, leaning toward \<design system or aesthetic family>."**
    27тЖТ    27тЖТ
    28тЖТ    28тЖТExample reads:
    29тЖТ    29тЖТ- *"Reading this as: B2B SaaS landing for technical buyers, with a Linear-style minimalist language, leaning toward Tailwind utilities + Geist + restrained motion."*
    30тЖТ    30тЖТ- *"Reading this as: solo designer portfolio for hiring managers, with an editorial / kinetic-type language, leaning toward native CSS + scroll-driven animation + custom typography."*
    31тЖТ    31тЖТ- *"Reading this as: redesign of a public-sector service site, with a trust-first language, leaning toward GOV.UK Frontend or USWDS."*
    32тЖТ    32тЖТ
    33тЖТ    33тЖТ### 0.C If the brief is ambiguous, ask one question, do not guess
    34тЖТ    34тЖТAsk exactly **one** clarifying question - never a multi-question dump - and only when the design read genuinely diverges. Example: *"Should this feel closer to Linear-clean or Awwwards-experimental?"*
    35тЖТ    35тЖТ
    36тЖТ    36тЖТIf you can confidently infer from context, **do not ask**. Just declare the design read and proceed.
    37тЖТ    37тЖТ
    38тЖТ    38тЖТ### 0.D Anti-Default Discipline
    39тЖТ    39тЖТDo not default to: AI-purple gradients, centered hero over dark mesh, three equal feature cards, generic glassmorphism on everything, infinite-loop micro-animations everywhere, Inter + slate-900. These are the LLM defaults. Reach past them deliberately based on the design read.

    43тЖТ    43тЖТ## 1. THE THREE DIALS (Core Configuration)
    44тЖТ    44тЖТ
    45тЖТ    45тЖТAfter the design read, set three dials. Every layout, motion, and density decision below is gated by these.
    46тЖТ    46тЖТ
    47тЖТ    47тЖТ* **`DESIGN_VARIANCE: 8`** - 1 = Perfect Symmetry, 10 = Artsy Chaos
    48тЖТ    48тЖТ* **`MOTION_INTENSITY: 6`** - 1 = Static, 10 = Cinematic / Physics
    49тЖТ    49тЖТ* **`VISUAL_DENSITY: 4`** - 1 = Art Gallery / Airy, 10 = Cockpit / Packed Data
    50тЖТ    50тЖТ
    51тЖТ    51тЖТ**Baseline:** `8 / 6 / 4`. Use these unless the design read overrides them. Do not ask the user to edit this file - overrides happen conversationally.
    52тЖТ    52тЖТ
    53тЖТ    53тЖТ### 1.A Dial Inference (design read тЖТ dial values)
    54тЖТ    54тЖТ| Signal | VARIANCE | MOTION | DENSITY |
    55тЖТ    55тЖТ|---|---|---|---|
    56тЖТ    56тЖТ| "minimalist / clean / calm / editorial / Linear-style" | 5-6 | 3-4 | 2-3 |
    57тЖТ    57тЖТ| "premium consumer / Apple-y / luxury / brand" | 7-8 | 5-7 | 3-4 |
    58тЖТ    58тЖТ| "playful / wild / Dribbble / Awwwards / experimental / agency" | 9-10 | 8-10 | 3-4 |
    59тЖТ    59тЖТ| "landing page / portfolio / marketing site (default)" | 7-9 | 6-8 | 3-5 |
    60тЖТ    60тЖТ| "trust-first / public-sector / regulated / accessibility-critical" | 3-4 | 2-3 | 4-5 |
    61тЖТ    61тЖТ| "redesign - preserve" | match existing | +1 | match existing |
    62тЖТ    62тЖТ| "redesign - overhaul" | +2 | +2 | match existing |

    82тЖТ    82тЖТ## 2. BRIEF тЖТ DESIGN SYSTEM MAP
    83тЖТ    83тЖТ
    84тЖТ    84тЖТOnce you have the design read (Section 0) and dials (Section 1), pick the right foundation. Do not invent CSS for things that have an official package. Do not pretend an aesthetic trend is an official system.
    85тЖТ    85тЖТ
    86тЖТ    86тЖТ### 2.A When to reach for a real design system (use official packages)
    87тЖТ    87тЖТ| Brief reads asтАж | Reach for | Why |
    88тЖТ    88тЖТ|---|---|---|
    89тЖТ    89тЖТ| Microsoft / enterprise SaaS / dashboards | `@fluentui/react-components` or `@fluentui/web-components` | Official Fluent UI, Microsoft tokens, accessibility done |
    90тЖТ    90тЖТ| Google-ish UI, Material-flavored product | `@material/web` + Material 3 tokens | Official, theme-able via Material Theming |
    91тЖТ    91тЖТ| IBM-style B2B / enterprise analytics | `@carbon/react` + `@carbon/styles` | Official Carbon, mature data-density patterns |
    92тЖТ    92тЖТ| Shopify app surfaces | `polaris.js` web components / Polaris React | Required for Shopify admin UI |
    93тЖТ    93тЖТ| Atlassian / Jira-style product | `@atlaskit/*` + `@atlaskit/tokens` | Official Atlassian DS |
    94тЖТ    94тЖТ| GitHub-style devtool / community page | `@primer/css` or `@primer/react-brand` | Official Primer; Brand variant for marketing |
    95тЖТ    95тЖТ| Public-sector UK service | `govuk-frontend` | Legally / regulatorily expected |
    96тЖТ    96тЖТ| US public-sector / trust-first | `uswds` | Same |
    97тЖТ    97тЖТ| Fast local-business / agency MVP | Bootstrap 5.3 | Boring, fast, works |
    98тЖТ    98тЖТ| Modern accessible React foundation | `@radix-ui/themes` | Primitives + polished theme |
    99тЖТ    99тЖТ| Modern SaaS where you own the components | shadcn/ui (`npx shadcn@latest add ...`) | You own the code, easy to customise; never ship default state |
   100тЖТ   100тЖТ| Tailwind-based modern SaaS / AI marketing | Tailwind v4 utilities + `dark:` variant | Default for indie + small team builds |
   101тЖТ   101тЖТ
   102тЖТ   102тЖТ**Honesty rule:** if the brief reads as one of the systems above, install and use the **official** package. Do not recreate its CSS by hand. Do not import a system's tokens but then override 90% of them.
   103тЖТ   103тЖТ
   104тЖТ   104тЖТ**One system per project.** Do not mix Fluent React with Carbon in the same tree. Do not import shadcn/ui components into a Material 3 app.
   105тЖТ   105тЖТ
   106тЖТ   106тЖТ### 2.B When the brief is an aesthetic, not a system
   107тЖТ   107тЖТFor these directions, there is **no single official package**. Build with native CSS + Tailwind + a maintained component library. Be honest in code comments about what is borrowed inspiration vs. official material.
   108тЖТ   108тЖТ
   109тЖТ   109тЖТ| Aesthetic | Honest implementation |
   110тЖТ   110тЖТ|---|---|
   111тЖТ   111тЖТ| Glassmorphism / "frosted glass" | `backdrop-filter`, layered borders, highlight overlays. Provide solid-fill fallback for `prefers-reduced-transparency`. |
   112тЖТ   112тЖТ| Bento (Apple-style tile grids) | CSS Grid with mixed cell sizes. No single library owns this. |
   113тЖТ   113тЖТ| Brutalism | Native CSS, monospace, raw borders. No library. |
   114тЖТ   114тЖТ| Editorial / magazine | Serif type, asymmetric grid, generous whitespace. No library. |
   115тЖТ   115тЖТ| Dark tech / hacker | Mono + accent neon, terminal motifs. No library. |
   116тЖТ   116тЖТ| Aurora / mesh gradients | SVG or layered radial gradients. No library. |
   117тЖТ   117тЖТ| Kinetic typography | Native CSS animations, scroll-driven animations, GSAP for hijacks. No library. |
   118тЖТ   118тЖТ| **Apple Liquid Glass** | Apple documents this for Apple platforms only. **There is no official `liquid-glass.css`.** Web implementations are approximations using `backdrop-filter` + layered borders + highlights. Label clearly as approximation. |

   122тЖТ   122тЖТ## 3. DEFAULT ARCHITECTURE & CONVENTIONS
   123тЖТ   123тЖТ
   124тЖТ   124тЖТUnless the design read picks a real design system (Section 2.A), these are the defaults:
   125тЖТ   125тЖТ
   126тЖТ   126тЖТ### 3.A Stack
   127тЖТ   127тЖТ* **Framework:** React or Next.js. Default to Server Components (RSC).
   128тЖТ   128тЖТ  * **RSC SAFETY:** Global state works ONLY in Client Components. In Next.js, wrap providers in a `"use client"` component.
   129тЖТ   129тЖТ  * **INTERACTIVITY ISOLATION:** Any component using Motion, scroll listeners, or pointer physics MUST be an isolated leaf with `'use client'` at the top. Server Components render static layouts only.
   130тЖТ   130тЖТ* **Styling:** **Tailwind v4** (default). Tailwind v3 only if the existing project demands it.
   131тЖТ   131тЖТ  * For v4: do NOT use `tailwindcss` plugin in `postcss.config.js`. Use `@tailwindcss/postcss` or the Vite plugin.
   132тЖТ   132тЖТ* **Animation:** **Motion** (the library formerly known as Framer Motion). Import from `motion/react` (`import { motion } from "motion/react"`). The `framer-motion` package still works as a legacy alias - prefer `motion/react` in new code.
   133тЖТ   133тЖТ* **Fonts:** Always use `next/font` (Next.js) or self-host with `@font-face` + `font-display: swap`. Never link Google Fonts via `<link>` in production.
   134тЖТ   134тЖТ
   135тЖТ   135тЖТ### 3.B State
   136тЖТ   136тЖТ* Local `useState` / `useReducer` for isolated UI.
   137тЖТ   137тЖТ* Global state ONLY for deep prop-drilling avoidance - Zustand, Jotai, or React context.
   138тЖТ   138тЖТ* **NEVER** use `useState` to track continuous values driven by user input (mouse position, scroll progress, pointer physics, magnetic hover). Use Motion's `useMotionValue` / `useTransform` / `useScroll`. `useState` re-renders the React tree on every change and collapses on mobile.
   139тЖТ   139тЖТ
   140тЖТ   140тЖТ### 3.C Icons
   141тЖТ   141тЖТ* **Allowed libraries (priority order):** `@phosphor-icons/react`, `hugeicons-react`, `@radix-ui/react-icons`, `@tabler/icons-react`.
   142тЖТ   142тЖТ* **Discouraged:** `lucide-react`. Acceptable only when the user explicitly asks for it or the project already depends on it.
   143тЖТ   143тЖТ* **NEVER hand-roll SVG icons.** If a glyph is missing, install a second library or compose from primitives - do not draw icon paths from scratch.
   144тЖТ   144тЖТ* **One family per project.** Do not mix Phosphor with Lucide in the same component tree.
   145тЖТ   145тЖТ* **Standardize `strokeWidth` globally** (e.g. `1.5` or `2.0`).
   146тЖТ   146тЖТ
   147тЖТ   147тЖТ### 3.D Emoji Policy
   148тЖТ   148тЖТDiscouraged by default in code, markup, and visible text. Replace symbols with icon-library glyphs. **Override:** allow emojis only when the user explicitly asks for a playful / chat-style / social-native vibe - and even then use them sparingly with intent.
   149тЖТ   149тЖТ
   150тЖТ   150тЖТ### 3.E Responsiveness & Layout Mechanics
   151тЖТ   151тЖТ* Standardize breakpoints (`sm 640`, `md 768`, `lg 1024`, `xl 1280`, `2xl 1536`).
   152тЖТ   152тЖТ* Contain page layouts using `max-w-[1400px] mx-auto` or `max-w-7xl`.
   153тЖТ   153тЖТ* **Viewport Stability:** NEVER use `h-screen` for full-height Hero sections. ALWAYS use `min-h-[100dvh]` to prevent layout jumping on mobile (iOS Safari address bar).
   154тЖТ   154тЖТ* **Grid over Flex-Math:** NEVER use complex flexbox percentage math (`w-[calc(33%-1rem)]`). ALWAYS use CSS Grid (`grid grid-cols-1 md:grid-cols-3 gap-6`).
   155тЖТ   155тЖТ
   156тЖТ   156тЖТ### 3.F Dependency Verification (mandatory)
   157тЖТ   157тЖТBefore importing ANY 3rd-party library, check `package.json`. If the package is missing, output the install command first. **Never** assume a library exists.

   161тЖТ   161тЖТ## 4. DESIGN ENGINEERING DIRECTIVES (Bias Correction)
   162тЖТ   162тЖТ
   163тЖТ   163тЖТLLMs default to clich├йs. Override these defaults proactively. Each rule has a context-aware override path.
   164тЖТ   164тЖТ
   165тЖТ   165тЖТ### 4.1 Typography
   166тЖТ   166тЖТ* **Display / Headlines:** Default `text-4xl md:text-6xl tracking-tighter leading-none`.
   167тЖТ   167тЖТ* **Body / Paragraphs:** Default `text-base text-gray-600 leading-relaxed max-w-[65ch]`.
   168тЖТ   168тЖТ* **Sans font choice:**
   169тЖТ   169тЖТ  * **Discouraged as default:** `Inter`. Pick `Geist`, `Outfit`, `Cabinet Grotesk`, `Satoshi`, or a brand-appropriate serif first.
   170тЖТ   170тЖТ  * **Override:** Inter is acceptable when the user explicitly asks for a neutral / standard / Linear-style feel, or when the brief is a public-sector / accessibility-first site.
   171тЖТ   171тЖТ* **Pairings to know:** `Geist` + `Geist Mono`, `Satoshi` + `JetBrains Mono`, `Cabinet Grotesk` + `Inter Tight`, `GT America` + `IBM Plex Mono`.
   172тЖТ   172тЖТ
   173тЖТ   173тЖТ* **SERIF DISCIPLINE (VERY DISCOURAGED AS DEFAULT):**
   174тЖТ   174тЖТ  * Serif is **very discouraged as the default font for any project.** "It feels creative / premium / editorial" is NOT a reason to reach for serif. The agent's default mental model that "creative brief = serif" is the single most-tested AI tell in production rounds.
   175тЖТ   175тЖТ  * **Serif is only acceptable when ONE of these is explicitly true:**
   176тЖТ   176тЖТ    - The brand brief literally names a serif font, OR
   177тЖТ   177тЖТ    - The aesthetic family is genuinely editorial / luxury / publication / manuscript / heritage / vintage AND you can articulate why this specific serif fits this specific brand
   178тЖТ   178тЖТ  * For everything else (creative agency, design studio, modern brand, premium consumer, portfolio, lifestyle), **default sans-serif display** (Geist Display, ABC Diatype, S├╢hne Breit, Cabinet Grotesk Display, Migra Sans, GT Walsheim, Inter Display, PP Neue Montreal). Sans display fonts are not "boring" тАФ they are the default for the same reason black is the default in fashion.
   179тЖТ   179тЖТ  * **EMPHASIS RULE (related):** When you want to emphasize a word within a headline (the kinetic "and `spatial` design" type move), use **italic or bold of the SAME font**. Do NOT inject a random serif word into a sans headline (or vice versa) just to add visual interest. Mixed-family emphasis is amateur. Italic/bold emphasis in the same family is the right move.
   180тЖТ   180тЖТ  * **Specifically BANNED as defaults:** `Fraunces` and `Instrument_Serif` (the two LLM-favorite display serifs).
   181тЖТ   181тЖТ  * **If a serif is justified** (rare, per the above), rotate from this pool, do NOT reuse the same serif across consecutive projects: PP Editorial New, GT Sectra Display, Cardinal Grotesque, Reckless Neue, Tiempos Headline, Recoleta, Cormorant Garamond, Playfair Display, EB Garamond, IvyPresto, Migra, Editorial Old, Saol Display, S├╢hne Breit Kursiv, Domaine Display, Canela, Schnyder, Tobias, NB Architekt, ITC Galliard.
   182тЖТ   182тЖТ
   183тЖТ   183тЖТ* **ITALIC DESCENDER CLEARANCE (mandatory):** When italic is used in display type and the word contains a descender letter (`y g j p q`), `leading-[1]` or `leading-none` will clip the descender. Use `leading-[1.1]` minimum and add `pb-1` or `mb-1` reserve on the wrapping element. Audit every italic word in display headlines before shipping.
   184тЖТ   184тЖТ
   185тЖТ   185тЖТ### 4.2 Color Calibration
   186тЖТ   186тЖТ* Max 1 accent color. Saturation < 80% by default.
   187тЖТ   187тЖТ* **THE LILA RULE:** The "AI Purple / Blue glow" aesthetic is discouraged as a default. No automatic purple button glows, no random neon gradients. Use neutral bases (Zinc / Slate / Stone) with high-contrast singular accents (Emerald, Electric Blue, Deep Rose, Burnt Orange, etc.).
   188тЖТ   188тЖТ* **Override:** if the brand or brief explicitly asks for purple / violet / lila, embrace it. But execute with intent: consistent palette, harmonised neutrals, restrained gradients. Not generic AI gradient slop.
   189тЖТ   189тЖТ* **One palette per project.** Do not fluctuate between warm and cool grays within the same project.
   190тЖТ   190тЖТ* **COLOR CONSISTENCY LOCK (mandatory):** Once an accent color is chosen for a page, it is used on the WHOLE page. A warm-grey site does not suddenly get a blue CTA in section 7. A rose-accented site does not get a teal status badge in the footer. Pick one accent, lock it, audit every component before shipping.
   191тЖТ   191тЖТ
   192тЖТ   192тЖТ* **PREMIUM-CONSUMER PALETTE BAN (mandatory, second-most-recurring AI-tell):**
   193тЖТ   193тЖТ  * For premium-consumer briefs (cookware, wellness, artisan, luxury, heritage craft, DTC home goods, etc.) the LLM default is **warm beige/cream + brass/clay/oxblood/ochre + espresso/ink dark text**. Concretely banned hex families as default backgrounds and accents:
   194тЖТ   194тЖТ    - Backgrounds: `#f5f1ea`, `#f7f5f1`, `#fbf8f1`, `#efeae0`, `#ece6db`, `#faf7f1`, `#e8dfcb` (all "warm paper / cream / chalk / bone")
   195тЖТ   195тЖТ    - Accents: `#b08947`, `#b6553a`, `#9a2436`, `#9c6e2a`, `#bc7c3a`, `#7d5621` (all "brass / clay / oxblood / ochre")
   196тЖТ   196тЖТ    - Text: `#1a1714`, `#1a1814`, `#1b1814` (all "espresso / warm near-black")
   197тЖТ   197тЖТ  * This palette is BANNED as the default reach for premium-consumer briefs. Every premium-consumer site you have ever shipped uses this exact palette. The brand becomes invisible.
   198тЖТ   198тЖТ  * **Default alternatives (rotate, do not reuse):**
   199тЖТ   199тЖТ    - **Cold Luxury:** silver-grey + chrome + smoke (think Tesla, Apple Watch Hermes-without-the-leather)
   200тЖТ   200тЖТ    - **Forest:** deep green + bone + amber accent (think Filson, Patagonia premium)
   201тЖТ   201тЖТ    - **Black and Tan:** true off-black + warm tan, sharp contrast, no beige
   202тЖТ   202тЖТ    - **Cobalt + Cream:** saturated blue against a single neutral, no brass
   203тЖТ   203тЖТ    - **Terracotta + Slate:** warm rust against cool grey, no brass
   204тЖТ   204тЖТ    - **Olive + Brick + Paper:** muted olive plus brick-red accent
   205тЖТ   205тЖТ    - **Pure monochrome + single saturated pop:** off-white + off-black + one bright accent (electric blue, emerald, hot pink, etc.)
   206тЖТ   206тЖТ  * **Palette-rotation rule:** if the previous premium-consumer project you generated used the beige+brass family, this one MUST use a different family. Do not ship the same warm-craft palette twice in a row.
   207тЖТ   207тЖТ  * **Override:** the beige+brass+espresso palette is acceptable ONLY when the brand brief explicitly names those colors, or when the brand identity is genuinely vintage / artisan / warm-craft AND you can articulate why this specific palette fits this specific brand. Default-reaching for it because "this is a cookware brief" is banned.
   208тЖТ   208тЖТ
   209тЖТ   209тЖТ### 4.3 Layout Diversification
   210тЖТ   210тЖТ* **ANTI-CENTER BIAS:** Centered Hero / H1 sections are avoided when `DESIGN_VARIANCE > 4`. Force "Split Screen" (50/50), "Left-aligned content / right-aligned asset", "Asymmetric white-space", or scroll-pinned structures.
   211тЖТ   211тЖТ* **Override:** centered hero is OK for editorial / manifesto / launch-announcement briefs where the message itself is the design.
   212тЖТ   212тЖТ
   213тЖТ   213тЖТ### 4.4 Materiality, Shadows, Cards
   214тЖТ   214тЖТ* Use cards ONLY when elevation communicates real hierarchy. Otherwise group with `border-t`, `divide-y`, or negative space.
   215тЖТ   215тЖТ* When a shadow is used, tint it to the background hue. No pure-black drop shadows on light backgrounds.
   216тЖТ   216тЖТ* For `VISUAL_DENSITY > 7`: generic card containers are banned. Data metrics breathe in plain layout.
   217тЖТ   217тЖТ* **SHAPE CONSISTENCY LOCK (mandatory):** Pick ONE corner-radius scale for the page and stick to it. Options: all-sharp (radius 0), all-soft (radius 12-16px), all-pill (full radius for interactive). Mixed systems are allowed only when there is a documented rule (e.g. "buttons are full-pill, cards are 16px, inputs are 8px") and that rule is followed everywhere. Round buttons in a square layout, or square cards on a pill-button page, is broken design.
   218тЖТ   218тЖТ
   219тЖТ   219тЖТ### 4.5 Interactive UI States
   220тЖТ   220тЖТLLMs default to "static successful state only." Always implement full cycles:
   221тЖТ   221тЖТ* **Loading:** Skeletal loaders matching the final layout's shape. Avoid generic circular spinners.
   222тЖТ   222тЖТ* **Empty States:** Beautifully composed; indicate how to populate.
   223тЖТ   223тЖТ* **Error States:** Clear, inline (forms), or contextual (toasts only for transient).
   224тЖТ   224тЖТ* **Tactile Feedback:** On `:active`, use `-translate-y-[1px]` or `scale-[0.98]` to simulate a physical push.
   225тЖТ   225тЖТ* **BUTTON CONTRAST CHECK (mandatory, a11y):** Before shipping any button, verify the button text is readable against the button background. White button + white text, `bg-white` CTA with `text-white` label, transparent button against the page background with no border тЖТ all banned. Audit every CTA: contrast ratio WCAG AA min (4.5:1 for body, 3:1 for large text 18px+). Same rule applies to ghost buttons over photographic backgrounds (use a backdrop, scrim, or stroke).
   226тЖТ   226тЖТ* **CTA BUTTON WRAP BAN (mandatory):** Button text MUST fit on one line at desktop. If a label like "VIEW SELECTED WORK" wraps to 2 or 3 lines, the button is broken. Fix by EITHER shortening the label (3 words max for primary CTAs, ideally 1-2) OR widening the button (do not artificially constrain `max-width` on CTAs). Wrapped CTAs at desktop are a Pre-Flight Fail.
   227тЖТ   227тЖТ* **NO DUPLICATE CTA INTENT (mandatory):** Two CTAs with the same intent on one page is a Pre-Flight Fail. Examples of same intent: "Get in touch" + "Contact us" + "Let's talk" + "Start a project" + "Start something" + "Reach out" = all "contact" intent тЖТ pick ONE label and use it everywhere on the page (nav, hero, footer). Same for "Try free" + "Get started" + "Sign up free" (all "signup" intent) and "View work" + "See selected work" + "Browse projects" (all "portfolio" intent). One label per intent.
   228тЖТ   228тЖТ* **FORM CONTRAST CHECK (mandatory, a11y):** Form inputs, placeholder text, focus rings, helper text, and error text all pass WCAG AA contrast against the section background. Light placeholders on a near-white form, white form on white page section, form labels grayer than 4.5:1 contrast тЖТ all banned. Audit every form before shipping.
   229тЖТ   229тЖТ
   230тЖТ   230тЖТ### 4.6 Data & Form Patterns
   231тЖТ   231тЖТ* Label ABOVE input. Helper text optional but present in markup. Error text BELOW input. Standard `gap-2` for input blocks.
   232тЖТ   232тЖТ* No placeholder-as-label. Ever.
   233тЖТ   233тЖТ
   234тЖТ   234тЖТ### 4.7 Layout Discipline (Hard Rules. Failing any of these is shipping broken work)
   235тЖТ   235тЖТ
   236тЖТ   236тЖТ* **Hero MUST fit in the initial viewport.** Headline max 2 lines on desktop, subtext max **20 words** AND max 3-4 lines, CTAs visible without scroll. If the copy is too long: reduce font scale OR cut copy. If you cannot describe the value-prop in 20 words of subtext, the value-prop is unclear, not the rule too tight. Never let the hero overflow and force scroll to find the CTA.
   237тЖТ   237тЖТ* **Hero font-scale discipline.** Plan font size and image size *together*. If the hero asset is large and the headline is more than 6 words, do not start at `text-7xl/text-8xl`. Default sensible range: `text-4xl md:text-5xl lg:text-6xl` for most heroes; `text-6xl md:text-7xl` only when the headline is 3-5 words. A 4-line hero headline is always a font-size error, never a copy-length error.
   238тЖТ   238тЖТ* **HERO TOP PADDING CAP (mandatory):** Hero top padding max `pt-24` (тЙИ6rem) at desktop. More than that means the hero content floats halfway down the viewport and reads as a layout bug, not as intentional space. If your hero needs more breathing room, increase font scale or asset size, not top padding.
   239тЖТ   239тЖТ* **HERO STACK DISCIPLINE (max 4 text elements).** The hero is a single moment, not a feature list. Allowed text elements, max 4 in total:
   240тЖТ   240тЖТ  1. Eyebrow (small uppercase label) OR brand strip OR neither - pick zero or one
   241тЖТ   241тЖТ  2. Headline (max 2 lines, see above)
   242тЖТ   242тЖТ  3. Subtext (max 20 words, max 4 lines)
   243тЖТ   243тЖТ  4. CTAs (1 primary + max 1 secondary)
   244тЖТ   244тЖТ  - **BANNED in the hero:** tiny tagline below CTAs ("Works with GitHub, GitLab, and self-hosted Git"), trust micro-strip ("Used by engineering teams at..."), pricing teaser ("Free for solo, $10/user for teams"), feature bullet list, social-proof avatar row. All of those move to dedicated sections directly below the hero.
   245тЖТ   245тЖТ  - If you have an eyebrow AND a tagline below CTAs in the same hero, drop the tagline. If you have a brand strip AND a tagline, drop the tagline. One small text element per hero, max.
   246тЖТ   246тЖТ* **"Used by" / "Trusted by" logo wall belongs UNDER the hero, never inside it.** The hero is for the value prop and primary CTA. The logo wall is a separate section directly below. Do not stuff trust logos into the same flex row as the hero copy.
   247тЖТ   247тЖТ* **Navigation MUST render on a single line on desktop.** If items don't fit at `lg` (1024px), condense labels, drop secondary items, or move to a hamburger. A two-line nav at desktop is broken design.
   248тЖТ   248тЖТ* **Navigation height cap: 80px max desktop, default 64-72px.** No huge "agency" nav bars that eat 15% of the viewport.
   249тЖТ   249тЖТ* **Bento grids MUST have rhythm, not one-sided repetition.** Do not stack 6 left-image / right-text rows. Vary the composition: alternate full-width feature rows, asymmetric tile sizes, vertical breaks.
   250тЖТ   250тЖТ* **BENTO CELL COUNT RULE (mandatory):** A bento grid has EXACTLY as many cells as you have content for. 3 items тЖТ 3 cells (1+2 split, or 2+1, or asymmetric trio). 5 items тЖТ 5 cells (2+3, 3+2, hero+4, etc.). If your grid has an empty cell in the middle or at the end, you planned wrong. Re-shape the grid; do not paste a blank tile.
   251тЖТ   251тЖТ* **Section-Layout-Repetition Ban.** Once you use a layout family for a section (e.g., 3-column-image-cards, full-width-quote, split-text-image), that family can appear at most ONCE on the page. "Selected commissions" must not look like "What we do." A landing page with 8 sections must use at least 4 different layout families.
   252тЖТ   252тЖТ* **ZIGZAG ALTERNATION CAP (mandatory).** Alternating "left-image + right-text" then "left-text + right-image" zigzag layout = banal. Max 2 sections in a row with this image+text-split pattern. The 3rd consecutive image+text split is a Pre-Flight Fail. Break the pattern with a full-width section, a vertical-stack section, a bento grid, a marquee, or a different layout family.
   253тЖТ   253тЖТ* **EYEBROW RESTRAINT (mandatory, the #1 violated rule in production tests).** An "eyebrow" is the small uppercase wide-tracking label sitting above a section headline (e.g. `FOUR COLORWAYS`, `SELECTED WORK`, `THE HARDWARE`, `Git-native task management`). Typical CSS signature: `text-[11px] uppercase tracking-[0.18em]`, `font-mono text-[10.5px] uppercase tracking-[0.22em]`. Every AI-built site puts an eyebrow above EVERY section header, producing the same templated rhythm. Hard rule:
   254тЖТ   254тЖТ  - **Maximum 1 eyebrow per 3 sections.** Hero counts as 1. So a page with 9 sections may use at most 3 eyebrows total.
   255тЖТ   255тЖТ  - If section A has an eyebrow, the next 2 sections cannot have one.
   256тЖТ   256тЖТ  - **Pre-Flight Check is mechanical:** count instances of `uppercase tracking` (or similar small-caps mono labels above headlines) across all section components. If count > ceil(sectionCount / 3), the output fails.
   257тЖТ   257тЖТ  - **What to do instead of an eyebrow:** drop it entirely. The headline alone is enough. If you need to categorize a section, the section's location on the page already categorizes it; no label needed.
   258тЖТ   258тЖТ* **SPLIT-HEADER BAN (mandatory).** The pattern "left big headline + right small explainer paragraph" as a section header (left col-span-7/8, right col-span-4/5 with a small body paragraph floating in the right column) is **banned as default**. Sections should have ONE focused message. If you genuinely need both a headline and an explainer paragraph, stack them vertically (headline on top, body below, max-width 65ch). Reach for the split-header pattern only when there is a real compositional reason (e.g., the right column carries a visual or interactive element, not just filler text).
   259тЖТ   259тЖТ* **Bento Background Diversity (mandatory).** Bento and feature-grid sections cannot be 6 white-on-white cards with text inside. At least 2-3 cells in any multi-cell grid need real visual variation: a real image, a brand-appropriate gradient (not AI-purple), a pattern, a tinted background. A cream-on-cream bento with only typography inside reads as boring AI default, even when the rest of the page is good.
   260тЖТ   260тЖТ* **Mobile collapse must be explicit per section.** For every multi-column layout, declare the `< 768px` fallback in the same component. No "it'll work, Tailwind handles it" assumptions.
   261тЖТ   261тЖТ
   262тЖТ   262тЖТ### 4.8 Image & Visual Asset Strategy
   263тЖТ   263тЖТ
   264тЖТ   264тЖТLanding pages and portfolios are **visual products**. Text-only pages with fake-screenshot divs are slop.
   265тЖТ   265тЖТ
   266тЖТ   266тЖТ**Priority order for visual assets:**
   267тЖТ   267тЖТ1. **Image-generation tool first.** If ANY image-gen tool is available in the environment (`generate_image`, MCP image tool, IDE-integrated gen, OpenAI image tools, etc.) you MUST use it to create section-specific assets: hero photography, product shots, texture backgrounds, mood images. Generate at the right aspect ratio for the section. Do not skip this step because hand-rolled CSS feels faster.
   268тЖТ   268тЖТ2. **Real web images second.** When no gen tool is available, use real photography sources. Acceptable defaults:
   269тЖТ   269тЖТ   * `https://picsum.photos/seed/{descriptive-seed}/{w}/{h}` for placeholder photography (seed should describe the section, e.g. `marrow-cookware-kitchen`)
   270тЖТ   270тЖТ   * Actual stock or brand URLs when the brief provides them
   271тЖТ   271тЖТ   * Open-license sources (Unsplash via direct URL, Pexels) if explicitly allowed
   272тЖТ   272тЖТ3. **Last resort: tell the user.** If neither is possible, do NOT fill the page with hand-rolled SVG illustrations or div-based "fake screenshots." Instead, leave clearly-labeled placeholder slots (`<!-- TODO: hero product photo, 1600x1200 -->`) and at the end of the response say: *"This page needs real images at: \[list of placements\]. Please generate or provide them."*
   273тЖТ   273тЖТ
   274тЖТ   274тЖТ**Even minimalist sites need real images.** A pure-text page is not minimalism. It is incomplete work. Even an editorial Linear-style site needs at least 2-3 real images (hero, one product/lifestyle shot, one supporting image). Generate B&W minimalist photography if the brief is restrained; do not skip images entirely because the dial is low.
   275тЖТ   275тЖТ
   276тЖТ   276тЖТ**Real company logos for social proof.** When the brief calls for a "Trusted by / Used by / Customers" logo wall, do NOT default to plain text wordmarks (`<span>Acme Co</span>` styled in a row). Use real SVG logos:
   277тЖТ   277тЖТ* **Source: Simple Icons** (`https://cdn.simpleicons.org/{slug}/ffffff` for any color, or `simple-icons` npm package). Covers most known brands.
   278тЖТ   278тЖТ* **Alternative: devicon** for tech-stack logos (`@svgr/cli` or CDN).
   279тЖТ   279тЖТ* **Make-up the brand name? Then make-up an SVG mark too.** Generate a simple monogram (one letter in a circle, two-letter ligature, abstract glyph) rendered as an inline `<svg>` matching the page style. Plain text wordmarks for invented brand names look generic.
   280тЖТ   280тЖТ* **Always** ensure logos render in both light and dark mode (white-on-dark, black-on-light, or single-color theme variable).
   281тЖТ   281тЖТ* **LOGO-ONLY rule (mandatory):** logo wall = logos and nothing else. Do NOT print industry / category labels below each logo (no `Vercel` + `hosting` underneath, no `Stripe` + `payments`, no `Cloudflare` + `infra`). The logo is the credibility, the label adds nothing the user does not already know. Optional: brand name as alt-text for screen readers, optional link to the brand's site. That is it.
   282тЖТ   282тЖТ
   283тЖТ   283тЖТ**Hand-rolled illustrations:**
   284тЖТ   284тЖТ* SVG icons from libraries: fine (see Section 3.C).
   285тЖТ   285тЖТ* Hand-rolled decorative SVGs (custom illustrations, logos, marks): **strongly discouraged**, never as default. Acceptable only when:
   286тЖТ   286тЖТ  - The brief explicitly calls for it ("draw me an SVG logo")
   287тЖТ   287тЖТ  - It's a single, simple geometric mark (a square, a circle, a wordmark in display type)
   288тЖТ   288тЖТ  - You're confident in the output quality
   289тЖТ   289тЖТ
   290тЖТ   290тЖТ**Div-based fake screenshots are banned.** A "hand-built product preview" rendered with `<div>` rectangles, fake task lists, fake dashboards, fake terminal windows is a Tell. If you need to show a product:
   291тЖТ   291тЖТ* Use a real screenshot URL if one exists
   292тЖТ   292тЖТ* Generate one via image tool
   293тЖТ   293тЖТ* Use a real component preview (an actual mini-version of the UI inside the page)
   294тЖТ   294тЖТ* Or skip the preview entirely and use editorial photography
   295тЖТ   295тЖТ
   296тЖТ   296тЖТ**Hero needs a real visual.** Text + gradient blob is not a hero - it's a placeholder.
   297тЖТ   297тЖТ
   298тЖТ   298тЖТ### 4.9 Content Density
   299тЖТ   299тЖТ
   300тЖТ   300тЖТLanding pages live on the **first impression**, not the full read. Cut ruthlessly.
   301тЖТ   301тЖТ
   302тЖТ   302тЖТ* **Default content shape per section:** short headline (тЙд 8 words) + short sub-paragraph (тЙд 25 words) + one visual asset OR one CTA. Anything more must be justified by the section's job.
   303тЖТ   303тЖТ* **No data-dump sections.** A 20-row publication table, a 30-row award list, a giant pricing matrix on a marketing page = wrong layout. Use:
   304тЖТ   304тЖТ  - Top 3-5 highlights + "View full list" link
   305тЖТ   305тЖТ  - Marquee / carousel for breadth
   306тЖТ   306тЖТ  - Different page entirely if the data is the product
   307тЖТ   307тЖТ* **Long lists need a different UI component, not a longer list.** Default `<ul>` with bullets / `divide-y` rows is the lazy choice. If you have > 5 items, reach for one of these instead:
   308тЖТ   308тЖТ  - 2-column split with grouped items
   309тЖТ   309тЖТ  - Card grid with image + label per item
   310тЖТ   310тЖТ  - Tabs / accordion if items are categorisable
   311тЖТ   311тЖТ  - Horizontal scroll-snap pills
   312тЖТ   312тЖТ  - Carousel for breadth-heavy lists (testimonials, logos, capabilities)
   313тЖТ   313тЖТ  - Marquee for "lots-of-things-that-don't-need-individual-attention"
   314тЖТ   314тЖТ  A spec sheet with 10 rows + a hairline under every row is the WORST default. Either group rows into 2-3 chunks with sparse dividers, or move to a card-per-spec layout.
   315тЖТ   315тЖТ* **Spec sheets specifically (the Marrow-cookware pattern).** A long product specification table with `border-b` on every row is the AI default for cookware / hardware / apparel / artisan-goods briefs. Banned. Concrete alternatives:
   316тЖТ   316тЖТ  - **2-col card grid:** each spec gets its own card with the spec name, the value (large display number), and a one-line "why it matters" body. Cards arranged 2-col on desktop, 1-col mobile.
   317тЖТ   317тЖТ  - **Scroll-snap horizontal pills:** each spec is a pill, user can flick through.
   318тЖТ   318тЖТ  - **Grouped chunks:** group 10 specs into 3 logical clusters (e.g. "Materials", "Cooking", "Warranty"), each cluster gets ONE soft divider and a cluster heading.
   319тЖТ   319тЖТ  - **Featured-vs-rest:** 3-4 hero specs visualised as large display tiles, the rest collapsed under a "View full specifications" disclosure.
   320тЖТ   320тЖТ
   321тЖТ   321тЖТ* **COPY SELF-AUDIT (mandatory before ship):** Before declaring any task done, re-read every visible string on the page (headlines, subheads, eyebrows, button labels, body copy, captions, alt text, footer text, error messages). Flag any string that is:
   322тЖТ   322тЖТ  - **Grammatically broken** ("free on its past", "two plans but one is honest", "to put it on the table" out of context)
   323тЖТ   323тЖТ  - **Has unclear referents** ("we plan to stay that way" without prior context)
   324тЖТ   324тЖТ  - **Sounds like AI hallucination** (cute-but-wrong wordplay, forced metaphors that don't track, "elegant nothing" phrases)
   325тЖТ   325тЖТ  - **Reads like an LLM trying to sound thoughtful** (passive-aggressive humility, fake-craftsman labels, mock-poetic micro-meta)
   326тЖТ   326тЖТ  Rewrite every flagged string. If unsure whether a string makes sense, replace it with a plain functional sentence. AI-generated cute copy is worse than boring copy.
   327тЖТ   327тЖТ* **Fake-precise numbers are flagged.** Numbers like `92%`, `4.1├Ч`, `48k`, `5.8 mm`, `13.4 lb` either:
   328тЖТ   328тЖТ  - Come from real data (brief, brand guidelines, public metrics) - fine
   329тЖТ   329тЖТ  - Are explicitly labeled as mock (`<!-- mock -->`, "example", "sample data") - fine
   330тЖТ   330тЖТ  - Are AI-invented spec aesthetics - banned. Don't fake engineering precision the brand doesn't claim.
   331тЖТ   331тЖТ* **One copy register per page.** Don't mix technical mono ("47 tasks ┬╖ 0.6 ctx-switches/day"), editorial prose, and marketing punch in the same composition unless the brand voice explicitly calls for it.
   332тЖТ   332тЖТ
   333тЖТ   333тЖТ### 4.10 Quotes & Testimonials
   334тЖТ   334тЖТ
   335тЖТ   335тЖТ* **Max 3 lines** of quote body. Never 6. If the original quote is longer тЖТ cut it. A landing-page quote is a snippet, not the full review.
   336тЖТ   336тЖТ* For very small font sizes (e.g. footer-style testimonials), the line cap can stretch slightly. Spirit: "fits in a glance."
   337тЖТ   337тЖТ* **No em-dashes inside the quote text** as design flourish (long pauses, kinetic em-dashes, em-dash-bullets). See Section 9.G - em-dash is completely banned.
   338тЖТ   338тЖТ* Attribution: name + role + (optionally) company. Never name only ("- Sarah").
   339тЖТ   339тЖТ* Quote marks: use real typographic quotes ( " " ) or none at all. Not straight ASCII ( " ).
   340тЖТ   340тЖТ
   341тЖТ   341тЖТ### 4.11 Page Theme Lock (Light / Dark Mode Consistency)
   342тЖТ   342тЖТ
   343тЖТ   343тЖТThe page has ONE theme. Sections do not invert.
   344тЖТ   344тЖТ
   345тЖТ   345тЖТ* If the page is dark mode, ALL sections are dark mode. No light-mode-warm-paper section sandwiched between dark sections (or vice versa). The user must not feel they walked into a different website mid-scroll.
   346тЖТ   346тЖТ* The exception: if the brief explicitly calls for a "Color Block Story" or "Theme Switch on Scroll" device AND that is a deliberate composition (one full theme switch with a strong transition, not random alternation), it is allowed once per page.
   347тЖТ   347тЖТ* Default behaviour: pick light, dark, or auto (`prefers-color-scheme`) at the page level and lock it. Section-level background tints within the same theme family are fine (`bg-zinc-950` next to `bg-zinc-900`); flipping to `bg-amber-50` in the middle of a `bg-zinc-950` page is broken.
   348тЖТ   348тЖТ* When using a design system with built-in theming (Radix Themes, shadcn/ui with `<Theme>`), set the theme ONCE in `layout.tsx` or the page root. Do not let individual sections override.

   352тЖТ   352тЖТ## 5. CONTEXT-AWARE PROACTIVITY
   353тЖТ   353тЖТ
   354тЖТ   354тЖТThese are tools, not defaults. Use them when the design read calls for them. **None of these fire automatically.**
   355тЖТ   355тЖТ
   356тЖТ   356тЖТ* **Liquid Glass / Glassmorphism:** Appropriate for premium consumer, Apple-adjacent, luxury brand, or media-overlay vibes. Inappropriate for dashboards, public-sector, or "boring B2B." When used, go beyond `backdrop-blur`: add a 1px inner border (`border-white/10`) and a subtle inner shadow (`shadow-[inset_0_1px_0_rgba(255,255,255,0.1)]`) for physical edge refraction. Provide a solid-fill fallback under `prefers-reduced-transparency`.
   357тЖТ   357тЖТ* **Magnetic Micro-physics:** Use when `MOTION_INTENSITY > 5` AND the brief reads premium / playful / agency. Implement EXCLUSIVELY with Motion's `useMotionValue` / `useTransform` outside the React render cycle. Never `useState`. See Section 3.B.
   358тЖТ   358тЖТ* **Perpetual Micro-Interactions** (Pulse, Typewriter, Float, Shimmer, Carousel): Use when `MOTION_INTENSITY > 5` AND the section actively benefits from motion (status indicators, live feeds, AI-feel). **Not every card needs an infinite loop.** If a section is informational, leave it still. Apply Spring Physics (`type: "spring", stiffness: 100, damping: 20`) - no linear easing.
   359тЖТ   359тЖТ* **"Motion claimed, motion shown."** If `MOTION_INTENSITY > 4`, the page must actually move: entry transitions on hero, scroll-reveal on key sections, hover physics on CTAs, at minimum. A static page that claims `MOTION_INTENSITY: 7` is broken. Conversely, if you cannot ship working motion in the available scope, drop the dial to 3 and ship a clean static page. Never half-build motion that breaks (cut-off ScrollTriggers, jumpy enters, missing cleanups).
   360тЖТ   360тЖТ* **MOTION MUST BE MOTIVATED (mandatory).** Before adding any animation, ask: "what does this animation communicate?" Valid answers: hierarchy (drawing attention to the right thing), storytelling (revealing content in sequence that matches a narrative), feedback (acknowledging a user action), state transition (showing something changed). Invalid answer: "it looked cool". GSAP everywhere because GSAP is available is amateur. Each ScrollTrigger, each marquee, each pinned section needs a reason. If you cannot articulate the reason in one sentence, drop the animation.
   361тЖТ   361тЖТ* **MARQUEE MAX-ONE-PER-PAGE (mandatory).** Horizontal scrolling text marquees ("logos endlessly scrolling", "manifesto scrolling sideways", "kinetic word strip") are appropriate at most ONCE per page. Two or more marquees on the same page reads as lazy filler. Pick the one section where the marquee actually serves the content; the others get a different layout.
   362тЖТ   362тЖТ* **GSAP Sticky-Stack Pattern (when scroll-stack is used).** A "card stack on scroll" must be a REAL sticky-stack, not a sequential reveal list. See Section 5.A below for the canonical code skeleton. Common failure: trigger fires halfway through scroll instead of pinning at viewport top. Fix: `start: "top top"` not `start: "top center"` or `"top 80%"`.
   363тЖТ   363тЖТ* **GSAP Horizontal-Pan Pattern (when horizontal scroll-hijack is used).** See Section 5.B below for the canonical skeleton. Common failure: animation starts before the section is pinned, so the user sees half a slide. Same fix: `start: "top top"`, pin the wrapper, scrub the inner track.
   364тЖТ   364тЖТ
   365тЖТ   365тЖТ### 5.A Sticky-Stack - Canonical Skeleton
   366тЖТ   366тЖТ
   367тЖТ   367тЖТ```tsx
   368тЖТ   368тЖТ"use client";
   369тЖТ   369тЖТimport { useRef, useEffect } from "react";
   370тЖТ   370тЖТimport { gsap } from "gsap";
   371тЖТ   371тЖТimport { ScrollTrigger } from "gsap/ScrollTrigger";
   372тЖТ   372тЖТimport { useReducedMotion } from "motion/react";
   373тЖТ   373тЖТ
   374тЖТ   374тЖТgsap.registerPlugin(ScrollTrigger);
   375тЖТ   375тЖТ
   376тЖТ   376тЖТexport function StickyStack({ cards }: { cards: React.ReactNode[] }) {
   377тЖТ   377тЖТ  const ref = useRef<HTMLDivElement>(null);
   378тЖТ   378тЖТ  const reduce = useReducedMotion();
   379тЖТ   379тЖТ
   380тЖТ   380тЖТ  useEffect(() => {
   381тЖТ   381тЖТ    if (reduce || !ref.current) return;
   382тЖТ   382тЖТ    const ctx = gsap.context(() => {
   383тЖТ   383тЖТ      const cardEls = gsap.utils.toArray<HTMLElement>(".stack-card");
   384тЖТ   384тЖТ      cardEls.forEach((card, i) => {
   385тЖТ   385тЖТ        if (i === cardEls.length - 1) return;
   386тЖТ   386тЖТ        ScrollTrigger.create({
   387тЖТ   387тЖТ          trigger: card,
   388тЖТ   388тЖТ          start: "top top",                              // pin at viewport top
   389тЖТ   389тЖТ          endTrigger: cardEls[cardEls.length - 1],
   390тЖТ   390тЖТ          end: "top top",
   391тЖТ   391тЖТ          pin: true,
   392тЖТ   392тЖТ          pinSpacing: false,
   393тЖТ   393тЖТ        });
   394тЖТ   394тЖТ        gsap.to(card, {
   395тЖТ   395тЖТ          scale: 0.92,
   396тЖТ   396тЖТ          opacity: 0.55,
   397тЖТ   397тЖТ          ease: "none",
   398тЖТ   398тЖТ          scrollTrigger: {
   399тЖТ   399тЖТ            trigger: cardEls[i + 1],
   400тЖТ   400тЖТ            start: "top bottom",
   401тЖТ   401тЖТ            end: "top top",
   402тЖТ   402тЖТ            scrub: true,
   403тЖТ   403тЖТ          },

   406тЖТ   406тЖТ    }, ref);
   407тЖТ   407тЖТ    return () => ctx.revert();
   408тЖТ   408тЖТ  }, [reduce]);
   409тЖТ   409тЖТ

   427тЖТ   427тЖТ### 5.B Horizontal-Pan - Canonical Skeleton
   428тЖТ   428тЖТ
   429тЖТ   429тЖТ```tsx
   430тЖТ   430тЖТ"use client";
   431тЖТ   431тЖТimport { useRef, useEffect } from "react";
   432тЖТ   432тЖТimport { gsap } from "gsap";
   433тЖТ   433тЖТimport { ScrollTrigger } from "gsap/ScrollTrigger";
   434тЖТ   434тЖТimport { useReducedMotion } from "motion/react";
   435тЖТ   435тЖТ
   436тЖТ   436тЖТgsap.registerPlugin(ScrollTrigger);
   437тЖТ   437тЖТ
   438тЖТ   438тЖТexport function HorizontalPan({ children }: { children: React.ReactNode }) {
   439тЖТ   439тЖТ  const wrap = useRef<HTMLDivElement>(null);
   440тЖТ   440тЖТ  const track = useRef<HTMLDivElement>(null);
   441тЖТ   441тЖТ  const reduce = useReducedMotion();
   442тЖТ   442тЖТ
   443тЖТ   443тЖТ  useEffect(() => {
   444тЖТ   444тЖТ    if (reduce || !wrap.current || !track.current) return;
   445тЖТ   445тЖТ    const ctx = gsap.context(() => {
   446тЖТ   446тЖТ      const distance = track.current!.scrollWidth - window.innerWidth;
   447тЖТ   447тЖТ      gsap.to(track.current, {
   448тЖТ   448тЖТ        x: -distance,
   449тЖТ   449тЖТ        ease: "none",
   450тЖТ   450тЖТ        scrollTrigger: {
   451тЖТ   451тЖТ          trigger: wrap.current,
   452тЖТ   452тЖТ          start: "top top",                              // pin starts when section top hits viewport top
   453тЖТ   453тЖТ          end: () => `+=${distance}`,                    // scroll distance = track width minus viewport
   454тЖТ   454тЖТ          pin: true,
   455тЖТ   455тЖТ          scrub: 1,
   456тЖТ   456тЖТ          invalidateOnRefresh: true,
   457тЖТ   457тЖТ        },
   458тЖТ   458тЖТ      });
   459тЖТ   459тЖТ    }, wrap);
   460тЖТ   460тЖТ    return () => ctx.revert();
   461тЖТ   461тЖТ  }, [reduce]);
   462тЖТ   462тЖТ
   463тЖТ   463тЖТ  return (
   464тЖТ   464тЖТ    <section ref={wrap} className="relative overflow-hidden">
   465тЖТ   465тЖТ      <div ref={track} className="flex h-[100dvh] items-center">
   466тЖТ   466тЖТ        {children}
   467тЖТ   467тЖТ      </div>
   468тЖТ   468тЖТ    </section>
   469тЖТ   469тЖТ  );
   470тЖТ   470тЖТ}
   471тЖТ   471тЖТ```
   472тЖТ   472тЖТ
   473тЖТ   473тЖТCritical points: `start: "top top"`, `pin: true`, `end: "+=${distance}"` (scroll length = horizontal travel needed), `scrub: 1`. The wrapper is pinned, the inner track slides horizontally as the user scrolls vertically.
   474тЖТ   474тЖТ
   475тЖТ   475тЖТ### 5.C Scroll-Reveal Stagger - Canonical Skeleton (lighter alternative)
   476тЖТ   476тЖТ
   477тЖТ   477тЖТFor simple "items appear as they enter viewport" (no pinning), prefer Motion's `whileInView` over GSAP - lighter, no ScrollTrigger needed:
   478тЖТ   478тЖТ
   479тЖТ   479тЖТ```tsx
   480тЖТ   480тЖТ"use client";
   481тЖТ   481тЖТimport { motion, useReducedMotion } from "motion/react";
   482тЖТ   482тЖТ
   483тЖТ   483тЖТexport function RevealStagger({ items }: { items: string[] }) {
   484тЖТ   484тЖТ  const reduce = useReducedMotion();
   485тЖТ   485тЖТ  return (
   486тЖТ   486тЖТ    <ul className="grid gap-6">
   487тЖТ   487тЖТ      {items.map((item, i) => (
   488тЖТ   488тЖТ        <motion.li
   489тЖТ   489тЖТ          key={item}
   490тЖТ   490тЖТ          initial={reduce ? false : { opacity: 0, y: 24 }}
   491тЖТ   491тЖТ          whileInView={{ opacity: 1, y: 0 }}
   492тЖТ   492тЖТ          viewport={{ once: true, amount: 0.3 }}
   493тЖТ   493тЖТ          transition={{
   494тЖТ   494тЖТ            duration: 0.6,
   495тЖТ   495тЖТ            delay: i * 0.06,
   496тЖТ   496тЖТ            ease: [0.16, 1, 0.3, 1],
   497тЖТ   497тЖТ          }}
   498тЖТ   498тЖТ        >
   499тЖТ   499тЖТ          {item}
   500тЖТ   500тЖТ        </motion.li>
   501тЖТ   501тЖТ      ))}
   502тЖТ   502тЖТ    </ul>
   503тЖТ   503тЖТ  );
   504тЖТ   504тЖТ}
   505тЖТ   505тЖТ```
   506тЖТ   506тЖТ
   507тЖТ   507тЖТUse this for: feature lists, testimonial grids, logo walls, anything that just needs "enter on scroll." Save GSAP for actual pin/scrub work.
