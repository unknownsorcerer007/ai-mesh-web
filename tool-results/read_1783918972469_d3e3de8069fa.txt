   295→### Cognitive Load Assessment
   296→
   297→Cognitive load is the total mental effort required to use an interface. Overloaded users make mistakes, get frustrated, and leave. This reference helps identify and fix cognitive overload.
   298→
   299→---
   300→
   301→#### Three Types of Cognitive Load
   302→
   303→##### Intrinsic Load: The Task Itself
   304→Complexity inherent to what the user is trying to do. You can't eliminate this, but you can structure it.
   305→
   306→**Manage it by**:
   307→- Breaking complex tasks into discrete steps
   308→- Providing scaffolding (templates, defaults, examples)
   309→- Progressive disclosure: show what's needed now, hide the rest
   310→- Grouping related decisions together
   311→
   312→##### Extraneous Load: Bad Design
   313→Mental effort caused by poor design choices. **Eliminate this ruthlessly.** It's pure waste.
   314→
   315→**Common sources**:
   316→- Confusing navigation that requires mental mapping
   317→- Unclear labels that force users to guess meaning
   318→- Visual clutter competing for attention
   319→- Inconsistent patterns that prevent learning
   320→- Unnecessary steps between user intent and result
   321→
   322→##### Germane Load: Learning Effort
   323→Mental effort spent building understanding. This is *good* cognitive load; it leads to mastery.
   324→
   325→**Support it by**:
   326→- Progressive disclosure that reveals complexity gradually
   327→- Consistent patterns that reward learning
   328→- Feedback that confirms correct understanding
   329→- Onboarding that teaches through action, not walls of text
   330→
   331→---
   332→
   333→#### Cognitive Load Checklist
   334→
   335→Evaluate the interface against these 8 items:
   336→
   337→- [ ] **Single focus**: Can the user complete their primary task without distraction from competing elements?
   338→- [ ] **Chunking**: Is information presented in digestible groups (≤4 items per group)?
   339→- [ ] **Grouping**: Are related items visually grouped together (proximity, borders, shared background)?
   340→- [ ] **Visual hierarchy**: Is it immediately clear what's most important on the screen?
   341→- [ ] **One thing at a time**: Can the user focus on a single decision before moving to the next?
   342→- [ ] **Minimal choices**: Are decisions simplified (≤4 visible options at any decision point)?
   343→- [ ] **Working memory**: Does the user need to remember information from a previous screen to act on the current one?
   344→- [ ] **Progressive disclosure**: Is complexity revealed only when the user needs it?
   345→
   346→**Scoring**: Count the failed items. 0–1 failures = low cognitive load (good). 2–3 = moderate (address soon). 4+ = high cognitive load (critical fix needed).
   347→
   348→---
   349→
   350→#### The Working Memory Rule
   351→
   352→**Humans can hold ≤4 items in working memory at once** (Miller's Law revised by Cowan, 2001).
   353→
   354→At any decision point, count the number of distinct options, actions, or pieces of information a user must simultaneously consider:
   355→- **≤4 items**: Within working memory limits, manageable
   356→- **5–7 items**: Pushing the boundary; consider grouping or progressive disclosure
   357→- **8+ items**: Overloaded; users will skip, misclick, or abandon
   358→
   359→**Practical applications**:
   360→- Navigation menus: ≤5 top-level items (group the rest under clear categories)
   361→- Form sections: ≤4 fields visible per group before a visual break
   362→- Action buttons: 1 primary, 1–2 secondary, group the rest in a menu
   363→- Dashboard widgets: ≤4 key metrics visible without scrolling
   364→- Pricing tiers: ≤3 options (more causes analysis paralysis)
   365→
   366→---
   367→
   368→#### Common Cognitive Load Violations
   369→
   370→##### 1. The Wall of Options
   371→**Problem**: Presenting 10+ choices at once with no hierarchy.
   372→**Fix**: Group into categories, highlight recommended, use progressive disclosure.
   373→
   374→##### 2. The Memory Bridge
   375→**Problem**: User must remember info from step 1 to complete step 3.
   376→**Fix**: Keep relevant context visible, or repeat it where it's needed.
   377→
   378→##### 3. The Hidden Navigation
   379→**Problem**: User must build a mental map of where things are.
   380→**Fix**: Always show current location (breadcrumbs, active states, progress indicators).
   381→
   382→##### 4. The Jargon Barrier
   383→**Problem**: Technical or domain language forces translation effort.
   384→**Fix**: Use plain language. If domain terms are unavoidable, define them inline.
   385→
   386→##### 5. The Visual Noise Floor
   387→**Problem**: Every element has the same visual weight; nothing stands out.
   388→**Fix**: Establish clear hierarchy: one primary element, 2–3 secondary, everything else muted.
   389→
   390→##### 6. The Inconsistent Pattern
   391→**Problem**: Similar actions work differently in different places.
   392→**Fix**: Standardize interaction patterns. Same type of action = same type of UI.
   393→
   394→##### 7. The Multi-Task Demand
   395→**Problem**: Interface requires processing multiple simultaneous inputs (reading + deciding + navigating).
   396→**Fix**: Sequence the steps. Let the user do one thing at a time.
   397→
   398→##### 8. The Context Switch
   399→**Problem**: User must jump between screens/tabs/modals to gather info for a single decision.
   400→**Fix**: Co-locate the information needed for each decision. Reduce back-and-forth.

   404→### Heuristics Scoring Guide
   405→
   406→Score each of Nielsen's 10 Usability Heuristics on a 0–4 scale. Be honest: a 4 means genuinely excellent, not "good enough."
   407→
   408→#### Nielsen's 10 Heuristics
   409→
   410→##### 1. Visibility of System Status
   411→
   412→Keep users informed about what's happening through timely, appropriate feedback.
   413→
   414→**Check for**:
   415→- Loading indicators during async operations
   416→- Confirmation of user actions (save, submit, delete)
   417→- Progress indicators for multi-step processes
   418→- Current location in navigation (breadcrumbs, active states)
   419→- Form validation feedback (inline, not just on submit)
   420→
   421→**Scoring**:
   422→| Score | Criteria |
   423→|-------|----------|
   424→| 0 | No feedback; user is guessing what happened |
   425→| 1 | Rare feedback; most actions produce no visible response |
   426→| 2 | Partial; some states communicated, major gaps remain |
   427→| 3 | Good; most operations give clear feedback, minor gaps |
   428→| 4 | Excellent; every action confirms, progress is always visible |
   429→
   430→##### 2. Match Between System and Real World
   431→
   432→Speak the user's language. Follow real-world conventions. Information appears in natural, logical order.
   433→
   434→**Check for**:
   435→- Familiar terminology (no unexplained jargon)
   436→- Logical information order matching user expectations
   437→- Recognizable icons and metaphors
   438→- Domain-appropriate language for the target audience
   439→- Natural reading flow (left-to-right, top-to-bottom priority)
   440→
   441→**Scoring**:
   442→| Score | Criteria |
   443→|-------|----------|
   444→| 0 | Pure tech jargon, alien to users |
   445→| 1 | Mostly confusing; requires domain expertise to navigate |
   446→| 2 | Mixed; some plain language, some jargon leaks through |
   447→| 3 | Mostly natural; occasional term needs context |
   448→| 4 | Speaks the user's language fluently throughout |
   449→
   450→##### 3. User Control and Freedom
   451→
   452→Users need a clear "emergency exit" from unwanted states without extended dialogue.
   453→
   454→**Check for**:
   455→- Undo/redo functionality
   456→- Cancel buttons on forms and modals
   457→- Clear navigation back to safety (home, previous)
   458→- Easy way to clear filters, search, selections
   459→- Escape from long or multi-step processes
   460→
   461→**Scoring**:
   462→| Score | Criteria |
   463→|-------|----------|
   464→| 0 | Users get trapped; no way out without refreshing |
   465→| 1 | Difficult exits; must find obscure paths to escape |
   466→| 2 | Some exits; main flows have escape, edge cases don't |
   467→| 3 | Good control; users can exit and undo most actions |
   468→| 4 | Full control; undo, cancel, back, and escape everywhere |
   469→
   470→##### 4. Consistency and Standards
   471→
   472→Users shouldn't wonder whether different words, situations, or actions mean the same thing.
   473→
   474→**Check for**:
   475→- Consistent terminology throughout the interface
   476→- Same actions produce same results everywhere
   477→- Platform conventions followed (standard UI patterns)
   478→- Visual consistency (colors, typography, spacing, components)
   479→- Consistent interaction patterns (same gesture = same behavior)
   480→
   481→**Scoring**:
   482→| Score | Criteria |
   483→|-------|----------|
   484→| 0 | Inconsistent everywhere; feels like different products stitched together |
   485→| 1 | Many inconsistencies; similar things look/behave differently |
   486→| 2 | Partially consistent; main flows match, details diverge |
   487→| 3 | Mostly consistent; occasional deviation, nothing confusing |
   488→| 4 | Fully consistent; cohesive system, predictable behavior |
   489→
   490→##### 5. Error Prevention
   491→
   492→Better than good error messages is a design that prevents problems in the first place.
   493→
   494→**Check for**:
   495→- Confirmation before destructive actions (delete, overwrite)
   496→- Constraints preventing invalid input (date pickers, dropdowns)
   497→- Smart defaults that reduce errors
   498→- Clear labels that prevent misunderstanding
   499→- Autosave and draft recovery
   500→
   501→**Scoring**:
   502→| Score | Criteria |
   503→|-------|----------|
   504→| 0 | Errors easy to make; no guardrails anywhere |
   505→| 1 | Few safeguards; some inputs validated, most aren't |
   506→| 2 | Partial prevention; common errors caught, edge cases slip |
   507→| 3 | Good prevention; most error paths blocked proactively |
   508→| 4 | Excellent; errors nearly impossible through smart constraints |
   509→
   510→##### 6. Recognition Rather Than Recall
   511→
   512→Minimize memory load. Make objects, actions, and options visible or easily retrievable.
   513→
   514→**Check for**:
   515→- Visible options (not buried in hidden menus)
   516→- Contextual help when needed (tooltips, inline hints)
   517→- Recent items and history
   518→- Autocomplete and suggestions
   519→- Labels on icons (not icon-only navigation)
   520→
   521→**Scoring**:
   522→| Score | Criteria |
   523→|-------|----------|
   524→| 0 | Heavy memorization; users must remember paths and commands |
   525→| 1 | Mostly recall; many hidden features, few visible cues |
   526→| 2 | Some aids; main actions visible, secondary features hidden |
   527→| 3 | Good recognition; most things discoverable, few memory demands |
   528→| 4 | Everything discoverable; users never need to memorize |
   529→
   530→##### 7. Flexibility and Efficiency of Use
   531→
   532→Accelerators, invisible to novices, speed up expert interaction.
   533→
   534→**Check for**:
   535→- Keyboard shortcuts for common actions
   536→- Customizable interface elements
   537→- Recent items and favorites
   538→- Bulk/batch actions
   539→- Power user features that don't complicate the basics
   540→
   541→**Scoring**:
   542→| Score | Criteria |
   543→|-------|----------|
   544→| 0 | One rigid path; no shortcuts or alternatives |
   545→| 1 | Limited flexibility; few alternatives to the main path |
   546→| 2 | Some shortcuts; basic keyboard support, limited bulk actions |
   547→| 3 | Good accelerators; keyboard nav, some customization |
   548→| 4 | Highly flexible; multiple paths, power features, customizable |
   549→
   550→##### 8. Aesthetic and Minimalist Design
   551→
   552→Interfaces should not contain irrelevant or rarely needed information. Every element should serve a purpose.
   553→
   554→**Check for**:
   555→- Only necessary information visible at each step
   556→- Clear visual hierarchy directing attention
   557→- Purposeful use of color and emphasis
   558→- No decorative clutter competing for attention
   559→- Focused, uncluttered layouts
   560→
   561→**Scoring**:
   562→| Score | Criteria |
   563→|-------|----------|
   564→| 0 | Overwhelming; everything competes for attention equally |
   565→| 1 | Cluttered; too much noise, hard to find what matters |
   566→| 2 | Some clutter; main content clear, periphery noisy |
   567→| 3 | Mostly clean; focused design, minor visual noise |
   568→| 4 | Perfectly minimal; every element earns its pixel |
   569→
   570→##### 9. Help Users Recognize, Diagnose, and Recover from Errors
   571→
   572→Error messages should use plain language, precisely indicate the problem, and constructively suggest a solution.
   573→
   574→**Check for**:
   575→- Plain language error messages (no error codes for users)
   576→- Specific problem identification ("Email is missing @" not "Invalid input")
   577→- Actionable recovery suggestions
   578→- Errors displayed near the source of the problem
   579→- Non-blocking error handling (don't wipe the form)
   580→
   581→**Scoring**:
   582→| Score | Criteria |
   583→|-------|----------|
   584→| 0 | Cryptic errors; codes, jargon, or no message at all |
   585→| 1 | Vague errors; "Something went wrong" with no guidance |
   586→| 2 | Clear but unhelpful; names the problem but not the fix |
   587→| 3 | Clear with suggestions; identifies problem and offers next steps |
   588→| 4 | Perfect recovery; pinpoints issue, suggests fix, preserves user work |
   589→
   590→##### 10. Help and Documentation
   591→
   592→Even if the system is usable without docs, help should be easy to find, task-focused, and concise.
   593→
   594→**Check for**:
   595→- Searchable help or documentation
   596→- Contextual help (tooltips, inline hints, guided tours)
   597→- Task-focused organization (not feature-organized)
   598→- Concise, scannable content
   599→- Easy access without leaving current context
   600→
   601→**Scoring**:
   602→| Score | Criteria |
   603→|-------|----------|
   604→| 0 | No help available anywhere |
   605→| 1 | Help exists but hard to find or irrelevant |
   606→| 2 | Basic help; FAQ or docs exist, not contextual |
   607→| 3 | Good documentation; searchable, mostly task-focused |
   608→| 4 | Excellent contextual help; right info at the right moment |
   609→

   611→
   612→#### Score Summary
   613→
   614→**Total possible**: 40 points (10 heuristics × 4 max)
   615→
   616→| Score Range | Rating | What It Means |
   617→|-------------|--------|---------------|
   618→| 36–40 | Excellent | Minor polish only; ship it |
   619→| 28–35 | Good | Address weak areas, solid foundation |
   620→| 20–27 | Acceptable | Significant improvements needed before users are happy |
   621→| 12–19 | Poor | Major UX overhaul required; core experience broken |
   622→| 0–11 | Critical | Redesign needed; unusable in current state |

   626→#### Issue Severity (P0–P3)
   627→
   628→Tag each individual issue found during scoring with a priority level:
   629→
   630→| Priority | Name | Description | Action |
   631→|----------|------|-------------|--------|
   632→| **P0** | Blocking | Prevents task completion entirely | Fix immediately; this is a showstopper |
   633→| **P1** | Major | Causes significant difficulty or confusion | Fix before release |
   634→| **P2** | Minor | Annoyance, but workaround exists | Fix in next pass |
   635→| **P3** | Polish | Nice-to-fix, no real user impact | Fix if time permits |
   636→
   637→**Tip**: If you're unsure between two levels, ask: "Would a user contact support about this?" If yes, it's at least P1.

   641→### Persona-Based Design Testing
   642→
   643→Test the interface through the eyes of 5 distinct user archetypes. Each persona exposes different failure modes that a single "design director" perspective would miss.
   644→
   645→**How to use**: Select 2–3 personas most relevant to the interface being critiqued. Walk through the primary user action as each persona. Report specific red flags, not generic concerns.
   646→
   647→---
   648→
   649→#### 1. Impatient Power User: "Alex"
   650→
   651→
   652→**Profile**: Expert with similar products. Expects efficiency, hates hand-holding. Will find shortcuts or leave.
   653→
   654→**Behaviors**:
   655→- Skips all onboarding and instructions
   656→- Looks for keyboard shortcuts immediately
   657→- Tries to bulk-select, batch-edit, and automate
   658→- Gets frustrated by required steps that feel unnecessary
   659→- Abandons if anything feels slow or patronizing
   660→
   661→**Test Questions**:
   662→- Can Alex complete the core task in under 60 seconds?
   663→- Are there keyboard shortcuts for common actions?
   664→- Can onboarding be skipped entirely?
   665→- Do modals have keyboard dismiss (Esc)?
   666→- Is there a "power user" path (shortcuts, bulk actions)?
   667→
   668→**Red Flags** (report these specifically):
   669→- Forced tutorials or unskippable onboarding
   670→- No keyboard navigation for primary actions
   671→- Slow animations that can't be skipped
   672→- One-item-at-a-time workflows where batch would be natural
   673→- Redundant confirmation steps for low-risk actions
   674→
   675→---
   676→
   677→#### 2. Confused First-Timer: "Jordan"
   678→
   679→**Profile**: Never used this type of product. Needs guidance at every step. Will abandon rather than figure it out.
   680→
   681→**Behaviors**:
   682→- Reads all instructions carefully
   683→- Hesitates before clicking anything unfamiliar
   684→- Looks for help or support constantly
   685→- Misunderstands jargon and abbreviations
   686→- Takes the most literal interpretation of any label
   687→
   688→**Test Questions**:
   689→- Is the first action obviously clear within 5 seconds?
   690→- Are all icons labeled with text?
   691→- Is there contextual help at decision points?
   692→- Does terminology assume prior knowledge?
   693→- Is there a clear "back" or "undo" at every step?
   694→
   695→**Red Flags** (report these specifically):
   696→- Icon-only navigation with no labels
   697→- Technical jargon without explanation
   698→- No visible help option or guidance
   699→- Ambiguous next steps after completing an action
   700→- No confirmation that an action succeeded
   701→
   702→---
   703→
   704→#### 3. Accessibility-Dependent User: "Sam"
   705→
   706→**Profile**: Uses screen reader (VoiceOver/NVDA), keyboard-only navigation. May have low vision, motor impairment, or cognitive differences.
   707→
   708→**Behaviors**:
   709→- Tabs through the interface linearly
   710→- Relies on ARIA labels and heading structure
   711→- Cannot see hover states or visual-only indicators
   712→- Needs adequate color contrast (4.5:1 minimum)
   713→- May use browser zoom up to 200%
   714→
   715→**Test Questions**:
   716→- Can the entire primary flow be completed keyboard-only?
   717→- Are all interactive elements focusable with visible focus indicators?
   718→- Do images have meaningful alt text?
   719→- Is color contrast WCAG AA compliant (4.5:1 for text)?
   720→- Does the screen reader announce state changes (loading, success, errors)?
   721→
   722→**Red Flags** (report these specifically):
   723→- Click-only interactions with no keyboard alternative
   724→- Missing or invisible focus indicators
   725→- Meaning conveyed by color alone (red = error, green = success)
   726→- Unlabeled form fields or buttons
   727→- Time-limited actions without extension option
   728→- Custom components that break screen reader flow
   729→
   730→---
   731→
   732→#### 4. Deliberate Stress Tester: "Riley"
   733→
   734→**Profile**: Methodical user who pushes interfaces beyond the happy path. Tests edge cases, tries unexpected inputs, and probes for gaps in the experience.
   735→
   736→**Behaviors**:
   737→- Tests edge cases intentionally (empty states, long strings, special characters)
   738→- Submits forms with unexpected data (emoji, RTL text, very long values)
   739→- Tries to break workflows by navigating backwards, refreshing mid-flow, or opening in multiple tabs
   740→- Looks for inconsistencies between what the UI promises and what actually happens
   741→- Documents problems methodically
   742→
   743→**Test Questions**:
   744→- What happens at the edges (0 items, 1000 items, very long text)?
   745→- Do error states recover gracefully or leave the UI in a broken state?
   746→- What happens on refresh mid-workflow? Is state preserved?
   747→- Are there features that appear to work but produce broken results?
   748→- How does the UI handle unexpected input (emoji, special chars, paste from Excel)?
   749→
   750→**Red Flags** (report these specifically):
   751→- Features that appear to work but silently fail or produce wrong results
   752→- Error handling that exposes technical details or leaves UI in a broken state
   753→- Empty states that show nothing useful ("No results" with no guidance)
   754→- Workflows that lose user data on refresh or navigation
   755→- Inconsistent behavior between similar interactions in different parts of the UI
   756→
   757→---
   758→
   759→#### 5. Distracted Mobile User: "Casey"
   760→
   761→**Profile**: Using phone one-handed on the go. Frequently interrupted. Possibly on a slow connection.
   762→
   763→**Behaviors**:
   764→- Uses thumb only; prefers bottom-of-screen actions
   765→- Gets interrupted mid-flow and returns later
   766→- Switches between apps frequently
   767→- Has limited attention span and low patience
   768→- Types as little as possible, prefers taps and selections
   769→
   770→**Test Questions**:
   771→- Are primary actions in the thumb zone (bottom half of screen)?
   772→- Is state preserved if the user leaves and returns?
   773→- Does it work on slow connections (3G)?
   774→- Can forms use autocomplete and smart defaults?
   775→- Are touch targets at least 44×44pt?
   776→
   777→**Red Flags** (report these specifically):
   778→- Important actions positioned at the top of the screen (unreachable by thumb)
   779→- No state persistence; progress lost on tab switch or interruption
   780→- Large text inputs required where selection would work
   781→- Heavy assets loading on every page (no lazy loading)
   782→- Tiny tap targets or targets too close together
